# OpenCode Subagent 派发指南

> 基于 T020 重构实战 + 14 次系统对照实验的验证结论。
> OpenCode + agate dispatch prompt 格式。

## 核心结论

**agate dispatch prompt 格式使 subagent 极其可靠。** 14/14 实验任务全部成功，覆盖了测试（1-3 文件）、实现（1-2 组件）、异构（实现+测试混合）等所有常见任务类型。

**目前唯一已知的失败场景**：T020 DiagramBlock 完整版（5 个行为 × 3 个语言分支 = 15 个交叉决策点在一个组件内）。这是极端情况，正常开发不会遇到。

## 何时拆分

以下情况可以放心合并到一个 subagent：
- 给 ≤3 个独立组件写测试
- 创建 ≤2 个独立组件 + 它们的测试
- 创建 1 个 composable + 它的测试
- 文件数 ≤4

以下情况必须拆分：
- 单个组件内 **5+ 个行为维度**，且 **行为之间有交叉分支**（如"mermaid 改文字、plantuml 不改、svg 改文字"这种类型 × 行为矩阵）
- 一个 subagent 产出 >4 个文件
- 需要同时理解 5+ 个不同技术领域

## Dispatch Prompt 模板

使用 agate 标准格式。已验证能使 subagent 在测试/实现/混合任务上 100% 成功：

```
你是 P{N} 阶段的 {角色名} 子 Agent。

## 你的角色定义
读取并遵循：~/.agate/assets/execution-roles/{role}.md

## 项目约定
- {project_root}/AGENTS.md

## 环境隔离
- {测试/构建命令}
- 严禁的操作

## 输入（自己读取，不要等我提供内容）
- {文件路径}（{为什么读它}）
- ...

## 任务
{一句话描述任务}

## 分阶段落盘
每完成一个关键步骤，立即追加写入 {progress_file_path}

## 输出
1. {产出文件路径}
2. {产出文件路径}

## 门槛
- {验证命令}
- {类型检查命令}

## 返回给我
只返回两行：
1. 产出文件路径列表
2. 一句话摘要（不超过 30 字）
```

## 已知门槛数据

| 任务类型 | 已验证上限 | 状态 |
|---------|-----------|------|
| 测试（独立组件） | 3 文件，含 MarkdownViewer 439 行 + LoginDialog 355 行 | ✅ |
| 实现（单组件） | 2 组件 + 2 测试 = 4 文件 | ✅ |
| 实现（composable） | 1 composable + 1 测试 = 2 文件 | ✅ |
| 实现（异构混合） | composable + test 一起 | ✅ |
| 实现（多 variant 组件） | 4 variant 分支 | ✅ |

## 当前未探明的边界

- 测试任务 4+ 文件是否安全 → **未测试**
- 实现任务 3+ 组件是否安全 → **未测试**
- 决策交叉数从 4 到 15 的中间值（6/8/10/12）→ **未测试**

这些边界在正常开发中不会触及。遇到时按"何时拆分"规则处理即可。

## 失败恢复

如果 subagent 空返回：
1. 检查 progress 文件是否有内容
2. `git log` + `git status` 看已完成的改动
3. 从断点拆成更小的子任务重派

## 与 agate 的关系

本指南不替代 agate dispatch-protocol，而是基于 peepview 项目的实战验证补充：
- agate 提供 dispatch 标准格式和流程规范
- 本指南提供 peepview 项目的**已验证安全阈值**
