# Workflow v3 评审迭代与多层质疑机制

> 目的：补齐 v3 缺失的「方案→评审→修改→再评审」外层循环，以及多轮评审失败后的"质疑上溯"机制
> 创建：2026-06-12
> 关联：T003 修复方案

---

## 一、问题诊断

v3 当前只有 **P1-P7 阶段内的重试机制**（dispatch-protocol.md 的 retry_count），缺少：

1. **阶段间的"再评审"**：architect 重写方案后，没人再评审一次
2. **多轮评审失败的处理**：连续 N 轮改不好怎么办？
3. **质疑上溯**：方案/设计/AC 反复改不好，要能质疑到原始需求
4. **用户介入边界**：什么情况下必须人介入？

T002 的 AC6 教训正是这个机制缺失的实证——AC6（为已否决方案写的验收标准）从 P1 一路漏到 P6，无人质疑。

---

## 二、设计目标

```
方案 → 评审 N1 → 修改 → 评审 N2 → ... → 评分 ≥ 门槛 → 进实施
         ↑                     ↓
         │              重试超限（3 轮）
         │                     ↓
         │              上溯一层（质疑输入）
         │                     ↓
         │              输入也改不好？
         │                     ↓
         └──────────── PAUSED，报告用户
```

**核心思想**：每一层都有自己的"评审 → 修改 → 再评审"循环，再评审失败就质疑上溯。质疑贯穿 P1-P7 全部阶段，**不只 P2**。

**修正后范围（基于评审反馈）**：
- **L1（阶段内再评审）**：✅ 实施
- **L2（阶段间上溯）**：✅ 实施，但简化为**确定性单规则**
- **L3（需求质疑）**：📝 保留为设计文档，不实施（需平台支持用户交互）

## 三、机制结构

### 3.1 三大质疑层级

| 层级 | 质疑对象 | 谁来质疑 | 触发上溯 |
|------|----------|----------|----------|
| **L1 任务内** | 当前阶段的产出文件 | 同阶段评审角色 | 3 轮评分 < 门槛 |
| **L2 阶段间** | 上游产出（设计有误）| 上一阶段执行角色 | 上游被驳回 3 轮 |
| **L3 需求层** | P1 原始问题定义 | 用户 | 需求模糊/矛盾/不可行 |

### 3.2 评审应用范围（修正后）

| 阶段 | 是否加评审 | 评审什么 | v3 现状 | 说明 |
|------|-----------|----------|---------|------|
| **P1 前** | ✅ 加（大任务）| `office-hours` 需求方向 | 已有 | P0/P1 任务 / 架构变更 / 3+ 模块时触发 |
| **P1 后** | ✅ 加（大任务）| `office-hours` AC 复审 | **缺失** | 审 AC 是否可验证、范围是否合理 |
| **P2** | ✅ 已有 | `plan-eng-review` + `plan-ceo-review` | 有 | 架构 / 方向 |
| **P3** | ❌ 不加 | gate 自检（TDD 红灯）| 缺失 | **修正**：TDD 红灯 = 最严格 gate，无需评审角色 |
| **P4 后** | ✅ 已有 | `review` + `cso` + `design-review` | 有 | bug / 安全 / UI |
| **P5** | ❌ 不加 | gate 自检（pytest 全绿）| 有 | **修正**：pytest 全绿 = 客观 gate，无需 qa 评审 |
| **P6** | ✅ 已有 | architect 双向检查 | 有但单向 | T003 C8 已修双向 |
| **P7** | ❌ 不加 | gate 自检（发布准备）| 缺失 | **修正**：`make pre-publish` exit 0 = gate，无需人验收 |

**关键认知修正**：
- 评审**不是越多越好**，v3 的「轻量化」设计选择用 gate 替代评审是合理的
- 只在**定性阶段**加评审（P1/P2/P4/P6），定量阶段用 gate（P3/P5/P7）
- P3 的 TDD 红灯、P5 的 pytest 全绿、P7 的 `make pre-publish` 本身就是最严格的客观判定

## 四、循环机制设计

### 4.1 单层循环（阶段内）

```
阶段内循环：
  阶段执行者产出文件 (Pn-xxx.md)
       ↓
  主 Agent 跑 gate 命令（A1 原则：跑命令不信文件）
       ↓
  通过？ ──是──→ 进入下一阶段
       ↓ 否
  派发评审角色读产出
       ↓
  评审角色产出 Pn-review.md (status: approved/rejected/needs-revision)
       ↓
  approved? ──是──→ 进入下一阶段
       ↓ 否
  retry_count += 1
       ↓
  retry_count > MAX_RETRY (默认 3)?
       ↓ 是
  触发上溯（见 4.2）
       ↓ 否
  执行者重写产出（带回评审反馈）
       ↓
  回到"主 Agent 跑 gate 命令"步骤
```

### 4.2 跨层上溯（多轮失败 → 单规则）

| 阶段 | 上溯对象 | 上溯动作 |
|------|----------|----------|
| P1 失败 3 轮 | **需求源**（用户原始输入）| PAUSED，报告用户需求可能不合理 |
| P2 失败 3 轮 | **P1** | P1 标记 needs-review，office-hours 复审 AC |
| P3 失败 3 轮 | **P2** | P2 标记 needs-review，architect 重新设计 |
| P4 失败 3 轮 | **P2** | P2 标记 needs-review，质疑设计方案 |
| P5 失败 3 轮 | **P4** | P4 标记 needs-review，重新实现 |
| P6 失败 3 轮 | **P2** | P2 标记 needs-review，质疑 AC（僵尸需求）|
| P7 失败 3 轮 | **P6** | P6 标记 needs-review，重新检查一致性 |

**修正说明**：原设计 P4/P5/P6 各有两个上溯对象（二选一），改为**单一上溯对象**——紧邻的最相关上游。

### 4.3 上溯规则（修正后：确定性单规则）

**原设计（多分支决策树）**：
```
├─ 上一阶段未完成 → 回到上一阶段补做
├─ 上一阶段已完成但评分低 → 回到上一阶段重做
├─ 上游多轮失败 → 进一步上溯
└─ 上游良好但本阶段仍失败 → 怀疑输入本身有问题
```

**问题**：多分支判断要求 LLM 做综合推理，T002 复盘已证明"LLM 不能稳定执行复杂判定"。

**修正后（确定性单规则）**：

```
任何阶段失败 3 轮 → 上溯到紧邻上游
上游自动标记为 status: needs-review
不区分原因，不判断分支
```

**为什么简单规则更可靠**：
- 主 Agent 只需读 `retry_count.Pn > 3` → 执行固定动作
- 无需分析"上游是否完成"、"评分是否低"等多变量
- 避免无限循环（反复上溯→下溯→上溯）

**上溯后的处理**：

| 上游阶段 | 动作 | 产出 |
|----------|------|------|
| P1（需求） | office-hours 重新评审 | P1-review.md |
| P2（设计） | architect 重新设计 | P2-design.md（v2） |
| P3（测试） | test-designer 重写测试 | P3-test-cases.md（v2） |
| P4（实现） | implementer 重新实现 | P4-implementation/（v2） |
| P5（验证） | verifier 重新验证 | P5-test-results/（v2） |
| P6（一致性） | architect 重新检查 | P6-consistency.md（v2） |

**L3（需求质疑）——保留为设计文档，不实施**：

```
L2 也失败 3 轮？
       ↓ 是
回到 L3（需求质疑）：
   派发 office-hours 角色
   审"原始需求是否合理、是否矛盾、是否可实现"
       ↓
L3 也无法闭环？
       ↓
PAUSED + 报告用户（见 4.4）
```

**注意**：L3 需要与用户交互，当前平台不支持自动用户交互。保留为设计目标，等平台支持后实施。

### 4.4 用户介入边界

**何时必须用户介入**：

| 情况 | 例子 |
|------|------|
| 1. 三层质疑都无法闭环 | 需求本身可能不合理 |
| 2. 涉及业务方向决策 | "这个功能要不要做" |
| 3. 涉及外部资源/权限 | 需要 API key、需要授权 |
| 4. 涉及安全/合规 | 安全策略选择 |
| 5. 涉及版本/发布 | 何时发版、发到哪个仓库 |
| 6. 重试上限被触发的所有情况 | 兜底机制 |

**用户介入方式**（占位符化）：

```markdown
[PAUSED] {task_id} 需用户介入

任务背景：{task_name}
当前阶段：{phase}
失败原因：连续 {retry_count} 轮 {phase} 评审发现 {issue_summary}

已尝试的解决方案：
  {attempted_solutions}

需要用户决策：
  - [ ] {option_1}
  - [ ] {option_2}
  - [ ] {option_3}

请回复选项或直接说明。
```

**示例**（T002 场景）：

```markdown
[PAUSED] T002 需用户介入

任务背景：数据库迁移机制修复
当前阶段：P6 一致性检查
失败原因：连续 3 轮 P6 评审发现 AC6 与实现冲突

已尝试的解决方案：
  1. 修改 P6 检查方法（失败）
  2. 重新审视 P4 实现（发现 AC6 与代码不一致）
  3. 上溯 P1 检查 AC 定义（AC6 确实是僵尸需求）

需要用户决策：
  - [ ] 删除 AC6（推荐）
  - [ ] 修改 AC6 为新需求
  - [ ] 保留 AC6 重新设计实现

请回复选项或直接说明。
```

## 五、重试上限与门槛

### 5.1 重试上限（修正后，与 T003 同步）

| 阶段 | 默认 MAX_RETRY | 来源 | 注意 |
|------|----------------|------|------|
| P1 | 3 | state-machine.md | 涉及需求，不自动降级 |
| P2 | 3 | state-machine.md | 涉及设计，不自动降级 |
| P3 | 2 | gate 自检 | TDD 红灯，少轮次 |
| P4 | 3 | state-machine.md | 实现复杂度高 |
| P5 | 2 | gate 自检 | pytest 全绿，少轮次 |
| P6 | 2 | gate 自检 | 一致性检查，少轮次 |
| P7 | 2 | gate 自检 | 发布准备，少轮次 |

**注意**：MAX_RETRY 值需同步到 `loop-orchestration.md` 中「每阶段重试上限默认 3」的旧值。

### 5.2 评分门槛（修正后，对应评审范围表）

每轮评审必须有可判定的通过标准：

| 评审角色 | 适用阶段 | 通过门槛 | 失败触发 |
|----------|----------|----------|----------|
| `plan-eng-review` | P2 | 无 BLOCKER | retry |
| `plan-ceo-review` | P2 | 方向无偏离 | retry |
| `office-hours`（AC 复审）| P1 后 | AC 全部可验证 | 上溯 P1 |
| `review` | P4 后 | 无生产级 bug | retry |
| `cso` | P4 后 | 无安全 BLOCKER | retry |
| `design-review` | P4 后（前端）| UI 无问题 | retry |

**注意**：P3/P5/P7 用 gate 自检，不加强制评审角色。见 §3.2 表。

### 5.3 评分累计

每次评审产出的 status + 评分写入 `.state.yaml`：

```yaml
phase: P2
retry_count: { P2: 2 }
review_scores:
  P2:
    - round: 1
      reviewer: plan-eng-review
      score: 7.5
      status: rejected
      feedback: "API 限流策略未考虑并发边界"
    - round: 2
      reviewer: plan-eng-review
      score: 8.5
      status: rejected
      feedback: "已补限流，但缺测试覆盖"
```

主 Agent 看到 `retry_count.P2 > 3` → 触发上溯。

### 阻塞修正（🔴）

#### 修正 1：上溯决策树 → 确定性单规则

**原设计**：4 分支决策树（基于上游状态、评分历史做判断）
**问题**：LLM 无法稳定执行多变量综合判断（与 T002 复盘教训一致）
**修正**：任何阶段失败 3 轮 → 上溯到紧邻上游。不区分原因，不判断分支。

#### 修正 2：评审范围收窄

**原设计**："每个阶段都应有评审"（P1/P3/P4/P5/P7）
**问题**：与 v3 轻量化理念冲突，P3/P5/P7 的 gate 本身就是最严格的客观判定
**修正**：只在定性阶段加评审（P1/P2/P4/P6），定量阶段用 gate 自检（P3/P5/P7）

#### 修正 3：P3 互评 → gate 自检

**原设计**：P3 新增 test-designer 互评
**问题**：v3 是单执行者模型，不存在另一个 test-designer 做互评
**修正**：P3 gate 自检（TC 数 vs AC 数、pytest 收集确认、TDD 红灯为 assertion failure）

### 建议修正（🟡/🟢）
