# Workflow v3 评审迭代与多层质疑机制

> 目的：补齐 v3 缺失的「方案→评审→修改→再评审」外层循环，以及多轮评审失败后的"质疑上溯"机制
> 创建：2026-06-12
> 关联：T003 修复方案

---

## 一、问题诊断

v3 当前只有 **P1-P7 阶段内的重试机制**（dispatch-protocol.md 的 retry_count），缺少：

1. **阶段间的"再评审"**：architect 重写方案后，没人再评审一次
2. **多轮评审失败的处理**：连续 N 轮改不好怎么办？
3. **质疑上溯**：方案/设计/AC 反复改不好，要能质疑到原始需求
4. **用户介入边界**：什么情况下必须人介入？

T002 的 AC6 教训正是这个机制缺失的实证——AC6（为已否决方案写的验收标准）从 P1 一路漏到 P6，无人质疑。

---

## 二、设计目标

```
方案 → 评审 N1 → 修改 → 评审 N2 → ... → 评分 ≥ 门槛 → 进实施
         ↑                     ↓
         │              重试超限（3 轮）
         │                     ↓
         │              上溯一层（质疑输入）
         │                     ↓
         │              输入也改不好？
         │                     ↓
         └──────────── PAUSED，报告用户
```

**核心思想**：每一层都有自己的"评审 → 修改 → 再评审"循环，再评审失败就质疑上溯。质疑贯穿 P1-P7 全部阶段，**不只 P2**。

---

## 三、机制结构

### 3.1 三大质疑层级

| 层级 | 质疑对象 | 谁来质疑 | 触发上溯 |
|------|----------|----------|----------|
| **L1 任务内** | 当前阶段的产出文件 | 同阶段评审角色 | 3 轮评分 < 门槛 |
| **L2 阶段间** | 上游产出（设计有误）| 上一阶段执行角色 | 上游被驳回 3 轮 |
| **L3 需求层** | P1 原始问题定义 | 用户 | 需求模糊/矛盾/不可行 |

### 3.2 评审应用范围

| 阶段 | 应有评审 | 评审什么 | v3 现状 |
|------|----------|----------|---------|
| **P1 前** | `office-hours` | 需求方向 | 已有（role-system.md 提）|
| **P1 后** | `office-hours` 复审 | AC 是否可验证、范围是否合理 | **缺失** |
| **P2** | `plan-eng-review` + `plan-ceo-review` | 架构 / 方向 | 有（按需）|
| **P3** | `test-designer` 互评 | 测试用例是否覆盖 AC | 缺失 |
| **P4** | `review` + `cso` + `design-review` | bug / 安全 / UI | 有（按需）|
| **P5** | `qa` | 验证完整性 | 有 |
| **P6** | architect 双向检查 | 设计 vs 实现 | 有但单向 |
| **P7** | 用户手动验收 | 准备充分性 | 缺失（应人验收）|

**关键认知**：评审**不是 P2 独有的**，每个阶段都应有评审（按需触发）。v3 当前的认知把评审窄化为"P2 阶段"是错的。

## 四、循环机制设计

### 4.1 单层循环（阶段内）

```
阶段内循环：
  阶段执行者产出文件 (Pn-xxx.md)
       ↓
  主 Agent 跑 gate 命令（A1 原则：跑命令不信文件）
       ↓
  通过？ ──是──→ 进入下一阶段
       ↓ 否
  派发评审角色读产出
       ↓
  评审角色产出 Pn-review.md (status: approved/rejected/needs-revision)
       ↓
  approved? ──是──→ 进入下一阶段
       ↓ 否
  retry_count += 1
       ↓
  retry_count > MAX_RETRY (默认 3)?
       ↓ 是
  触发上溯（见 4.2）
       ↓ 否
  执行者重写产出（带回评审反馈）
       ↓
  回到"主 Agent 跑 gate 命令"步骤
```

### 4.2 跨层上溯（多轮失败）

| 阶段 | 上溯对象 | 上溯动作 |
|------|----------|----------|
| P1 失败 3 轮 | **需求源**（用户原始输入）| PAUSED，报告用户需求可能不合理 |
| P2 失败 3 轮 | **P1（AC 定义）** | 质疑 AC 是否可验证、是否与需求矛盾 |
| P3 失败 3 轮 | **P2（设计方案）** | 质疑方案是否可测、是否过度设计 |
| P4 失败 3 轮 | **P2（设计）或 P1（需求）** | 质疑实现是否合理或需求是否清晰 |
| P5 失败 3 轮 | **P4（实现）或 P3（测试）** | 质疑测试是否过严或实现是否漏边界 |
| P6 失败 3 轮 | **P2（设计）或 P1（需求）** | 质疑 AC 是否还在适用（僵尸需求）|
| P7 失败 3 轮 | **P5（验证）或 P6（一致性）** | 质疑发布检查是否合理 |

### 4.3 上溯决策树

```
某阶段 N 失败 3 轮
       ↓
检查上游各阶段状态：
├─ 上一阶段未完成 → 回到上一阶段补做
├─ 上一阶段已完成但评分低 → 回到上一阶段重做
├─ 上游多轮失败 → 进一步上溯
└─ 上游良好但本阶段仍失败 → 怀疑输入（需求/设计）本身有问题
       ↓
回到 L2（阶段间质疑）：
   派发上一阶段执行角色 + 评审角色
   审议"是否上游的某些决策需要调整"
       ↓
L2 也失败 3 轮？
       ↓ 是
回到 L3（需求质疑）：
   派发 office-hours 角色
   审"原始需求是否合理、是否矛盾、是否可实现"
       ↓
L3 也无法闭环？
       ↓
PAUSED + 报告用户（见 4.4）
```

### 4.4 用户介入边界

**何时必须用户介入**：

| 情况 | 例子 |
|------|------|
| 1. 三层质疑都无法闭环 | 需求本身可能不合理 |
| 2. 涉及业务方向决策 | "这个功能要不要做" |
| 3. 涉及外部资源/权限 | 需要 API key、需要授权 |
| 4. 涉及安全/合规 | 安全策略选择 |
| 5. 涉及版本/发布 | 何时发版、发到哪个仓库 |
| 6. 重试上限被触发的所有情况 | 兜底机制 |

**用户介入方式**：

```markdown
[PAUSED] T002 需用户介入

任务背景：数据库迁移机制修复
当前阶段：P6 一致性检查
失败原因：连续 3 轮 P6 评审发现 AC6 与实现冲突

已尝试的解决方案：
  1. 修改 P6 检查方法（失败）
  2. 重新审视 P4 实现（发现 AC6 与代码不一致）
  3. 上溯 P1 检查 AC 定义（AC6 确实是僵尸需求）

需要用户决策：
  - [ ] 删除 AC6（推荐）
  - [ ] 修改 AC6 为新需求
  - [ ] 保留 AC6 重新设计实现

请回复选项或直接说明。
```

## 五、重试上限与门槛

### 5.1 重试上限

| 阶段 | 默认 MAX_RETRY | 可配置 |
|------|----------------|--------|
| P1 | 3 | 否（涉及需求）|
| P2 | 3 | 否（涉及设计）|
| P3 | 2 | 是（小问题）|
| P4 | 3 | 否（实现复杂）|
| P5 | 2 | 是（测试问题）|
| P6 | 2 | 是（一致性）|
| P7 | 2 | 是（发布准备）|

### 5.2 评分门槛

每轮评审必须有可判定的通过标准：

| 评审角色 | 通过门槛 | 失败触发 |
|----------|----------|----------|
| `plan-eng-review` | 无 BLOCKER | retry |
| `plan-ceo-review` | 方向无偏离 | retry |
| `office-hours`（AC 复审）| AC 全部可验证 | 上溯 P1 |
| `qa` | 全部 P1 问题在 P5 中验证 | retry 或上溯 P4 |
| `cso` | 无安全 BLOCKER | retry |

### 5.3 评分累计

每次评审产出的 status + 评分写入 `.state.yaml`：

```yaml
phase: P2
retry_count: { P2: 2 }
review_scores:
  P2:
    - round: 1
      reviewer: plan-eng-review
      score: 7.5
      status: rejected
      feedback: "API 限流策略未考虑并发边界"
    - round: 2
      reviewer: plan-eng-review
      score: 8.5
      status: rejected
      feedback: "已补限流，但缺测试覆盖"
```

主 Agent 看到 `retry_count.P2 > 3` → 触发上溯。
