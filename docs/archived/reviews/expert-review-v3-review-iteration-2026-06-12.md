## 评审结论

这是一个**架构正确但复杂度超标**的方案。问题诊断精确——AC6 从 P1 漏到 P6 的根因确实是缺少「改完谁再审」的上溯循环。三层质疑模型（L1 阶段内 / L2 阶段间 / L3 需求层）概念清晰。但**3 个问题阻碍直接实施**：上溯决策树 LLM 执行可靠性未验证、「每阶段都应有评审」与 v3 轻量化设计冲突、P3 互评不可行（单执行者模型）。**2 个建议**：范围收窄到 L1+L2（去掉 L3）、P3 互评改为 gate 自检。修完可并入 T003。

## 二、技术评审

### 发现 1（🔴 高危）：上溯决策树 §4.3 的 LLM 可执行性未验证

**位置**：§4.3 上溯决策树

**现象**：树中有 4 个分支基于「上一阶段状态」做判断：

```
├─ 上一阶段未完成 → 回到上一阶段补做
├─ 上一阶段已完成但评分低 → 回到上一阶段重做
├─ 上游多轮失败 → 进一步上溯
└─ 上游良好但本阶段仍失败 → 怀疑输入本身有问题
```

**机理分析**：这要求主 Agent 在判定「本阶段失败 3 轮后该做什么」时，**不仅读当前状态，还要读上游评审历史**。v3 的状态机设计是「读 active-tasks.md 的当前阶段 + 重试计数 → 做一步 → 更新状态」。这是单步、单变量的判定。上溯决策树是多变量（上游状态 × 评审历史 × 本阶段重试次数）的综合判断。

T002 复盘的核心教训是「LLM 不能稳定执行复杂判定」——这就是为什么 gate 要从「读文件」改成「跑命令」。但上溯决策树把一个需要跑命令的问题**升级成了需要 LLM 做综合判断的问题**。

**影响**：如果主 Agent 在上溯决策时判断错误（选了错误的分支），可能陷入无限循环（反复上溯→下溯→上溯）或错误 PAUSED。

**整改建议**：

方案 A（简化）：取消决策树分支，统一规则——**任何阶段失败 3 轮 → 上溯到紧邻上游，上游自动标记为待复审**。不区分「上游未完成」还是「上游评分低」。

方案 B（延迟）：上溯决策树保留为设计目标，但 L1（阶段内循环）先行实施，L2/L3 在 L1 稳定后再加。

无论是 A 还是 B，上溯行为必须是**确定性的单规则**，而非多分支判断。

**验证方式**：
- [ ] 模拟 T002 场景：P6 失败 3 轮 → 主 Agent 自动回退到 P2（单规则），不经过分支树判断

### 发现 2（🟡 中危）：「每阶段都应有评审」与 v3 设计理念冲突

**位置**：§3.2 评审应用范围表 + §4 结论段

**现象**：
```
关键认知：评审不是 P2 独有的，每个阶段都应有评审。v3 当前的认知把评审窄化为"P2 阶段"是错的。
```

v3 当前的设计理念是「轻量化流程」——P1（问题定义）无评审，P3（测试设计）评价标准是 gate（TDD 红灯），P5（验证）评价标准是 pytest 全绿。这些阶段用 **gate 替代评审**，是 v3 刻意为之的设计选择，不是设计缺陷。

**机理分析**：加了评审的阶段会增加至少 2 步额外派发（评审 + 组长汇总），从 P1-P7 的 7 步扩展为 12-14 步。上下文增量和时间成本线性增长。T003 F13 刚加了「每阶段输出进度」来减少用户焦虑——加评审会抵消这个改进。

**影响**：评审不是越多越好。P3 的 TDD 红灯本身就是最严格的 gate——测试不通过连 P4 都进不去，再加人工评审是冗余。

**整改建议**：收窄评审范围——只在「定性阶段」加评审：

| 阶段 | 是否加评审 | 理由 |
|------|-----------|------|
| P1 后 | ✅ 加（大任务） | 问题定义是定性工作，需要复审 |
| P2 | ✅ 已有 | 方案设计是定性工作 |
| P3 | ❌ 不加 | TDD 红灯 = gate，客观可判定 |
| P4 后 | ✅ 已有 | 代码实现是定性工作 |
| P5 | ❌ 不加 | pytest 全绿 = gate，客观 |
| P6 | ✅ 已有（双向后） | 一致性是定性工作 |
| P7 | ❌ 不加（但加用户验收） | 发布准备由 gate 判定 |

**修改文件**：§3.2 表，P3 行和 P1 前行标注为「gate 自检，不加强制评审」。

### 发现 3（🟡 中危）：P3「test-designer 互评」在单执行者模型下不可行

**位置**：§3.2 表 P3 行

**现象**：
```
P3 | test-designer 互评 | 测试用例是否覆盖 AC | 缺失
```

v3 的 P3 只有**一个** test-designer。不存在「另一个 test-designer 来做互评」。

**机理分析**：v3 的执行角色是单人模型——每个阶段一个执行者，没有平行冗余。互评需要至少两个独立 test-designer，这在 v3 的资源模型（按需派发 subagent）下不成立。

**整改建议**：P3 的质量把关改为「gate 自检 + 主 Agent 抽查」：

```
P3 gate 自检：
- 测试用例数与 P1 问题数一致
- 每个 P1 问题有对应 TC
- 测试文件可被 pytest 收集
- TDD 红灯为 assertion failure（非 error）

主 Agent 抽查：随机选 1-2 个 TC，读测试代码确认其验证了 AC
```

## 三、设计亮点

### 1. 三层质疑概念清晰

L1（阶段内评审）→ L2（阶段间质疑上游）→ L3（需求层质疑原始需求）的分层递进逻辑优雅。每一层有明确的触发条件和上溯动作。

### 2. 用户介入清单精准

6 条「何时必须用户介入」覆盖了自动化工作流的真实边界——不仅限于「改不动了」，还包括业务决策、外部资源、安全合规。这比简单的「重试超限就 PAUSED」范围全面。

### 3. 与 T003 关系标注诚实

§七明确列出了本方案与 T003 的重叠和新增，不假装独立，不重复造轮子。
