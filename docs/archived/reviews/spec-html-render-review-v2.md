## 本轮新发现问题

### 🟠 NEW-1：§3.6 安全风险表的"后端 CSP 收紧"说法在 Blob URL 下是错的

**位置：** §3.6 安全风险记录

**问题：**

当前写法：
> iframe 内 fetch 外泄 → ⚠️ 已知，接受 → 后续可通过后端 CSP 收紧

这句话在 Blob URL 场景下**不成立**。Blob URL（`blob:null/...`）不经过后端，后端下发的 HTTP 响应头（包括 CSP）对 Blob URL iframe 内的网络请求**完全无效**。

"后续可通过后端 CSP 收紧"会给将来读这份文档的人留下错误的安全预期，可能在某次"加固"时发现根本没效果。

**修复（一句话）：**

将该行改为：

> iframe 内 fetch 外泄 → ⚠️ 已知，接受（结构性限制：Blob URL 不经过后端，后端 CSP 无法管控 iframe 内请求，无后续收紧路径）

**建议在进 P2 前改掉。**

---

### 🟡 NEW-2：P1 测试项"CSP 继承测试"定位不清

**位置：** §6 P1 测试设计

**问题：**

当前 P1 有一项：
> CSP 继承测试：Blob URL iframe 在父文档有 CSP 时的行为

Blob URL 不继承父文档 CSP 是 spec 的**已知选型依据**，不是待探索的未知行为。把它放在 P1 测试里，实现者会困惑："这是验证还是探索？结论已经写在 §3.2 了，还要测什么？"

**修复：** 移到 P3，改写为：

> 回归验证：Blob URL iframe 确实不受父文档 CSP 限制（验证行为与 §3.2 选型理由一致）

---

### 🟡 NEW-3：DOMParser 相对路径检测的作用域边界未定义

**位置：** §3.4

**问题：**

§3.4 写了"DOMParser 解析 HTML 内容，统计相对路径引用数量"，但没有说明：

1. 检测范围是 `<head>` 还是全文档？（`<body>` 里的 `<img src="./pic.png">` 也是相对引用）
2. JS 动态生成的资源（`document.createElement('img')` 或 `fetch('./data.json')`）无法被 DOMParser 检测到，这是已知限制，但 spec 没有明说

不说清楚边界，实现者要么尝试检测 JS 内容（过度实现），要么漏掉 `<body>` 里的静态引用（实现不足）。

**修复：** §3.4 检测说明后加一句：

> 检测范围为静态 HTML 的属性引用（`href` / `src` / `action`），不覆盖 JS 动态创建的资源请求。此为已知限制，可接受。

---

### 🟢 NEW-4：§7"不在本期范围"缺一条结构性限制

**位置：** §7

**问题：**

§7 列了"后端 CSP header 加固"作为后续优化项，但结合 NEW-1，Blob URL iframe 内的网络层管控是结构性限制，**不是"后续可做"的优化项**。把它完全不提，或放在"后续优化"里，会让人误以为这是个待解决的 TODO。

**修复：** §7 加一条：

> Blob URL iframe 内请求的网络层管控（结构性限制，无可行方案，已在 §3.6 记录为接受风险）
