# PeekView 产品与竞品分析报告

> 与上一份报告的关系：上一份报告的重点是 agate 研发流程（P0–P8 协议本身的失效模式）。**那是开发这个项目用的支撑工具，不是项目本身**。这一份完全换焦——只谈产品：它是什么、市场上有谁在做同样的事、打得过打不过、值多少、往哪走。
> 数据来源：仓库实测（代码/API/MCP 工具/前端视图）+ 2026-07-21 当天的公开市场检索（竞品信息按检索时间点为准，这个赛道目前变化很快，多个竞品几周前才上线）

---

## 一、产品是什么——不看战略话术，只看交付物

先剥掉"Agent 可写的高质量可分享渲染器"这句定位语，看这个产品实际交付了什么：

- **CLI**：`peekview create/get/list/delete/serve`，pipx 一键安装，本地起服务。
- **MCP Server**（独立 npm 包）：故意只暴露 **5 个工具**——`createEntry`、`getEntry`、`listEntries`、`deleteEntry`、`publishFiles`。没有 `updateEntry` 工具（更新能力只在 REST API 层，MCP 层不暴露）。
- **渲染前端**：Vue3 单页应用，5 个主视图（Landing / EntryList / EntryDetail / ApiKeyList / NotFound），支持代码高亮（Shiki 100+ 语言）、Markdown（含 T059 新增的 KaTeX/脚注/任务列表/上下标）、Mermaid/PlantUML 图表、HTML 沙箱 iframe、图片、多文件树、全文搜索、分享链接、暗色模式。
- **Agent 读取接口**：`GET /api/v1/entries/{slug}/raw`，返回结构化 JSON（文本文件带 `content` 字段，二进制带 `file_url`）。

**这是一个很小的产品，功能集合克制到近乎苛刻**——MCP 工具只有 5 个，且刻意不给"更新"能力（这是一个值得注意的产品取舍，见 §五）。这种"少即是多"的克制本身是一种产品哲学的体现，不是功能不全，需要在评价时区分开。

## 二、竞品全景——这是本次分析的核心增量

先说结论：**PeekView 自己的战略文档（2026-06-29 校准）判断的竞争格局已经过时了**。文档当时认为"直接威胁只有 Claude Code Artifacts 一家，PreviewShip 算轻量替代"。但过去几周，**"把 AI Agent 产出的东西发布成可分享链接"这件事本身已经长成了一个独立的、有真实资本进入的赛道**，参与者远不止两家。

### 2.1 新赛道：Agent 产出外部分享工具

背景是同一个：Claude Code Artifacts（以及 OpenAI Codex Sites）解决了"Agent 产出渲染页面"，但**分享范围卡在组织内部**——Claude Team/Enterprise 版的 Artifact 完全不能公开分享给组织外的人；Free/Pro 版则只能"完全公开"，没有中间态（不能"只给我这家客户公司的人看"）。这个缺口催生了一批专门做"外部分享"的工具，其中一家（display.dev）**已经完成 47 万欧元融资（2026-07-07，Tech.eu 报道）**，说明这不是几个玩具项目，是有真实商业信号的赛道。

| 工具 | 定位一句话 | 自托管 | 版本化/持久链接 | 访问控制颗粒度 | Agent 可结构化读回 | 跨 Agent 中立 | 定价 |
|---|---|---|---|---|---|---|---|
| **PeekView** | Agent 写、人看、Agent 也能读的自托管渲染器 | ✅ 完全自托管 | ⚠️ 覆盖式更新，无版本历史 | 公开/私有/分享 token（三级） | ✅ `/raw` 结构化 JSON，专为 Agent 设计 | ✅ 任意 MCP client | 免费（自托管成本自理） |
| **display.dev** | 按"查看者公司 SSO"网关分享，而非公开或锁组织 | ❌ 纯 SaaS | ✅ 版本历史 + 评论可被 Agent 读取并据此重新发布 | ✅ 公开/公司域 SSO/具名邮箱三级，比 PeekView 更细 | ❌ 只做人看的页面，无结构化读回接口 | ✅ Claude Code/Codex/Cursor 等 | 免费层 10 个 gated artifact；Pro 团队打包价 €49/月 |
| **Stacktree** | 面向 Agent 工作流的隐私优先分享，支持自托管 | ✅ 部分：付费一次性 $999 可拿到自托管构建；标准版是托管 SaaS | ✅ `update_site` 原地替换同一 URL | 密码/邮箱域网关/端到端加密/自定义域名 | ❌ 只托管静态 HTML，无 Agent 读回 API | ✅ | 分层定价，自托管为一次性高价 |
| **ShareDuo** | 贴 HTML 秒出公开链接，带密码/过期/统计 | ❌ | ❌ 链接约 30 天过期，非持久 | 仅密码 | ❌ | ⚠️ 偏 Claude 场景 | 免费 |
| **PreviewShip** | 前端构建产物/HTML/Markdown 一键发布预览链接 | ❌ | ⚠️ 项目可复用同一固定链接，但语义是"预览"而非"发布记录" | 仅公开/密码 | ❌ | ✅ 任意 MCP client | 免费层 15MB，Pro 分层 |
| **Handoff.host** | 面向"会持续更新的规格/报告"的持久版本化链接 | ❌ | ✅ 持久 URL + 版本历史 | 仅密码 | ❌ | ⚠️ | 免费层+未公开付费层 |
| **Claude Code Artifacts** | 平台内建，Anthropic 官方 | ❌ | ⚠️ 有限 | Free/Pro：全公开；Team/Enterprise：仅组织内，**无法对外分享** | ❌（沙箱 CSP 禁止外部网络请求，谈不上"其他 Agent 读取"） | ❌ 仅 Claude | 含在订阅内 |
| **PrivateBin / Hastebin 等传统自托管 pastebin** | 自托管、加密、一次性阅读 | ✅ | ❌ | 密码/端到端加密 | ❌ 无 MCP/Agent 接口 | ✅（但不是为 Agent 设计） | 免费开源 |

### 2.2 这张表说明了什么

**PeekView 仍然独占的一格，且是全表唯一一格**：`/raw` 结构化 JSON 供其他 Agent 程序化读取。表里其余 7 家竞品全部是"人看"导向的静态 HTML 托管——没有一家提供"另一个 Agent 通过标准接口拿到结构化内容"的能力。这是 PeekView 定位文档里"Agent 也能读"这半句话**目前在整个已知竞品里都找不到对手**，应当被更大声地强调，而不是被"渲染质量"这类更容易被抄的能力盖过。

**PeekView 已经被追平、甚至被反超的一格**：**持久链接 + 版本化**。PeekView 战略文档把这项列为"重活层，等数据触发才做"，理由是"当前实践中几乎没有更新"。但 display.dev、Stacktree、Handoff.host **三家竞品已经把"内容可更新、URL 不变"做成了标配功能**（`update_site` 原地替换、版本历史、评论驱动重新发布）。这意味着：**"要不要做版本化"这件事，市场已经替 PeekView 做出了判断——多个新玩家把它当成起步必需品，而不是需要数据验证的可选项**。PeekView 的"数据驱动，不预设"原则在这里可能正在变成"因为不看别人在做什么，所以看不到风险"。这是本报告与上一份报告"数据驱动纪律值得肯定"这个判断的一个重要修正——**数据驱动不能只看自己的埋点数据，还要看竞品已经验证过的行为**。

**PeekView 明显落后的一格**：访问控制颗粒度。display.dev 做到了"按查看者公司域 SSO 网关"，Stacktree 做到了"端到端加密+自定义域名"，PeekView 目前只有"公开 / 私有 / 分享 token+cookie"三级，没有"只给某个公司域的人看"这种中间态。对于"发给客户看但不想公开"这个高频场景，PeekView 现在的方案（发一个 token 链接）在体验上比 display.dev 弱——token 链接一旦转发给第三方就等于泄露，没有身份核验。

**PeekView 仍然保留但优势正在缩小的一格**：自托管。Stacktree 已经推出了"一次性付费拿自托管构建"的选项，说明"自托管"这个卖点正在被高价值细分市场的竞品跟进，不再是 PeekView 一家独有，只是价格模型不同（PeekView 免费开源自托管，Stacktree 是 $999 一次性）。

### 2.3 对护城河判断的直接影响

上一版战略文档的"四点交集"论证（跨厂商中立 + 公开分享 + 持久快照 + 自托管）在 2026-06-29 写成时是成立的。**基于本次检索到的最新数据，这个交集论证需要更新为五点，且要把"Agent 可结构化读回"单独列出来，因为这才是真正没有人能抄的一点**：

- 跨厂商中立：仍然成立，但 display.dev/Stacktree/PreviewShip 同样做到了这一点，不再是差异化
- 公开分享（对组织外）：仍然成立，但 display.dev 做得更细（公司域网关），PeekView 在这一点上其实功能更弱
- 持久快照：**已经不再是差异化**，多家竞品做得比 PeekView 更完整（有版本历史）
- 自托管：仍然成立，但 Stacktree 已经开始跟进
- **Agent 可结构化读回**：**这是唯一在整个竞品列表里找不到对手的一点**，应该成为叙事的核心，而不是并列的第四点之一

**结论**：护城河依然存在，但支撑它的"哪一点最硬"已经变了。原文档把重心放在"自托管+公开分享+持久"的组合上，现实是这个组合已经被 Stacktree 之类的玩家开始逼近；真正独占的是"Agent 读"这条线——但 PeekView 自己的产品重心（T031 冷打开体验、T059 Markdown 扩展、T057/T058 分享面板重做）过去一个月几乎全部投入在"人看"这一侧的体验打磨，"Agent 读"这一侧（T032 埋点之后）反而没有新投入。**这是一个值得警惕的资源错配信号**：最容易被复制的能力（渲染质量、分享面板 UI）正在持续投入，最难被复制的能力（Agent 读回接口的丰富度、比如让读取方也能拿到语义化的元数据、引用关系）反而停滞在"埋了个点，没有下文"的状态。

## 三、产品质量评估（聚焦交付物本身，不谈开发流程）

### 3.1 渲染能力：产品最强的一块

Shiki 100+ 语言高亮、Mermaid/PlantUML 双图表引擎、KaTeX 数学公式、脚注/任务列表/上下标（T059）、HTML 沙箱支持 Three.js/Canvas/WebGL——这是一个非常扎实的渲染集合，覆盖了"Agent 产出的报告/代码/图表/交互演示"这几乎所有常见形态。放在竞品表里对比，**PreviewShip 只做静态构建产物，Tiiny.host 只做纯 HTML，ShareDuo/Handoff.host 基本是"贴 HTML 就完了"——没有一家在渲染深度上追上 PeekView**。这是 PeekView 产品叙事里"原则 4：渲染是产品的门面"兑现得最好的地方，建议继续投入，这条护城河目前扎实。

### 3.2 MCP 工具集的克制：优点，但有一个明显缺口

只暴露 5 个工具（不给 `updateEntry`）是一个经过深思的产品决策，减少了 Agent 选错工具的概率。但从竞品对比看，**display.dev 和 Stacktree 都把"内容更新、链接不变"做成了 Agent 可直接调用的能力（评论驱动重新发布、`update_site`）**，而 PeekView 的更新能力被有意排除在 MCP 工具外，只留在 REST API。这意味着：**如果一个 Agent 想"改一下刚才发布的报告"，PeekView 现在做不到——它必须删了重发,拿到一个新链接**，这与"持久快照"的叙事本身是有点矛盾的（详情见 §二 2.2）。这不是"该不该有 updateEntry 工具"的简单判断，而是**产品叙事（持久快照）和当前工具集（无更新能力）之间存在一条没有被文档正面回应的裂缝**。

### 3.3 前端完整度：够用，视觉细节返工频率偏高

5 个主视图覆盖了核心路径（列表/详情/API Key 管理/落地页），组件拆分合理（27 个组件，职责清晰：ShareDialog/OverflowMenu/FileTree/TocNav 等）。但从任务历史看，**分享面板这一个功能被重做了两次（T057→T058），移动端 header 被修复了两次（T049→T050）**——这不是说代码质量差，而是说明"UI 细节一次做对"的能力目前还不够稳，纯代码验证（vue-tsc/vitest）覆盖不到视觉回归，这一点在上一份报告里已经提过，这里从产品体验角度再次印证：**用户能感知到的是"这个链接打开体验好不好"，而这恰好是最容易被返工掩盖、最难靠单元测试保证的部分**。

### 3.4 安全与访问控制：够用，但颗粒度是短板

公开/私有/分享 token+cookie 三级模型，`max_views` 限次、`verify_share_token` 等实现基本合理（历史问题已修复）。但对照 display.dev 的"按查看者公司域网关"能力，**PeekView 目前没有任何"半公开"的中间态**——这在"发给客户但不想让全网看到"这个真实高频场景里是明显短板,而这个场景恰恰是 display.dev 融资时讲的核心故事。

## 四、定位重新评估

在看过完整竞品格局后，重新回答"PeekView 应该怎么定位自己"：

1. **不该继续把"持久快照/版本化"当作核心卖点**——这条已经被多个新玩家追平甚至做得更好，继续强调它只会让潜在用户去对比后发现 PeekView 反而是"覆盖式更新、无版本历史"的那个，处于劣势对比位置。
2. **应该把"Agent 可结构化读回"提到定位的第一位，而不是并列的四点之一**——这是目前唯一找不到对手的能力，值得优先投入（更丰富的 `/raw` 元数据、引用关系、也许支持 Agent 按内容语义查询而非仅按 slug 查询）。
3. **"自托管"仍然是真实卖点，但护城河变窄了**——竞品已经出现了"付费买自托管构建"的先例（Stacktree），说明市场认可这个卖点的商业价值，PeekView 作为"免费开源自托管"反而在这条线上有价格优势，可以考虑更明确地把这一点当成"对预算有限的重度自托管开发者"的定价叙事，而不只是一句"MIT 协议"。
4. **访问控制颗粒度需要补课**——不是为了追竞品功能列表，而是因为"发给特定一批人看、不公开"是一个真实产品需求，目前 token 链接的方案在体验和安全性上都弱于对手的"邮箱域网关"方案。

## 五、未来发展建议（按这次竞品分析重新排序，不同于上一份报告的建议清单）

**认清一件事**：这个赛道现在有资本进场（display.dev 融资），意味着**留给 PeekView 慢慢"用数据验证要不要做"的时间窗口在缩短**。以下建议按"对护城河影响大小"排序，而不是按"实现难度"排序：

1. **优先把"Agent 读"这条线做厚，而不是继续打磨"人看"这条线**——具体可以是：`/raw` 增加结构化的引用关系字段（这份内容引用了哪个 entry / 被哪个 entry 引用）、增加按语义/标签的 Agent 查询能力。这是唯一别人抄不走的方向，值得从"探针层"提升为"地板层"重新排期，哪怕探针数据还不够多——**因为这不是等数据验证要不要做，而是巩固现有优势**，性质和"总线愿景"那种要不要做新东西不一样。
2. **重新评估"版本化冻结"的决策**——不是说现在必须做完整版本历史，但至少应该做**"内容替换后不丢线程"**（比如替换时保留一个可选的历史快照，而不是硬覆盖），成本远低于完整版本系统，但能追平 Stacktree/display.dev 的基本预期，避免用户对比后觉得"这个工具连内容更新都要删了重发"。
3. **访问控制补一个"半公开"中间态**——哪怕只是"允许指定邮箱后缀"这种简化版本，也能补上目前 token 链接方案的体验短板，不需要做到 display.dev 那种 SSO 级别。
4. **继续投入渲染能力，但认识到这不再是可持续的差异化，只是及格线**——所有竞品都在补这条线，PeekView 现在领先，但领先幅度会被追平，不应该是未来资源投入的重点。
5. **T031（冷打开体验）确实该做，但优先级应该重新排序在"Agent 读"之后**——因为"人打开链接的第一印象"这件事，在这个已经拥挤的赛道里，是所有对手都在做的事，PeekView 在这上面投入再多，也很难做出别人做不到的差异化；而"Agent 读"这条线目前没人做，边际投入的差异化收益要高得多。

## 六、独立评审

**证据强度**：本节的核心论断（竞品格局、功能对比表）全部基于 2026-07-21 当天检索到的一手信息（含融资新闻、竞品官方文档/评测文章），不是转述训练数据里的旧印象——这一点很重要，因为这类"新兴 SaaS 赛道"信息老化极快，检索日期本身就是证据的一部分。功能对比表里的每一格判断都能对应到具体检索来源，可复核。

**局限与需要提醒的地方**：
1. 竞品对比信息来自第三方评测文章（如 display.dev 自己发布的"8 工具对比"），**这类文章通常是竞品自己写的营销内容，对自己有利的维度会被放大**（例如 display.dev 的文章把"公司 SSO 网关"列为最重要的评估维度，这恰好是它自己最强的一项）。本报告已经交叉参照了多篇独立来源（Peony、Stacktree 各自的文章）来降低单一来源偏差，但读者仍应该知道，如果要做正式的采购/竞争决策，建议直接去每个产品的官网核实定价和功能，而不是只依赖本报告转述的评测内容。
2. 没有找到 PeekView 定位文档里提到的 "TokDoc" 的独立信息源——检索未能确认这是一个公开可查的产品还是团队内部对某个工具的称呼，本报告对 TokDoc 的定位判断完全依赖 PeekView 自己文档里的描述，不构成独立验证。
3. 本报告的建议（尤其是"重新评估版本化冻结决策"）本质上是"看到别人在做就该跟"的论证，这类论证本身有风险——**竞品做了不代表是对的方向，也可能是竞品还没有经过市场验证的过度构建**。这一点在正式采纳建议前，应该由团队用自己的判断力核实，而不是照单全收。本报告的角色是提供一手竞品事实，最终"要不要跟"的判断权仍在团队。

**评审结论**：本节相对上一份报告的流程分析，证据来源更依赖第三方营销内容，建议在正式决策前对关键竞品（尤其是 display.dev、Stacktree）做一次直接的产品试用核实，而不是仅凭本报告的转述。
