# PeekView 下一步方向建议——以团队/作者视角

> 承接前两份报告(流程复盘、竞品分析)和刚才关于"资源错配 vs 产品节奏"的讨论。这份文档的立场从"外部分析者"切换到"如果我是 PeekView 团队,我会怎么排"。每个维度给出：现状 → 问题/机会 → 建议 → 优先级。优先级标注参考了上一轮讨论里校准过的原则——**区分"必须做的纠错"、"一次性低成本的结构决策(现在做便宜、以后做贵)"、"可选的持续性功能投入"三种性质,而不是一刀切按"重要不重要"排**。

---

## 一、先定调:三种性质的工作,排期逻辑不一样

在展开各维度之前,先把上一轮讨论里校准出的框架显式写出来,后面所有建议都会标注属于哪一类：

| 类型 | 定义 | 排期逻辑 |
|---|---|---|
| 🔧 纠错(correctness debt) | Bug、体验缺陷、安全漏洞——不管战略重心在哪都必须修 | 无条件优先,不占"选择投入哪里"的额度 |
| 🎯 一次性结构决策 | 现在做便宜、以后做贵的接口/schema/架构设计(不是运营推广) | 即使在早期阶段也该优先,因为窗口期是单向流逝的 |
| 🌱 持续性功能投入 | 需要用户规模/反馈才能验证价值的打磨型工作(UI 细节、渲染扩展) | 早期阶段边际收益低,可以放慢,不必现在铺开 |

下面每个建议都会标这三个标签之一。

---

## 二、Agent 访问格式与协议——这是当前最该优先的一类"一次性结构决策"

**现状**:`/raw` 返回结构化 JSON(文本 `content` 字段/二进制 `file_url`),MCP 只暴露 5 个工具(create/get/list/delete/publishFiles),无 `updateEntry`,无引用关系,无语义查询。

**机会**:上一份竞品报告已经确认——这是全表 8 个竞品里唯一没人做的能力。但"没人做"不代表"已经做到位了",现在的 `/raw` 只是**能读**,离**好读**还有距离。

**具体建议**(🎯 一次性结构决策,现在做便宜):
1. **给 entry 加引用关系字段**——`references`(这篇引用了哪些 slug)和 `referenced_by`(被哪些引用)。现在加是加一个字段;等发布量上去了再加,要么放弃回填历史数据,要么跑一次昂贵的迁移。这正是"现在便宜、以后贵"的典型例子。
2. **`/raw` 增加轻量语义元数据**——不是要做向量搜索,而是让创建时可选传入 `topic`/`kind`(如 `report`/`diagram`/`code-review`)这类结构化标签,给未来"Agent 按语义查询"留一个最小的挂钩点。现在这个字段几乎不花成本,以后想做检索却发现历史数据没有任何结构化标签,就要走"先污染时间长了再清洗"的弯路。
3. **`updateEntry` 是否要暴露给 MCP,需要一次正式决策,而不是沉默地不做**——目前的裂缝是:REST API 有更新能力,MCP 没有,而"持久快照"的产品叙事暗示应该有。这不是说现在就要做,而是**这个决策本身该被写成一份 P1 文档过一次评审**,哪怕结论是"明确不做,理由是 XXX",也比现在这种"没人提所以没做"的状态更负责任。

**不建议现在做的**(避免过度延伸到🌱持续投入区):完整的向量语义检索、Agent 间的评论/协作流(display.dev 那种"评论驱动重新发布")——这些需要真实的多 Agent 读取行为数据支撑,目前 T032 埋点数据是否显示这类行为还不清楚(见下一节),贸然做这类重功能会正好踩进"重活层"的坑,和团队自己写的两层纪律相矛盾。

---

## 三、探针数据闭环——比新功能更紧迫

**现状**:T032 埋点(entry_reads 表 + X-PeekView-Source header)v0.4.0 就上线了,过去两个多月没有看到任何"回头看数据说明了什么"的记录。

**建议**(🔧 纠错性质,不是新功能,是把已经做的事做完):这不是"要不要新增功能",是**把 T032 已经花的工时兑现成一个决策依据**。具体：跑一次数据导出,看看`entry_reads` 里有没有非创建者身份的读取记录、频率如何,哪怕样本很小,也该写一份"T032 数据回顾"贴进 roadmap。这件事的优先级应该高于任何新功能,因为它是判断"上一节的 Agent 读投入到底该加多重"的直接依据——现在等于是在没有仪表盘的情况下开车决定要不要加速。

---

## 四、商业化路径——基于上一轮讨论的具体化

结合 Tiiny.host($10k MRR/4年,纯 bootstrap)和 display.dev(€47万 pre-seed,靠团队履历不是收入)两个参照系,给出四条路径的具体排序建议,而不是简单列选项:

**如果目标是"项目能自我造血、不追求做大"**(🌱持续投入区决策,不紧急,但值得现在想清楚方向):
- 优先考虑"自托管付费支持包"(参考 Stacktree $999 一次性模式),而不是托管 SaaS。理由:这条路径**几乎不需要改动现有单机架构**,只是把"企业级功能+人工支持"打包卖给愿意付费的重度自托管用户,不需要解决多租户隔离、计费系统、内容审核这些托管 SaaS 必须解决的新问题。
- 不建议现在就做托管云版本正面对抗 PreviewShip/display.dev——那是一次业务性质的转型(从"卖软件"变成"运营服务"),现有团队的工程重心(渲染/流程自动化)和"运营一个 7x24 可用的多租户服务"所需要的能力(计费、滥用防护、客服)几乎不重叠,贸然做等于同时开两条新战线。

**如果目标是"保持纯开源,不追求商业化"**:也是完全合理的选择,不需要因为竞品在融资就产生"必须商业化"的压力——Tiiny.host 的案例恰恰说明这类工具真实的商业天花板本来就不高,选择不做商业化不是"错过机会",可能是"没有错过什么"。

---

## 五、渲染能力——从"核心投入"降级为"及格线维护"

**现状**:Shiki/Mermaid/PlantUML/KaTeX 已经是这个赛道里最强的一档。

**建议**(🌱持续投入区,建议主动降低权重):上一份报告已经指出这条线正在变成"人人都在补的及格线",不再是差异化。具体做法不是停止投入,而是**把验收标准从"持续加新格式"改成"响应用户报的具体渲染 bug",从主动开发转为被动维护**。省下来的带宽应该挪去二/三节的"Agent 读"结构决策和探针数据闭环。

---

## 六、访问控制与分享——一个中等优先级的真实缺口

**现状**:公开/私有/token+cookie 三级,没有"半公开"中间态。

**建议**(介于🎯结构决策和🌱持续投入之间,偏向🎯):不需要做到 display.dev 的公司域 SSO 网关那么重,但"允许限定邮箱后缀"这种轻量版本值得作为一次**接口层面的扩展**尽早规划——因为访问控制模型如果以后要改(比如从"token 是唯一凭证"扩展到"多种凭证类型并存"),现在的鉴权代码路径(Authorization header > Cookie > API key 三级优先级)如果没有为将来多一种"邮箱域校验"留扩展点,以后插入的改造成本会比现在设计时就把接口做成可扩展的形式更高。这属于"现在想清楚接口形状,不代表现在要把功能做完"的类型。

**版本化/内容替换**:上一份报告建议的"替换时保留可选历史快照(不是完整版本系统)"依然成立,建议归为🎯——现在的 `update_entry` 覆盖写入逻辑,如果以后要加"保留上一版",是在现有写入路径里加一个分支;等用户已经习惯了"更新即覆盖"的行为模式后再加,会有语义变更的兼容性问题(要不要给历史 entry 补历史快照、旧行为的用户会不会疑惑）。现在改比以后改成本低。

---

## 七、架构与存储

**StorageBackend 抽象**(🎯结构决策,优先于渲染类的持续投入):目前直接耦合文件系统。上一份报告已经指出这一点被长期搁置。给一个更具体的理由支撑现在做而不是以后做:**如果六、四节的建议(自托管付费支持包、面向企业自托管用户)被采纳,存储抽象几乎是前置条件**——重度自托管用户对存储位置的多样性需求(NAS/对象存储)会比现在假设的普通开发者用户更强烈,现在存储层和文件系统强耦合,以后要抽象出接口,需要动的代码面比现在先做好接口设计再实现要大得多。

**SQLite 并发边界文档化**(🔧纠错性质,是文档缺口,不是功能缺口):这不需要工程投入,只需要有人写清楚"这台机器大概能扛多少并发写入",让自托管用户有预期,现在没有文档本身就是一个该立刻补的空白,成本极低。

---

## 八、开源/闭源决策

**建议**:不建议现在讨论"要不要闭源"——目前的护城河(自托管+MIT协议)恰恰建立在完全开源之上,竞品里唯一提供类似自托管选项的 Stacktree 是靠"标准版闭源 SaaS + 高价一次性自托管授权"的模式,如果 PeekView 现在纠结开源/闭源这件事,等于在动摇自己现在唯一被验证过的差异化基础。如果未来要做 §四提到的"自托管付费支持包",完全可以在**开源核心不变**的前提下,把"支持/企业插件"部分做成闭源附加层,不需要牺牲现有核心的开源身份。

---

## 九、团队与流程(简要,呼应第一份报告)

不重复第一份报告的细节,只提炼一条和这次战略讨论相关的：**排期机制目前没有"给一次性结构决策留固定权重"这道工序**。active-tasks.md 目前是纯粹的任务流水线,没有区分"这是纠错""这是结构决策窗口期""这是可以慢慢来的打磨"。建议在任务创建的 P0-brief 阶段加一个简单的分类字段,这样"该不该现在做 Agent 读的 schema 设计"这类问题不需要每次都靠一轮完整的战略讨论才能达成共识,而是有一个日常就能对齐的排期规则。

## 十、总优先级清单(把以上综合成一份可以直接拿去排期的列表)

1. 🔧 **T032 探针数据回顾**——不是新工作,是把已经做的事写完,决定后续投入力度的前提
2. 🔧 SQLite 并发边界文档化——纯文档,零工程成本
3. 🎯 Entry 引用关系字段 + 轻量语义标签——现在加便宜,以后加要迁移历史数据
4. 🎯 `updateEntry` 是否暴露给 MCP 的正式决策(哪怕结论是不做)——消除产品叙事和实际能力之间的裂缝
5. 🎯 StorageBackend 接口抽象(设计,不必现在完整实现所有后端)
6. 🎯 访问控制模型预留"半公开"扩展点 + 内容替换保留历史快照的写入路径设计
7. 🌱 渲染能力从主动开发降级为被动维护
8. 🌱/商业 是否要做"自托管付费支持包"——不紧急,但值得团队现在有一次明确的立场(哪怕是"暂不做"),而不是含糊地悬着

## 十一、独立评审

**这份文档最大的风险**:它把很多建议标成"🎯一次性结构决策、现在做便宜以后做贵",这个论证模式本身很容易被滥用——几乎任何功能都可以被包装成"现在做便宜以后做贵"来插队。我在标注时尽量用了一个具体检验标准(是否涉及历史数据迁移、是否涉及已发布内容的兼容性),但读者(团队自己)在采纳这份清单时,应该对每一条都问一遍"这真的是接口设计问题,还是我在给一个我想做的功能找借口",这是这份文档结构性的自我局限,没法在写的时候完全消除,只能提醒使用时警惕。

**第二个局限**:§四商业化建议部分依然是基于两个案例(Tiiny.host、display.dev)的外推,样本量是 2,不构成统计意义上的行业基准,只能当作"两个真实参照点"而非"这个赛道的平均水平"。
