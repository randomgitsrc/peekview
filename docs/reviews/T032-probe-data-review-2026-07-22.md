## 结论

**零 Agent 通过 `/raw` 或 MCP 读取条目。**

所有 559 条记录都是人在浏览器里浏览（`discover` = 列表页, `read` = 详情页）。没有任何记录来自 MCP 客户端或 `/raw` API 端点。

这意味着：

1. **"Agent 可结构化读回"目前是零使用的独占优势。** 竞品报告说这是 PeekView 唯一没人能抄的能力——但如果没人用，独占就不等于护城河。

2. **"人看"是目前唯一被验证的使用场景。** 559 条记录中 484 条（87%）是非自读——确实有人在看，全是人在看。

3. **数据量太小，不足以支撑"Agent 读"的产品决策。** 5 个认证用户、27 个条目、3 周的数据，无法判断"Agent 读"是需求不存在还是产品没暴露。

## 建议

1. **T062（引用关系字段）优先级下调。** 在没有人用 `/raw` 之前，加引用关系字段是"为一个不存在的使用场景设计 schema"。
2. **"Agent 读"投入暂停。** 与其猜测 Agent 需要什么，不如先让更多人在浏览器里看到 PeekView 的内容。等使用量上去了，再回来看数据决定要不要投入。
3. **T031（冷打开体验）恢复优先级。** 87% 的读取是非自读的浏览器访问——"人打开链接的第一印象"比"Agent 怎么读"重要得多。

## 局限

- 3 周数据、5 个用户、27 个条目——样本量极小，结论方向性参考，不构成统计意义
- `/raw` 端点可能没有被任何外部系统知道——零使用可能是"没有需求"也可能是"没有发现"
- 埋点只记录了读取行为，没有记录"尝试读取但 404"的情况——无法判断是否有 Agent 尝试过但失败了
