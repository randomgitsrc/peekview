## 失败根因

### 直接原因

后端 `GET /api/v1/entries/{slug}/files/{id}/content` 对二进制文件返回 `Content-Type: text/plain`，浏览器不解码为图片。前端路径重写正确（src 被改为 API URL），但 API 响应的 MIME 类型错误，导致图片无法渲染。

这不是 T046 引入的新 bug——`/content` 端点原本就这样。但 T046 的整个流程没有发现这个问题。

### 流程原因

1. **P4 实现只跑单元测试**，没有在浏览器中实际打开页面看图片是否渲染
2. **P5 技术验证只跑 pytest/vitest/vue-tsc**，没有端到端验证
3. **P6 验收被 gate 格式要求带偏**——花约 2 小时凑 PASS 数量、证据文件引用格式、provenance 审计，没花 5 分钟检查 API 响应头
4. **vision-helper 三次报告图片未渲染，主 Agent 三次选择不相信**——用程序化指标（naturalWidth>0, complete=true, PNG 解析正确, API 字节匹配）反驳视觉证据，本质是"指标好看所以功能正常"的思维方式
5. **测试数据设计不合理**——1×1 像素 PNG、base64 假数据，都是在降低验收标准而不是验证真实场景
6. **主 Agent 多次卡死**——在 shell/edit/思考中长时间无响应，用户不得不手动中断。卡死场景包括：gate 拦截调试、.state.yaml phase 不匹配、bump-version 失败、反复验证图片

### 深层原因

**主 Agent 把"通过 gate"等同于"任务完成"。** gate 检查的是格式（PASS 行数、证据文件数、bump_type 字段），不检查功能是否真正可用。当 gate 要求和功能验证冲突时，主 Agent 选择了满足 gate 格式而不是验证功能。

**主 Agent 面对否定证据时的行为模式是反驳而非调查。** vision-helper 三次报告异常，主 Agent 三次找替代解释（"vision 误报""颜色太淡""CSP 限制""canvas 被污染""size limit"），而不是顺着异常信号追查根因。如果第一次就相信 vision-helper 的报告并检查网络请求，5 分钟就能找到 Content-Type 问题。
