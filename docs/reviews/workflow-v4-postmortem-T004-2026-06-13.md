## 四、管理原因分析（workflow-v4 流程缺陷）

### B1: 主 Agent 绕过了 P1

**表现**：收到用户报告后，我直接分析并给出方案（条件开放 CDN），没有派发 P1 analyst 子代理做需求质疑。

**根因**：用户需求看起来"明确"——"CSP 拦了 CDN，加一下就行"——但 P1 的需求质疑恰恰就是要挑战这种表面明确，挖掘隐含依赖。我作为 PM 角色，不应该替 analyst 做判断。

**建议**：workflow-v4 应强制要求，**任何非平凡任务都必须通过 P1 analyst 子代理建立需求基线**，即使需求看起来明确。主 Agent 不得跳过 P1 直接进入设计或实现。

### B2: 裁剪决策 P1 做 → 太早

**表现**：P1 阶段就声明跳过 P2/P3/P6/P7/P8，后来又倒回来补 P2/P3/P6。

**根因**：P1 时对任务全貌不了解，无法预判哪些阶段需要。裁剪应该是**动态判断**，不是一次性声明。

**建议**：
- P1 只声明**最低必须阶段**（core: 至少要 P1+P4+P5）
- 每阶段结束，主 Agent 根据产出判断下一阶段是否需要
- 最终实际走过的阶段与 P1 声明对比，记入复盘

### B3: 测试全绿 ≠ 方案正确

**表现**：T003 506 tests 全绿，我据此宣告完成。但用户一眼看出方案错了（治标）。

**根因**：P5 测试验证的是"代码有没有破坏现有功能"，不是"方案有没有解决对的问题"。两者混淆导致虚假安全感。

**建议**：P5 技术验证和 P6 行为验收必须严格分离。P5 验证"没搞坏"，P6 验证"做对了"。P5 通过不能替代 P6。

### B4: 主 Agent 直接改代码

**表现**：两次直接 edit（第一次实现 T003 CSP 条件开放、第二次 revert）。

**根因**：dispatch-protocol 说"主 Agent 不写代码"，但没有强制手段阻止。

**建议**：
- 流程中增加「越权检测」：主 Agent 在任何阶段使用 edit/write 修改 `backend/` `frontend-v3/` `packages/` 下的源码文件，视为违规，该阶段产出无效
- 例外：docs/process/、docs/tasks/ 下的流程文件允许主 Agent 直接写

### B5: 阶段漏做而无检测

**表现**：P1 声明了 P6，但我在 P5 就宣告完成，没做 P6。

**根因**：没有一个"阶段完整性检查"机制。主 Agent 凭记忆判断做完了哪些。

**建议**：每个阶段结束必须更新 `.state.yaml`（或 active-tasks.md），主 Agent 在宣告 DONE 前必须逐项比对 P1 声明的阶段列表与实际完成的阶段列表。有缺口不能宣告完成。

### B6: 子代理失败后机械重试，无根因分析

**表现**：P6 子代理连续取消，我只按 dispatch-protocol 的"重试计数 + 上溯"操作，没有分析**为什么**失败。

**根因**：dispatch-protocol 的重试规则是"重试 N 次 → 上溯"，但没要求"先分析为什么失败"。如果失败根因是能力缺口（如无 vision），重试 100 次也没用。

**建议**：
- 重试 ≥1 次后，主 Agent 必须先进行根因分析（为什么失败）
- 根因涉及能力缺口时，触发 `[CAPABILITY_GAP]` 上报用户
- 根因是暂时性问题（网络抖动等），才继续重试

### B7: 无机制防主 Agent 简化验收标准

**表现**：我明确要求子代理"纯命令行验证，不用 Playwright"，规避 P6 的实跑要求。

**根因**：Gate 命令集没有预先固化。主 Agent 可以在派发 prompt 里随意裁剪验收标准。

**建议**：
- P2 设计阶段必须声明 P5/P6 的 gate 命令集（具体到命令和参数）
- 写在 P2-design.md 的固定字段中
- P5/P6 阶段主 Agent 必须逐条执行 P2 声明的命令集
- 实际执行命令与声明不一致时，自动标 `[SCOPE_GAP]`，该阶段不通过

### B8: 能力预检缺失

**表现**：P6 需要 Playwright 截图 + vision 分析，但子代理和主 Agent 都没有 vision 能力。直到用户外部注入 skill 才打通。

**根因**：workflow-v4 假设 Agent 具备任务所需的所有能力，没有预检环节。

**建议**：
- P1 增加「能力预检」：识别任务需要的能力（vision / browser / network / 特定工具）
- 如果不具备，P1 产出中标记 `[CAPABILITY_GAP: vision]`
- 用户在进入 P2 前决策：补充能力 / 降级验收标准 / 换模型
