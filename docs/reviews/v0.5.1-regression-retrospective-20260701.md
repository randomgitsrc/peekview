# v0.5.1 回归复盘 — 空行塌缩与诊断流程失守

## 1. 时间线

| 时间 | 事件 |
|------|------|
| T045 发布 | v0.5.1 上线。Markdown 代码块 `.line { height: 1.6em }` 缺失（设计写了 `display:block`，漏了 `height`） |
| 用户反馈 | "代码一行、空一行、代码一行、空一行"（display:flex→block 导致的空行 bug）；"连续蓝色交错行"（空行塌缩 0 高度 + 配色策略丑） |
| 第 1 轮 | 改 `display:block` → `flex`。code.css 和 DiagramBlock 修复，MarkdownViewer 本就正确。**认为修完了** |
| 第 2 轮 | 改配色 5% → 8%/6%（灰度→蓝色调对比度）。用户：**"没让你提高对比度"** |
| 第 3 轮 | 改配色回到 5%，加 `.line:empty { transparent }`。**仍未定位到空行塌缩** |
| 第 4 轮 | 跑 `getBoundingClientRect().height` → 空行 h=0。发现 MarkdownViewer `.line` 缺 `height:1.6em`。修复 |
| 定位耗时 | 约 5 轮对话、多次 CSS 修改、2 次无意义的配色调整 |

## 2. 根因：为什么 agate 没有拦住

### 2.1 P6 证据质量：读源码 ≠ 实跑

T045 P6 验收的 B01 evidence：

> "Code: MarkdownViewer.vue:369-371 — `.line { display: block }`"

Verifier 确认了 `display:block` 写在源码里，就当"zebra 整行背景" PASS 了。没有做运行时验证——没检查 `.line` 的实际渲染高度、没测试含空行的代码。

**agate 的 P6 gate 要求** 是"BDD 每条有实跑结果"，但 gate 本身只检查 `exit 0`，不检查 evidence 是"读了源码"还是"跑了浏览器"。

### 2.2 hotfix 零流程

整个修复过程没有经过任何 agate 阶段：

- 无 P0-brief（连一句话的 root cause 确认都没有）
- 无 gate 命令（没在修改前跑 `getBoundingClientRect` 确认问题）
- 修改后只跑 build + 测试，没做 DOM 定量断言

如果有一个最小化的 hotfix gate，第 1 步就是"跑诊断命令确认根因"——`document.querySelectorAll('.line').forEach(l => console.log(l.getBoundingClientRect().height))`——空行 h=0 立即暴露，不会绕 4 圈。

### 2.3 用户反馈 → 先猜后测

用户给了精确行号（Python 8,9 / TypeScript 6,7,8 / Shell 2,3），并且在截图附件中提供了可见的证据。但我的响应模式是：

1. 用户说"连续蓝色" → 我猜"5% 透明度太低" → 改配色
2. 用户再说 → 我猜"空行有背景" → 加 `.line:empty`
3. 我又猜 → "display:block→flex 没生效" → 检查构建产物
4. 最后才跑诊断 → `h=0` → 定位根因

**每一步都没跑诊断命令。每次都在基于猜测修改代码。**

## 3. agate 改进建议

以下建议是通用的（非 CSS 特定），可融入 agate 协议本身。

### 建议 1：P6 evidence 分级

当前 P6 gate 只判断"evidence 存在"，不区分 evidence 类型。建议在 dispatch-protocol.md 中增加 evidence 质量要求：

| Evidence 等级 | 示例 | 可信度 |
|-------------|------|--------|
| L1 源码声明 | "Code: file.vue:123 — `.foo { color: red }`" | 低（只证明写了，不证明生效） |
| L2 静态分析 | CSS 优先级计算、color math | 中 |
| L3 运行时断言 | `getComputedStyle().color = rgb(255,0,0)`, `getBoundingClientRect().height = 22` | 高 |

对于 UI/渲染类 BDD，要求 evidence 至少 L3。Gate 判定时检查 evidence 是否包含运行时数据（含 `getComputedStyle` / `getBoundingClientRect` / pixel sample 等）。

### 建议 2：hotfix 协议

在 `WORKFLOW.md` 或 `LIMITATIONS.md` 增加 hotfix 流程：

```
hotfix 最小 gate:
  1. 一句话根因（跑命令确认，非猜测）
  2. 改动文件列表
  3. 验证：build + test + 针对根因的诊断命令（证明修复前失败、修复后通过）
  4. 人确认效果
  5. commit（wf(hotfix): ...）
```

hotfix 不要求完整 P0-P8，但必须有诊断和验证。

### 建议 3：bug 响应规则：诊断优先

在 `dispatch-protocol.md` 中增加一条：

```
收到外部 bug 反馈（用户报告 / SCOPE+ / 自己的视觉验证异常）→ 
  第一步：跑诊断命令获取定量数据（DOM 高度/颜色/display 值）
  第二步：根据诊断数据定位根因
  第三步：修改
禁止 猜测→修改→猜测→修改 的试错循环。
```

这条看似"软"建议，但可以放进 pre-commit hook 警告或 retry pattern 中——如果 retries[Pn] > 2 且 same prompt，触发诊断提醒。

### 建议 4：cross-artifact 一致性（已有迹象但未落地）

上一份复盘（t044-t045）第 8 节已经提到：

> "存在共享 CSS class 被多处修改 → 不可跳"

T045 的 `.line` 规则出现在 3 个文件：`code.css`、`MarkdownViewer.vue`、`DiagramBlock.vue`。P2 设计对 MarkdownViewer 只写了"新增 `.line { display: block }`"，没列出完整属性清单（包括 `height`）。结果只有 2 个文件有 `height`，MarkdownViewer 漏了。

上一份复盘的建议是"P1 gate 时声明 shared_styles，check-pruning.sh 校验"。但这个建议只覆盖了"是否跳 P7"，没覆盖"P4 产出的多文件一致性"。

建议扩展为**P5 gate 自动 consistency check**：

```bash
# 提取所有 .line 规则，对比属性白名单
agate-check-consistency ".line{display,height,min-width,...}" \
  code.css MarkdownViewer.vue DiagramBlock.vue
```

如果某个文件缺关键属性 → P5 gate fail。不需要等 P7。

## 4. 总结

| 我在这次违反的 | 如果 agate 有 X |
|-------------|---------------|
| P6 信了"读源码"当证据 | P6 evidence 分级：UI 任务必须 L3（运行时断言） |
| hotfix 零流程，改了 5 轮 | hotfix 协议：必须跑诊断命令确认根因 |
| 用户反复反馈，我反复猜测 | bug 响应规则：禁止试错，诊断优先 |
| 3 个文件里 2 个正确、1 个漏属性 | P5 gate CSS consistency check 自动发现 |

核心教训：**主 Agent 在每个阶段的 gate 判定中不能只确认"做了"，还必须确认"做到位了"。**"做了"是 subagent 产出文件存在；"做到位了"是运行时数据证明文件生效且正确。
