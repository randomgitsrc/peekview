# PeekView 定位深思：从"活/死"之争到 single-shot 现实

> 日期：2026-06-29
> 性质：战略定位分析，承接 Claude Code Artifacts 对比讨论
> 一句话主旨：PeekView 的战略在为一个还没出生的世界写功能清单；今天它的真身是一个 single-shot 的"agent 可写的高质量可分享渲染器"——而这个更小的真身，恰恰是 Artifact 杀不死的。

---

## 0. 这份文档要纠正什么

本文包含一次**对前序建议的自我纠正**。在 Artifact 对比中，曾建议 PeekView "补版本化 + 显式时间契约"。用"功能是否服务于真实存在的行为"这把尺子重新衡量后，该建议被判定为**超前**，在此降级。文档末尾的纪律——"让真实数据而非战略文档决定每一个重功能"——同样适用于本文作者给出的所有建议。

---

## 1. 起点：一个被说反结论的正确直觉

讨论的触发点是一个观察：

> "Artifact 像甩出一个窗口，能看到链接权限内的内容变化；而 PeekView 的链接是固定历史内容，不能变。"

这个直觉**方向对、结论需要修正**两处：

### 修正一：PeekView 的内容其实**能**变

`update_entry` 存在，文件能增删，`updated_at` 字段就是为更新而设。但它的更新方式是**原地覆盖、不留版本**——改了旧内容就没了，URL 不变。

所以真实的三态对比是：

| | 内容会变吗 | 旧版本还在吗 | URL 语义 |
|---|---|---|---|
| **Artifact** | 会（会话推进时实时变） | 在（每次 republish 记一个 version） | "看最新的实时窗口" |
| **PeekView 现状** | 会（owner 可覆盖） | **不在（覆盖即丢）** | **不明确** |
| **纯快照（直觉以为的）** | 不会 | — | "看一个冻结的历史" |

PeekView 现在卡在**最差的中间态**：内容可被悄悄改掉（所以不是可信引用），又不能 pin 到某版本（所以拿不回稳定性）。两头不靠。

### 修正二：真正的轴不是"活/死"，是"进程视图 vs 发布记录"

- **Artifact = 一个活进程的窗口**。价值绑在"会话还在跑"上。你看的是 agent **正在做**的事。
- **PeekView = 一次发布动作的记录**。"我做完了，这是结果。"持久、可公开、跨 agent、可归属。

**关键判断**：PeekView 在"活"这条轴上永远赢不了 Artifact——因为 Artifact 在会话里面，PeekView 在会话外面。追"活"就是去 Artifact 的主场正面打，没胜算。

---

## 2. 把观察推到底：PeekView 今天是个 single-shot 产品

用户的进一步判断——"PeekView 服务的根本没有这种长期会话"——比表面更狠。诚实看 PeekView 今天的真实使用形态：

> **生成 → 发布 → 查看 →（也许）丢。一次性，一发即走。**

拆开三条：

1. **没有会话**（用户已确认）；
2. **没有多方读取**——"agent B 读 agent A 的产出"是战略文档的设想，不是正在发生的事；
3. **实践中几乎没有更新**——`update_entry` 在代码里，但有几个人真去改一个已发布的 entry？

三条加起来：**PeekView 今天是个 single-shot 产品。**

---

## 3. 暴露出的核心问题：战略与现实之间有一条裂缝

**战略文档描述的**：AI Agent 输出基础设施、多 agent 内容总线、agent 之间内容流转、可追溯性——一个丰富、持续、多方参与的内容生态。

**实际在跑的**：一个 agent 生成一坨东西、发布、人看一眼。

> 战略描述的是一种**还没发生的行为**，产品却已经按那个行为在规划功能。

这条裂缝是理解之前所有困惑的钥匙。"/embed 没场景"、"live 没会话"——这些不是孤立的功能质疑，而是**同一条裂缝在不同位置的投影**。每次撞见的，都是"我们在为不存在的行为造基础设施"。

---

## 4. 用同一把尺子量四个功能（包括自我纠正）

把"它服务的行为现在是否存在"作为统一判据：

| 需求 | 它服务的行为 | 行为现在存在吗 | 该不该现在做 |
|---|---|---|---|
| **live 更新** | 长期会话观察 | 不存在（已确认） | ❌ 不做 |
| **版本化 / 时间契约** | 内容被改写、多 agent 引用 | 不存在 / 设想 | ⏸ 不做，观察 |
| **/embed iframe** | 内容嵌进宿主页面 | 不存在 | ⏸ 不做，观察 |
| **多 agent 总线（方向1）** | agent 互相读取 | 设想 | 🔍 不直接造，留探针 |

四个里有三个是在为没发生的事造东西。这不是巧合，是那条裂缝的四个投影。

### 4.1 自我纠正：版本化也是超前的

前序讨论曾把"版本化 + 时间契约"说成该补的事。用上表的尺子量自己：

- 版本化解决"内容被改写时引用还能否稳定"和"多 agent 引用时钉版本"；
- 但内容实践中**不被改写** → 引用稳定性问题不发生；
- 多 agent 总线**不存在** → 钉版本是设想需求。

所以版本化和 /embed 一样，是为不存在的行为造的基础设施。**唯一勉强成立的角落**：T027 刚上分享链接，"分享出去后内容若被改，对方看到的和我不一样"这个一致性问题在 single-shot 下也成立。但更新本就罕见，角落低风险。

**结论：版本化降级为"观察项"，与 /embed 同等待遇。**

---

## 5. 好消息：剥掉设想层后，真身依然站得住

把战略的设想层剥掉，PeekView 今天的真身是：

> **一个 agent 能写入的、渲染质量很高的、可分享的链接。**

听起来比"基础设施"小很多。但关键好消息是——**这个更小的产品本身就站得住，且不靠总线愿景也能扛住 Artifact：**

| | Artifact | PeekView 真身 |
|---|---|---|
| 公开分享 | 不能（org 内） | **能**（T027） |
| 跨厂商 | 单一厂商 | **跨**（任何 agent） |
| 持久性 | 会话绑定 | **持久快照** |
| 自托管 | 否 | **是** |

"跨厂商 + 公开分享 + 自托管 + 持久渲染"这个交集，**就算多 agent 总线永远不发生，依然是一个真实的、Artifact 杀不死的小生态位**。

### 战略含义（重要）

> **总线愿景是 upside，不是地板。**

PeekView 不需要总线来证明自己有存在理由。所以它**不该把 roadmap 押在总线上**——该把地板做到极致，然后用便宜的方式去探总线到底会不会来。

---

## 6. Roadmap 该被什么驱动：两层，纪律不同

### 地板层（现在就做——因为这是真实在用的）

把 single-shot 那条路做到无可挑剔：

- **冷打开一个链接的体验**：加载速度、移动端、"这是什么 / 谁发的 / 什么时候"一眼清楚。这是真实高频路径，值得打磨。
- **渲染质量**：已经强（Shiki 100+ 语言 / Markdown / Mermaid / PlantUML / HTML sandbox），继续。
- **OG 预览卡片**：唯一一个 embed-相邻、因 T027 而**刚刚**变得有点合理的东西。但守纪律——T027 让分享变得**可能**，不等于分享正在**发生**。先埋点看链接到底有没有在被传出去，再决定。

### 探针层（不造重的，只留可观测的最小原语）

总线行为会不会冒出来，取决于一件事——**有没有 agent 真的去读另一个 agent 的产出**。

- 这条路径技术上已存在：MCP `getEntry` / `listEntries` 工具都在（`packages/mcp-server/src/tools/`）。
- 但**没有任何读取埋点**：T027 的 `view_count` 只服务分享链接，不区分人/agent，也不在 MCP 读取路径上。所以你**不知道**这条路径有没有被用。
- **该做的不是**现在去加强检索、加 `parent_slug`、做语义查询（那是假设行为存在）；
- **该做的是**给这条路径埋一个最小探针：**有没有非创建者的身份（尤其是 agent / 不同 API key）在读 entry？读的频率？**

判据：
- 半年数据都是零 → 总线愿景从战略文档**降级**；
- 开始出现信号 → 那时再**按真实形状**去加强它（检索、引用关系、语义查询）。

### 重活（一个都别提前）

版本化 / embed / live / 时间契约 / 引用关系图谱——**等数据显示出需要它们的行为，再做**。

---

## 7. 决策一览表

| 层 | 项目 | 动作 | 触发条件 |
|---|---|---|---|
| 地板 | 冷打开链接体验（速度/移动端/元信息） | **现在做** | 已是高频真实路径 |
| 地板 | 渲染质量 | **持续** | — |
| 地板 | OG 预览卡片 | 先埋点，后做 | 数据显示链接在被传出去 |
| 探针 | MCP 读取路径埋点（谁在读、读频率） | **现在做（轻）** | 这是判断总线的唯一信号源 |
| 重活 | 版本化 / 时间契约 | 冻结，观察 | 出现"内容被改写"或"多 agent 引用"行为 |
| 重活 | /embed iframe | 冻结，观察 | 出现"链接被贴进宿主页面"行为 |
| 重活 | live 更新 | 不做 | （会话观察是 Artifact 主场） |
| 战略 | 多 agent 总线（方向1） | 不直接造，看探针 | 探针出现非创建者读取信号 |

---

## 8. 一句话总结

> 对 live 的判断是对的，但它指向的不止是"live 不做"——它指向"PeekView 的战略在为一个还没出生的世界写功能清单"。今天的 PeekView 是个 single-shot 的"agent 可写的高质量可分享渲染器"，这个更小的真身恰恰是 Artifact 杀不死的。所以正确的纪律不是"按总线愿景补功能"，而是 **把 single-shot 地板做到极致 + 给总线留便宜的探针 + 让真实数据（不是战略文档、也不是评审建议）决定每一个重功能要不要做**。

> 用户对 /embed 用的那把尺子，应该量到整份战略上——包括量到本文给出的每一条建议。

*本分析基于 2026-06-29 的 PeekView 仓库（v0.3.0，含 session 泄漏修复 commit 656ca5a8）。代码事实（update_entry 覆盖式更新、MCP 读取路径无埋点、view_count 仅服务分享链接）已核对。战略判断为独立观点，建议与项目方实际数据交叉验证后采纳。*
