# Workflow v3 主动批判性评审

> 评审框架：gstack（/review 偏执 Staff Engineer + /plan-eng-review 工程经理）
> 日期：2026-06-12
> 评审方式：跳出"修别人提的问题"模式，主动以攻击者视角通读全貌找结构性缺陷
> 前置：三轮评审（自评审 + round2 + 文档体系）均已闭环，本轮找前面都没发现的问题

---

## 评审定位

前三轮评审都聚焦"文档写得对不对、字段全不全"。本轮换个视角：**假设 v3 真的跑起来了，跑 10 个任务、跑很久、跑复杂任务，会在哪里塌？** 找的是结构性缺陷，不是措辞问题。

找到 5 个之前所有评审都没碰到的真问题。

## 问题 1（🔴 结构性）：主 Agent 自身的上下文增长无界——"常数级"是错觉

**v3 反复声称**（loop-orchestration.md:126）：
> 主 Agent 的上下文增量是常数级（几行摘要 + 当前状态），不会触发压缩

**这个判断只对"单个任务"成立，对"主 Agent 的生命周期"不成立。**

拆开看：
- 一个任务 7 阶段 = 7 行摘要 ✓（单任务确实是常数级）
- 但主 Agent 不是只跑一个任务。它跑完 T001 跑 T002，跑 10 个任务就是 70 行摘要 + 10 次状态读取 + 每次派发的 prompt + 每次返回校验的记录
- **主 Agent 的上下文随"它经手的任务总数"线性增长，不是常数**

更严重的是：每次派发，主 Agent 上下文里留下的不只是"7 行摘要"。还有：
- 它读 active-tasks.md 的内容（每轮都读，累积）
- 它发出的派发 prompt（每个几十行）
- subagent 返回的校验过程
- 门槛判定的推理

跑久了，主 Agent 自己会触发压缩——**这正是 v3 要解决的问题，只是从"单会话内"转移到了"主 Agent 长期运行"层面。**

**v3 给子 Agent 设计了上下文隔离，却没给主 Agent 自己设计上下文管理。**

### 修正建议
1. loop-orchestration.md 明确："常数级"的限定词改为"单任务内常数级，主 Agent 长期运行仍需上下文管理"
2. 设计**主 Agent 的上下文回收**：每完成一个任务（到 DONE），主 Agent 应该能"忘掉"该任务的所有中间摘要——因为状态已落盘，记忆里不需要保留
3. 利用 state-machine 已有的"每轮重读 active-tasks.md"：主 Agent 理论上可以**无状态化**——每轮只依赖文件，处理完一步就可以丢弃本轮的工作记忆。把这个"主 Agent 尽量无状态"作为显式原则写进文档

## 问题 2（🔴 结构性）：评审打回后，architect 拿不到评审意见

**dispatch-protocol.md:230 的 rejected 处理**：
> rejected → 重试 architect（retry_count=1），带上评审意见

**"带上评审意见"是一句空话——评审意见在 P2-review.md 里，怎么"带上"？**

两种可能，都有问题：
- **可能 A**：主 Agent 读 P2-review.md 全文，塞进重试 prompt → 违反"主 Agent 不碰文件全文"铁律，且评审意见可能很长，污染主 Agent 上下文
- **可能 B**：让 architect 自己读 P2-review.md → 文档没说，而且 architect 角色定义的"输入"里没有 P2-review.md

**这是一个断链**：评审产出的意见，没有明确的路径回流给执行角色。重试时 architect 不知道上次为什么被打回，很可能产出同样的东西，再次被打回，直到 retry 耗尽 PAUSED。

### 修正建议
明确"评审意见回流"机制：
- rejected 时，主 Agent 的重试派发 prompt 里加一行："上次方案被打回，评审意见见 docs/tasks/Txxx/P2-review.md，请先读取该文件了解问题再修正"
- architect 角色定义的"输入"补充："重试时额外读取上一轮的 review 文件"
- 这样评审意见通过**文件路径**回流（符合"只传路径"原则），architect 自己读，主 Agent 不碰全文

## 问题 3（🟠 能力缺失）：完全没有并行能力，但有些阶段天然可并行

v3 是严格串行的 P1→P2→...→P7。但真实任务里存在天然并行的场景：

- **多个独立问题的实现**：P1 定义了 3 个互不依赖的问题，P4 实现时本可以派发 3 个 implementer 并行做
- **多角度评审**：P2 的方案可以同时派 plan-eng-review 和 cso 并行评，而不是串行等两次
- **多文件的验证**：P5 验证多个模块可以并行

业界框架（LangGraph）的核心卖点之一就是<并行执行>。v3 完全放弃了这个能力，所有 subagent 串行派发。对大任务，这是显著的效率损失。

### 修正建议
不必现在就实现，但应该：
1. 在 loop-orchestration.md 标注"当前为串行执行，并行是已知的未实现能力"
2. 识别哪些阶段可安全并行（无数据依赖的同阶段多 subagent）
3. 注意并行的代价：多个 subagent 同时写文件、同时 commit 会加剧 git 冲突（git-integration.md 已提到的问题会放大）。并行需要配套的文件隔离（每个 subagent 写不同文件）

## 问题 4（🟠 协调缺失）：评审角色的选择是"主 Agent 判断"，但判断依据不可判定

role-system.md:166：
> 主 Agent 根据任务内容判断需要哪些评审角色，可以串联多个

**"根据任务内容判断"是模糊的，和 v3 自己批判 v2 的"模糊指令"是同一类问题。**

主 Agent 怎么知道 P2 该用 plan-eng-review 还是 plan-ceo-review，还是都要？怎么知道 P4 要不要加 cso？文档给了一个决策表（role-system.md），但那是"涉及架构→plan-eng-review"这种需要主 Agent 理解任务性质的判断。

LLM 做这种判断不稳定——同一个任务，这次派了 cso，下次可能忘了。**门槛判定 v3 要求"可判定"，但评审角色选择却留了模糊判断。**标准不一致。

### 修正建议
- 把评审角色选择**显式写进任务的 P1 或 P2 产出**：P1 分析师在定义问题时，顺便声明"本任务涉及 [安全/架构/前端/产品]"，主 Agent 据此机械地映射评审角色（涉及安全→必加 cso），不靠临场判断
- 或在 task 的 Header 里加 `review_roles: [plan-eng-review, cso]` 字段，由 P1/P2 填写，主 Agent 照着派

## 问题 5（🟡 现实缺口）：嵌套深度限制与"执行角色需要再派子 Agent"的冲突

dispatch-protocol.md:200 提到 Codex 的 `max_depth 默认 1`（禁止深层嵌套）。

但有些执行角色天然需要再派子 Agent：
- architect 设计大方案时，可能想派一个 explore subagent 去调研现有代码
- implementer 实现复杂功能时，可能想拆成多个子任务

**max_depth=1 意味着：被主 Agent 派发的 subagent，不能再派发自己的 subagent。** 这限制了执行角色处理复杂阶段的能力——它只能自己一个上下文里硬扛，又回到了"单 agent 上下文可能爆"的问题，只是下沉了一层。

### 修正建议
- 明确 v3 的层级约定：主 Agent（L0）→ 执行/评审 subagent（L1），不依赖 L2
- 如果某阶段复杂到 L1 subagent 扛不住，应该在 P1/P2 阶段就**拆成多个任务**（Txxx-a, Txxx-b），而不是靠深层嵌套。即"用任务拆分替代 agent 嵌套"
- 这个约定写进 README，避免实际使用时 L1 subagent 试图再派发而撞 max_depth 限制

## 没问题的部分（确认 v3 的扎实之处）

1. 派发三铁律 + 返回校验：闭环完整，验证报告已证明可用
2. 状态落盘 + git 持久化：抗中断设计正确
3. 方法 B（prompt 注入角色）：验证报告证明它绕过了 issue #29616，是对的主路径
4. 门槛可判定（status/failed）：方向正确（除了问题 4 的评审角色选择例外）

## 评分与优先级

| 问题 | 严重性 | 本质 |
|------|--------|------|
| 1. 主 Agent 上下文无界增长 | 🔴 结构性 | v3 给子 Agent 隔离了上下文，却没管主 Agent 自己 |
| 2. 评审意见回流断链 | 🔴 结构性 | rejected 重试是空转，architect 拿不到打回原因 |
| 3. 无并行能力 | 🟠 能力缺失 | 大任务效率损失，但非阻塞 |
| 4. 评审角色选择不可判定 | 🟠 协调缺失 | 与 v3 自己的"可判定"标准矛盾 |
| 5. 嵌套深度冲突 | 🟡 现实缺口 | 复杂阶段 L1 扛不住时无解，需用任务拆分替代 |

**最该先修：问题 1 和问题 2。** 这两个是结构性的——problem 1 让 v3 在长期运行后重蹈上下文爆炸（只是换了层面），problem 2 让评审打回机制实际空转。两个都直接影响 v3 能否"真的有效运行"，而不只是"文档上看起来对"。

问题 3/4/5 是改进项，可在实际使用中按需推进。

## 元评注

前三轮评审都没发现这 5 个问题，因为它们都在"文档内部一致性"层面工作（字段全不全、定义一不一致）。这 5 个问题需要的是**把 v3 当成一个要长期运行的系统去压力测试**的视角——跑很久（problem 1）、跑失败路径（problem 2）、跑大任务（problem 3/5）、跑多样任务（problem 4）。

这印证了一条评审原则：**文档自洽 ≠ 系统可用。** 自洽性评审找的是"写错了什么"，压力测试视角找的是"没设计什么"。后者往往是更致命的缺陷。
