# Workflow v3 主动批判性评审

> 评审框架：gstack review（/review 偏执视角 + /plan-eng-review）
> 日期：2026-06-12
> 评审对象：workflow-v3 全量（2497 行，含 L1/L2/L3 迭代机制）
> 性质：主动找结构性缺陷，非响应外部评审。带"这套东西真跑起来会在哪炸"的视角。

---

## 评审立场

前面几轮评审都在"文档自洽性"层面（status 字段、门槛定义、角色完整性），这些已经很完善了。这轮换一个视角：**假设有人真的用 v3 跑 100 个任务，哪些设计会在规模化时暴露问题？** 找的是结构性盲点，不是文档瑕疵。

---

## 发现 1（🔴 结构性）：v3 在快速膨胀，正在变成它要解决的问题

**现象：** v3 从最初 1386 行涨到 2497 行（+80%），主流程文档 1751 行。一个使用者要真正用起来，至少得读 README + dispatch-protocol + state-machine = 838 行。

**问题：** v3 的立身之本是"解决复杂度"——让 Agent 不被上下文淹没。但 v3 自身正在变成一个需要大量上下文才能理解的复杂系统。这是自相矛盾的滑坡：

- 每次评审发现问题 → 加文档解决 → 文档变长 → 理解成本变高
- L1/L2/L3 三层质疑、7 个阶段、双层角色、三档自动化、上溯决策树……概念负载已经很重
- 讽刺的是：如果一个 subagent 要"读懂 v3 再执行"，它读 800 行文档本身就消耗大量上下文——正是 v3 要避免的

**根因：** v3 一直在"加法"——每个问题都用"加一段文档/加一个机制"解决，没有人做"减法"。缺少一个对抗复杂度膨胀的力量。

**建议：**
- 区分"使用者必读"和"参考查阅"。使用者应该只需读一个 < 150 行的快速上手，其余是出问题时才查的参考
- 设一条硬约束：**核心必读路径（让人能开始用的最小文档）不超过 200 行**。超了就是信号——机制太复杂，该简化而非加文档
- 警惕"每个边界情况都要有文档覆盖"的冲动。有些边界情况的正确答案是"PAUSED 交给人"，不需要为它设计一套机制

---

## 发现 2（🔴 结构性）：全自动 /loop 与"需要用户介入"的 L3 质疑根本冲突

**现象：** 
- loop-orchestration 档位 C 说"全自动跑完 P1-P7，全程不停"
- 但评审迭代机制的 L3（需求质疑）和用户介入边界列了 6 类必须人介入的情况
- `--auto-approve` 可以跳过人工确认

**问题：** 这两个设计在哲学上矛盾。L3 质疑的本质是"需求可能不合理，需要人决策"——这是**不可自动化**的（Agent 无权决定"这个功能要不要做"）。但全自动 /loop 的承诺是"不停"。

当 `--auto-approve=true` 且任务触发了 L3 质疑（比如发现 AC 自相矛盾），会发生什么？
- 要么 /loop 停下来（违背"全自动"承诺）
- 要么 Agent 自己替用户做了业务决策（越权，危险）

文档没有明确这个冲突点的解法。

**建议：**
- 明确：**L3 需求质疑是 /loop 的硬性中断点，`--auto-approve` 不能跳过它**。auto-approve 只能跳过"低风险的阶段门槛确认"（如 P2 设计评审通过），不能跳过"需求层质疑"和"P7 发布"
- 在 loop-orchestration 里画一条线：哪些确认可以 auto-approve（阶段内门槛），哪些绝对不可以（L3 需求 + P7 发布 + 安全决策）
- 全自动的真实含义应该是"自动跑完不需要决策的部分，遇到需要决策的硬中断点必停"，而不是"全程不停"

---

## 发现 3（🟠 重要）：多 Agent 并发跑 v3 的状态文件竞争未解决

**现象：** 用户的真实场景是 5 个 agent 同机跑。但 v3 的状态管理假设单写者：
- `.state.yaml`（重试计数、review_scores）
- `active-tasks.md`（任务看板）

git-integration 提了 push 冲突用 rebase 解决，但**没解决文件级的并发写**——两个 agent 同时跑各自的任务，同时改 active-tasks.md，rebase 能合并行级改动，但如果同时改 `.state.yaml` 的同一任务块，会丢更新。

**更深的问题：** 同一个任务会不会被两个 agent 同时跑？v3 没有"任务锁"机制。如果 agent A 和 agent B 都读到 T005 在 P2，都去派发 P2，就重复执行了。

**建议：**
- 每个任务的 `.state.yaml` 加一个 `owner` 字段（哪个 agent 在跑）+ `lease_until` 时间戳。开跑前检查：有 owner 且未过期 → 跳过（别人在跑）
- active-tasks.md 的并发写改为"每任务一个独立状态文件"（`docs/tasks/Txxx/.state.yaml`），看板由这些状态文件聚合生成，而不是多 agent 直接写同一个看板文件
- 明确文档化："同一任务同一时刻只能一个 agent 跑"是 v3 的前提

---

## 发现 4（🟠 重要）：v3 用自己开发自己，但从未用自己验证过自己

**现象：** T003 在"修复 v3 的 16 项缺陷"，但 T003 自己**没有走 v3 的 P1-P7 流程**（没有 docs/tasks/T003/ 的阶段产出目录）。v3 的所有迭代都是直接改文档，不是用 v3 流程改的。

**问题：** 这是一个"卖减肥药的自己没减肥"的信任问题。v3 声称能编排复杂任务，但它自己的开发（一个相当复杂的任务）从没用过自己的流程。validation-report 验证的是"派发机制能跑"（一个 echo 测试），不是"v3 能完成真实复杂任务"。

**这不是说 v3 不好，而是说它的核心主张还没有被自身实践检验过。**

**建议：**
- 找一个真实的中等任务（比如 backlog 里的 embed 路由或 progress notification），**完整走一遍 v3 的 P1-P7**，作为第一个真实 dogfooding 案例
- 把这次 dogfooding 的完整阶段产出留在 docs/tasks/ 作为"v3 真的能跑完整任务"的证据和范例
- 在那之前，v3 的"可用"结论应该限定为"派发机制可用"，而非"完整流程可用"

## 发现 5（🟡 观察）：文档里还有未更新的过时表述

**现象：** validation-report 已证实方法 A（自定义 subagent）因 issue #29616 失败、方法 B 是唯一可用路径。但：
- README 平台适配仍写"task 工具 + markdown 文件定义自定义角色"，没标注"经验证方法 A 在 OpenCode 不可用，用方法 B"
- 目录结构里 custom-role.md 模板仍在，但没指明它走的是已验证失败的路径

**建议：** README 平台适配章节加一句"OpenCode 经 validation-report 验证：自定义 subagent（方法 A）不可用，统一用 general + prompt 注入（方法 B）"。让文档和验证结论一致。

---

## 发现 6（🟡 观察）：缺少"流程跑偏"的自检机制

**现象：** v3 假设主 Agent 会忠实执行派发协议。但 validation 也发现过"主 Agent 可能自己干而非派发"。v3 没有一个机制让主 Agent **自检"我是不是又开始自己干了"**。

**建议：** 加一个轻量自检——主 Agent 每完成一个阶段，检查"这个阶段的产出文件是不是 subagent 产出的（而非我自己写的）"。如果是自己写的，说明跑偏了，记录并纠正。这是对最初病灶的运行时防御，不只是文档约定。

---

## 核心判断

前几轮评审让 v3 在文档层面非常完善，但都在"把文档写对"这个维度。这轮的发现集中在一个更根本的问题上：

**v3 还停留在"设计得很好的文档"，没有跨越到"被真实复杂任务验证过的系统"。** 发现 1（复杂度膨胀）和发现 4（没 dogfood）是一体两面——因为没有真实跑过，所以只能靠不断加文档来"预防想象中的问题"，导致越来越重；而越重就越难真正跑起来。

**最有价值的下一步不是继续加机制，而是：用 v3 完整跑一个真实任务，让实践告诉你哪些机制是必要的、哪些是过度设计。** 跑完之后，大概率能删掉一些现在觉得"必须有"的东西。

减法比加法更难，但对现在的 v3 更重要。
