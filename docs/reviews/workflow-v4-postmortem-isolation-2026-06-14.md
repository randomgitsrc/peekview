# Workflow-v4 改进建议 Review

> 基于 2026-06-13 ~ 2026-06-14 PeekView 项目开发周期中的生产数据隔离问题复盘
> 目标：将问题根因抽象为可复用的 workflow-v4 最佳实践

## 问题回顾

| # | 问题描述 | 根因 |
|---|----------|------|
| 1 | pytest 写入生产库 1000 条测试数据 | conftest.py 只隔离 config file，未隔离 storage 路径 |
| 2 | E2E 测试 4 个"flaky"失败 | debug 服务器读取生产 config.yaml 的 captcha 配置 |
| 3 | 改代码后直接跑测试，未验证隔离 | 缺少"改代码→清库→跑测→验库"流程意识 |
| 4 | 新 agent 可能重蹈覆辙 | 依赖文档规则，非代码层面强制保障 |

## 改进建议

### R1: P5 验证增加环境隔离检查

**问题**：P5 阶段验证功能正确性，但未检查测试是否污染生产数据。

**建议**：
- P5 验证步骤增加"生产数据 snapshot 对比"
- 测试开始前记录生产库状态（��记录数、关键表的 checksum）
- 测试结束后对比，差值 > 0 则 P5 失败

**适用场景**：所有涉及数据库/文件系统读写的项目。

---

### R2: P1 问题定义必须由主 Agent 撰写

**问题**：subagent 撰写的 P1 可能遗漏关键上下文约束。

**建议**：
- `P1-problems.md` 必须由派发任务的主 Agent 亲自撰写
- 主 Agent 在 P1 中需明确标注**本项目的关键约束**（如本项目的"生产数据安全不可犯"）
- subagent 只能在主 Agent 定义的约束范围内工作

**适用场景**：所有多 Agent 协作任务。

---

### R3: P6 验收阶段默认不裁剪

**问题**：之前任务裁剪到 P5 被质疑太激进。

**建议**：
- P6 是质量保障的最后防线，**默认保留**
- 微/小任务可裁剪，但需在 P1 产出中说明理由
- 裁剪理由由主 Agent 审核

**适用场景**：所有任务。

---

### R4: 调试环境启动前检查生产环境状态

**问题**：开发过程中可能无意识地污染生产库，调试半天后发现问题来源是生产数据被修改。

**建议**：
- `make debug` 或等价命令在启动前，检查生产数据是否异常
- 异常指标示例：
  - 最近 N 分钟内的记录数变化
  - 存在测试特征的数据（如 slug 包含 `test-`、`perf-`、`e2e-`）
- 检查不通过应阻止调试环境启动，并输出醒目警告

**适用场景**：有本地开发模式且与生产环境共用/相似数据存储的项目。

---

### R5: 环境隔离机制的通用模式

**问题**：测试/调试环境与生产环境的数据隔离依赖开发者的手动配置，容易遗漏。

**建议**：项目应实现以下四层隔离机制，按强制力排序：

| 层级 | 机制 | 触发条件 | 效果 |
|------|------|----------|------|
| L1 | **代码层面强制隔离** | pytest 运行 | 自动设置 storage 路径到 tmp，实现"测试零配置隔离" |
| L2 | **Debug 模式自动隔离** | 设 `DEBUG=1` 等环境变量 | 所有裸调用自动重定向到临时目录 |
| L3 | **生产路径警告** | 裸调用指向生产路径且非服务进程 | 打印可感知警告（如 `⚠` 字符） |
| L4 | **文档规则** | — | 告知开发者正确操作方式（最弱） |

**实现要点**：
- **显式激活**：项目应有明确的 debug/dev 模式开关
- **自动隔离**：debug 模式下存储路径自动重定向
- **显式覆盖优先**：环境变量或参数传入的路径 > 默认隔离路径
- **生产警告**：裸调用指向生产路径时打印���目警告

**适用场景**：所有有"开发/测试模式 vs 生产模式"区分的项目。
