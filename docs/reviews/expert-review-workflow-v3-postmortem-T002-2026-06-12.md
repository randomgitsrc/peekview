## 评审结论

这是本项目中质量最高的复盘文档。它不是简单的「发生了什么」的记录，而是对 v3 工作机制的**系统性批判**——精确识别了 v3 的 4 个核心设计缺陷（gate 信任 subagent 产出、P7 定义错误、串行单评审、active-tasks.md 并发冲突）。16 项改进中仅 1 项存在可行性疑虑（B4 subagent 存活监控），其余均具操作性。优先级排序合理，不变项识别准确。

**唯一阻塞项**：B4 的「存活监控」方案（主 Agent 检查 subagent 产出文件是否有进展）在当前 OpenCode 的 task 工具模型下无法实现——主 Agent 在 subagent 运行期间被阻塞，无法并发检查文件。应标注为平台依赖或降优先级至 P2。

## 二、逐类评审

### A. Gate 与验证（影响正确性）— 3 项

#### A1（🔴高）：Gate 判定必须跑命令，不只读文件

**准确性**：✅ 完全正确。

T002 提供了具体证据：P7-release.md 写「回归测试 429 passed」，但同期的 P5 unit.md 也写 429，而实际 pytest 收集到 473（后增至 487）。subagent 写了错误数字，主 Agent 照单全收。

门控改造方向正确——`pytest -q` 的 exit code 是机器可验证的客观事实，`unit.md` 里的 `failed: 0` 是 subagent 的主观声明。

**建议补充**：命令输出引入量极小（pytest 最后一行 `N passed, 0 failed`），不影响上下文隔离原则。可明确：主 Agent 只捕获最后一行，不捕获完整测试输出。

#### A2（🔴高）：主 Agent 校验增加独立验证步骤

**准确性**：✅ 正确。现有 4 条校验全是存在性检查（文件在/Header 全/内容非空），缺少「声明 vs 现实」的对照。

**设计一致性**：与 A1 互补——A1 定义了新的 gate 判定标准，A2 将其编入 dispatch-protocol 的校验流程。两处修改需同步落到 `dispatch-protocol.md`。

#### A3（🔴高）：P7 重定义为「发布准备」

**准确性**：✅ 正确。P7 的原始定义 `P7-release.md 存在 → DONE` 有两层设计错误：
1. 不区分「准备」和「发布」——不是每个任务都需要发版 PyPI
2. 发布是不可逆操作，不应自动化

**对齐现有流程**：项目已有 `make pre-publish` 和 `make publish` 两个独立命令，P7 gate = `make pre-publish` exit 0 完美映射。

### B. Subagent 安全（影响可靠性）— 2 项

#### B4（🔴高）：Subagent 存活监控

**准确性**：⚠️ 意图正确，但方案在 OpenCode 下不可行。

问题诊断正确——subagent 可能无限等待。但「主 Agent 检查产出文件是否有进展」在 OpenCode 的 `task` 工具模型下无法实现：主 Agent 调用 `task` 工具后**被阻塞**直到 subagent 返回，无法并发执行文件检查。

**修正建议**：
1. 降优先级至 🟡 或 P2（平台依赖）
2. 替代方案 A：在派发 prompt 中要求 subagent 每隔若干操作输出一条进展标记到 stdout（而非文件），让平台日志可追溯
3. 替代方案 B：仅保留硬超时（`task` 工具已有 timeout 参数），移除「存活检查」部分

**验证**：先确认 OpenCode `task` 工具的超时行为——超时后是 kill 还是返回部分结果？这会影响硬超时策略。

#### B5（🟡中）：Subagent 升级机制 + P1 把关

**准确性**：✅ 正确。`[UPGRADE]` 标记方案简洁且不依赖平台特性。P1 后增加 `office-hours` 可选评审是对症下药——T002 的 P1 问题定义质量（AC6 为已否决方案而写）直接影响了后续所有阶段。

### C. 评审与质量（影响质量）— 3 项

#### C6（🟡中）：专家组并行评审 + 组长汇总

**准确性**：✅ 正确，且与现有 `EXPERTS.md` 评审会话设计形成呼应。

这是 v3 评审体系的自然演进——当前串行单评审无法覆盖 P2 多维审查需求（工程+产品+安全）。并行 + 组长汇总的设计已被 `EXPERTS.md` 中的「评审专家组组长」角色验证为可行模式。

**组长规则合理**：任何专家标 BLOCKER → rejected；全票无 BLOCKER → approved；分歧标出交人工。边界清晰。

**注意**：并行需要平台支持同时发起多个 `task` 调用。OpenCode 支持同一消息中多个 tool call，可行。

#### C7（🟡中）：P1 问题定义增加可选评审

**准确性**：✅ 正确。触发条件（P0/P1 优先级 / 架构变更 / 跨越 3+ 模块）务实，不强制所有任务都加 P1 评审。

#### C8（🟡中）：P6 增加双向一致性检查

**准确性**：✅ 正确。`[DEVIATION]` 标记精准覆盖了「设计文档有但已不适用」的检测方向。T002 的 AC6 正是这种情况——方案变更后 AC6 成为僵尸需求。

### D. 上下文与 Handoff — 3 项

#### D9（🟡中）：派发 Prompt 增加项目上下文固定段

**准确性**：✅ 正确。subagent 每次从零开始，项目级知识（包名 peekview、DI 模式、禁止 uvicorn 直接启动）是重复踩坑的高发区。

**注入方式务实**：在 prompt 模板中硬编码固定段而非让主 Agent 每次组装，减少遗漏。

**建议**：固定段的具体内容应与 CLAUDE.md / OPENCODE.md 保持同步，建议在模板末尾标注「由 CLAUDE.md 摘要生成，定期检查一致性」。

#### D10（🟢低）：P2→P4 方案答疑闭环

**准确性**：✅ 正确。`[CLARIFY: xxx]` 标记是低侵入的方案，不影响正常流程。

#### D11（🟢低）：文件路径显式声明

**准确性**：✅ 第 1 句错误，第 2 句正确。

文档说「v3 模板约定测试代码在 `docs/tasks/Txxx/P3-test-code/`」。但 v3 的 `task-files.md` 确实写 `P3-test-code/` 目录，而 `execution-roles/test-designer.md` 未约束输出路径。实际 T002 的测试在 `backend/tests/test_migration.py`（项目代码目录），与模板路径不一致。

修复方向（「不在模板中推断，派发 prompt 引用显式声明」）是正确的，但应补充：模板中删掉 `P3-test-code/` 和 `P4-implementation/` 的固定路径，改为占位符 `{实际代码路径，由 P3-test-cases.md / implementation.md 声明}`。

### E. 状态管理 — 1 项

#### E12（🟡中）：每任务独立 `.state.yaml`

**准确性**：✅ 正确。`active-tasks.md` 多 Agent 并发修改的冲突问题是 v3 的真实痛点。

`git-integration.md` 的缓解策略（「只改自己那行」）在 git rebase 场景下不可靠——即使按行修改，rebase 仍可能因上下文行变化产生冲突。

`.state.yaml` 按任务隔离的方案从根本上消除冲突源，设计清晰。

**建议补充**：`.state.yaml` 的 schema 应加入 v3 规范，且 `active-tasks.md` 重建脚本应放在 `scripts/` 下可执行。
