## 二、新发现的问题

### 🟠 NEW-1：P3 红灯门槛无法区分「测试正确地失败」vs「测试本身写错了」

state-machine P3→P4 门槛：`测试代码存在 AND 运行后失败(红灯)`。

TDD 要求测试先失败，但"失败"有两种本质不同的原因：
1. **正确的红灯**：测试逻辑对，因为实现还没写所以失败（期望的）
2. **错误的红灯**：测试本身有语法错误 / import 错误 / 断言写反，根本跑不起来

当前门槛只看 `failed > 0`，两种都满足，会把"测试写错了"误判为"可以进 P4"。结果 P4 实现完成后，P5 才发现测试根本是坏的，回退成本高。

**修正建议：** P3 门槛细化为"测试**因断言不满足**而失败，而非因 collection error / import error 失败"。test-designer 角色产出时需附"失败原因分类"（assertion failure ✓ / error ✗），主 Agent 判定时要求是 assertion failure。

---

### 🟠 NEW-2：retry 计数归零时机可能绕过上限

state-machine：「每次进入新阶段：重试计数归 0」。

考虑这个序列：
```
P2 retry 用到 2/3 → 评审 approved 进 P3 → retry 归 0
P3 因某种原因回退到 P2（比如发现 P2 设计有问题）→ retry 又从 0 开始
```

如果存在阶段回退（P5 failed 回 P4 是设计内的，但 P3/P6 发现上游问题需要回退到 P2 的情况文档没禁止），retry 归零会让同一个阶段被反复重试远超 MAX_RETRY，绕过护栏 2 的全局步数上限之前可能已烧掉大量 token。

**修正建议：** 明确两件事：（a）是否允许非相邻阶段回退（P5→P4 是相邻允许，P6→P2 是否允许？）；（b）若允许回退，retry 归零应改为"按阶段独立计数"，而非"进入新阶段就归零"，否则全局步数上限（护栏 2，默认 20）是唯一兜底，偏松。

---

### 🟡 NEW-3：主 Agent「自己 grep 摘要」与「不读文件全文」存在张力

dispatch-protocol 铁律 3 + 返回校验第 2 条：subagent 若违规返回全文，主 Agent「不读全文，只提取路径，自己 grep 摘要」。

但"自己 grep 摘要"意味着主 Agent 要读产出文件来生成摘要——这与"主 Agent 上下文只增加路径+摘要、永不碰文件全文"的核心原则冲突。grep 一个大文件提取摘要，grep 的输出仍可能很长，污染主 Agent 上下文。

**修正建议：** 明确"自己 grep 摘要"的边界——只 grep Header 区域（前 N 行）或特定字段（status/failed），不全文 grep。或者：subagent 违规返回全文时，直接判失败重试（要求它重新只返回摘要），而不是主 Agent 替它补救。后者更符合"主 Agent 不碰全文"的洁癖，且惩罚违规 subagent。

---

### 🟡 NEW-4：P6 由 architect 兼任，存在「自己审自己」的独立性问题

README 阶段表：P2 设计和 P6 一致性检查都是 architect 角色。P6 要检查"实现是否符合 P2 设计"，但 P2 设计正是 architect 自己产出的。

虽然 P6 的 architect 是新派发的独立 subagent（独立上下文，不记得 P2 怎么写的），物理上隔离了。但角色认知模式相同，容易带入"我当初就是这么想的"的确认偏误，对自己设计的方案宽容。

**修正建议：** 要么 P6 换一个独立角色（如 reviewer 而非 architect），要么在 architect.md 的 P6 部分明确"以批判第三方视角检查，假设设计可能有错"，削弱自我确认倾向。影响较小，因为有独立 subagent 隔离兜底。

## 三、设计优点（独立确认）

1. **三铁律 + 返回校验** 形成完整闭环：派发约束 + 失败兜底
2. **状态落盘 + 每轮重读** 是 LLM 长流程的正确范式，抗压缩抗中断
3. **status 映射表** 优雅解决多评审角色门槛判定的一致性
4. **四道护栏**（重试上限/步数上限/检查点/一致性检查）防失控完整
5. **方法 B（prompt 注入角色）** 规避 OpenCode issue #29616，跨平台最稳
6. **落地建议「档位 A→B→C 渐进」** 务实，不冒进上全自动
