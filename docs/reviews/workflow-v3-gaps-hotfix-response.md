# Workflow-v3 缺口分析：热修复与响应型任务

## 背景

T002（数据库迁移修复）发版 v0.1.53 后，生产环境 `peekview user list` 仍崩溃。排查过程耗时十几个回合，最终定位根因为 `cli.py` 中 `from sqlalchemy import select` 返回 `Row` 对象而非 `User` ORM 实体，Cython `BaseRow` 无法通过 `__getattr__` 映射 `is_admin` 字段。修复只需一行。

但在排查过程中，主 Agent 找到根因后**直接编辑代码、测试、bump、发布**，完全跳过了 v3 的任务创建和 subagent dispatch 流程。这不是故意不遵守规范，而是 **v3 当前设计不支持这种场景**。

---

## 缺口 1：「热修复快速通道」缺失

### 现象

修复只需一行代码（`from sqlalchemy import select` → `from sqlmodel import select`），单文件改动。走完整 P1-P7 是过度工程——创建一个任务、派发 analyst 子 Agent 理解问题、派发 implementer 重新理解同样的代码，消耗远大于收益。

### 根因

v3 的「适用边界」只按**任务规模**分类（微/小/中/大），但没有按**任务性质**区分：

| 类型 | 特征 | 合适流程 |
|------|------|----------|
| 微任务 | typo、文案、单行配置 | 直接做，不走 v3 |
| 小任务 | 明确的 bug 修复 | P1+P4+P5 |
| 中任务 | 新功能 | 完整 P1-P7 |
| **热修复** | 生产崩溃，根因明确，修复 ≤5 行 | **缺失** |

当前 v3 把「生产崩溃找到根因后的一行修复」强行塞进「小任务」分类，但小任务也要求 P1+P4+P5 + subagent dispatch。debug 上下文中已经定位到精确修复点，不应该再创建任务让 subagent 重新理解一遍。

### 建议修复

v3 README「适用边界」表增加热修复路径：

```
| 热修复 | 生产 bug，根因已明确，修复 ≤5 行单文件 | 直接改 + commit [hotfix] + 事后补 P5 验证记录 |
```

热修复的约束：
- 修复行数 ≤5 行，单文件
- 根因已在 debug 中确认（非猜测）
- 主 Agent 自己跑测试验证（pytest -q exit 0）
- 事后补一个简化的 P5 验证文件（`HOTFIX-{date}.md`）
- 不要求 subagent dispatch，不要求 P1/P2/P3 阶段文件

---

## 缺口 2：v3 只支持「计划型任务」，不支持「响应型任务」

### 现象

v3 的 P1-P7 流程完全是从需求出发自上而下的**计划模式**：

```
需求 → P1(问题定义) → P2(方案设计) → P3(测试) → P4(实现) → P5(验证) → P6(一致性) → P7(发布)
```

但这次问题暴露链条是：

```
生产崩溃 → 分析代码 → 定位根因 → 修复 → 验证 → 发版
```

这是自下而上的**响应型工作流**——从异常出发，追溯到根因，修复，验证。v3 没有对应这个方向的阶段流。

### 根因

v3 的设计前提是「所有工作都由任务驱动」。但生产 crash 不是任务——它是突发事件。当前的处理方式是把突发事件强行转成任务（T002 → T003），但转换过程中丢失了排查现场的大量上下文。

### 建议修复

增加 `R1-R4` 响应型阶段，与计划型 P1-P7 并列：

| 计划型 (P) | 响应型 (R) | 触发方向 |
|------------|-----------|----------|
| P1 问题定义 → | R1 问题捕获 | 用户报告 / 监控告警 |
| P2 方案设计 → | R2 根因分析 | debug / 日志分析 |
| P3 测试设计 → | — | （响应型跳过，修复后补测试） |
| P4 代码实现 → | R3 修复实现 | 基于根因的直接修复 |
| P5 逐项验证 → | R4 验证发版 | 测试 + pre-publish |
| P6 一致性检查 → | — | （小修跳过） |
| P7 发布准备 → | — | （合并入 R4） |

R 流的设计原则：
- R 流是方向，不强制创建任务。主 Agent 可直接执行 R1-R4 步骤
- R2 阶段的产出是「根因分析」而非「问题定义」（一个是 what went wrong，一个是 what should be done）
- R3 修复 ≤ 20 行直接做，> 20 行转 P 流创建任务
- R4 后强制写一个简要的响应记录文件（`RESPONSE-{date}.md`）

## 缺口 3：「排查现场」→「固化任务」的认知切换成本

### 现象

排查耗时十几个回合，每个回合都在积累对问题的理解。当根因最终浮出水面时，主 Agent 对修复方案已经了然于胸。此时要求切换回「创建任务 → 写任务描述 → 派发 subagent → subagent 重新理解代码 → 实现修复」，这是认知资源的极大浪费。

### 根因

v3 的「主 Agent 只编排不执行」原则在计划型任务中是**正确**的——主 Agent 不应该在思路不清的时候直接上手写代码。但在响应型任务中，主 Agent 已经做了大量分析工作（读代码、跑诊断、写 debug 脚本），已经具备了子 Agent 难以复现的现场上下文。强制切回子 Agent dispatch 等于**丢弃排查现场的所有上下文**。

### 建议修复

允许主 Agent 在以下条件下直接提交修复，不回退到任务创建：

1. 修复来源于正在进行的排查过程
2. 修复行数 ≤ 20 行
3. 主 Agent 自己跑测试验证通过
4. 提交时标注 `[direct-fix]` 前缀
5. 事后补「自动生成的任务」记录到 `docs/tasks/`，标注 `auto-generated: true`，供后续回顾但不阻塞修复

这类似于「医生在急诊中找到出血点直接缝合」，而不是「缝合前要先写医案、开病历、挂号流程」。

---

## 总结

| # | 缺口 | 影响 |
|---|------|------|
| 1 | 缺热修复快速通道 | ≤5 行生产修复被强制走 P1-P7 |
| 2 | 只支持计划型，不支持响应型 | 生产崩溃 → 排查 → 修复 没有对应的阶段流 |
| 3 | 排查现场到固化任务的切换成本高 | 查完根因后回头创建任务让 subagent 重走一遍 |

这些缺口导致了主 Agent 在 debug 后发现修复后「不想用 v3」的局面——不是 v3 不好，而是 **v3 当前只覆盖了计划型工作流，没有给响应型工作流留入口**。

建议在 T003（或新建 T004）中纳入 R1-R4 响应型阶段 + 热修复快速通道 + direct-fix 标记机制。
