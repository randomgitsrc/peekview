## Step 0 — 前提、杠杆、轨迹

### 挑战前提

专家评审报告对 v4 的诊断基本准确——12 个问题点真实存在。但**问题存在 ≠ 需要立即修**。

这个修复方案的前提假设是"专家评审列出的每一条都应该被修复"，这个假设本身值得质疑。Workflow v4 是一个**内部开发流程文档**，不是面向用户的产品。流程文档的质量不取决于它覆盖了多少边缘情况，而取决于它是否让团队**正确地完成任务**。

validation-report.md 已经判定 v4 可用，实际任务（T001、T002、T005）已经跑通了 v4。这说明当前版本在实战中**能用**。修复方案试图在"能用"的基础上做到"完美"——这是工程冲动的自然产物，但不是 CEO 应该批准的投入方向。

### 找杠杆

修复范围里存在一个明显的不对称：

- **Fix-1**（创建 check-tdd-red.sh）：唯一真正的实操阻塞项。P3 gate 实际上跑不通。修这一个，P3 就能转起来。
- **Fixes 8-12**（第 3 批全量）：边缘情况的机制增强。SCOPE+ 矩阵、BDD 模糊词 grep、P7 覆盖度计数、链式上溯拦截——每一条都在为**尚未发生过的失败模式**增加规则。这些规则被写入后，将成为每一个读 workflow 文档的人必须理解和遵守的认知负担。

杠杆点很清楚：**Fix-1 的 ROI 是 12 个 Fix 中最高的**，因为它是唯一"不修就走不通"的项。其余 11 项修与不修，v4 都能继续跑任务。

### 轨迹（现在 → 12 个月）

```
现在（v4 可用，7 分）  ──使用中自然暴露真痛点──→  6 个月（v4 实战打磨，8 分）  ──持续迭代──→  12 个月（v4 成熟稳定，9 分）
                          │
                    在这个阶段过度设计 ← 修复方案的 Batch 3
                    会拖慢"使用→发现→修复"的反馈循环
```

最好的流程改进方式不是坐在房间里审查文档找缺口，而是**跑 10 个真实任务，记录哪些地方实际卡住过人**，然后只修那些。

## 方向结论：收窄

整体方向正确——修阻塞项、补文档矛盾。但范围过大，第 3 批（5 个 P2 机制增强）是**过度设计**，应在实战积累数据后再决定是否做、做哪些。

下面逐项分析。

## 分项评审

### 第 1 批（P0）：全做，方向对

| Fix | 判定 | 理由 |
|-----|------|------|
| Fix-1 创建 check-tdd-red.sh | ✅ 做 | 唯一实操阻塞项。不修 P3 gate 跑不通 |
| Fix-2 统一 D8 小结触发时机 | ✅ 做 | 两文档互相矛盾，影响主 Agent 行为 |
| Fix-3 修正 P7 grep 命令 | ✅ 做 | 当前 `grep -L` 表达晦涩，`! grep -q` 直观可判定 |

**结论：第 1 批无争议，直接执行。**

---

### 第 2 批（P1）：做 3 项，精简 1 项

| Fix | 判定 | 理由 |
|-----|------|------|
| Fix-4 P3 跳过后 P4 门槛 | ✅ 做 | 真缺口。小任务 skip P3 是高频场景，不定义确实走不通 |
| Fix-5 8 处 v3→v4 标注 | ⚠️ 降级 | 纯 cosmetic。这些文件在 `workflow-v4/` 目录下，v3 标注不会造成实际混淆。**建议：改为顺手改，不作为独立 Fix 追踪。** 投入产出不成比例——8 处修改 + 逐行 grep 验证 + commit message，收益为零行为改变 |
| Fix-6 PAUSED 恢复协议 | ✅ 做 | 合理。PAUSED 是硬中断点，没有恢复路径意味着人卡住了也不知道怎么解。这个协议填补了真实操作缺口 |
| Fix-7 P8 gate 补注 | ✅ 做 | 合理。两文件脱节，补一行引用即可 |

**结论：Fix-4/6/7 保留，Fix-5 降级为顺手扫尾，不单独追踪。**

---

### 第 3 批（P2）：全量搁置，等实战数据

| Fix | 判定 | 理由 |
|-----|------|------|
| Fix-8 SCOPE+ 矩阵 | ❌ 搁置 | 当前"主 Agent 判断"的描述在实际任务中管用。矩阵化意味着将 4 种抽象分类写死进流程——分类本身是否准确、是否覆盖全、是否会误导判断，都未经验证。**在 SCOPE+ 实际触发过 5+ 次后再总结规律，而不是提前设计规律。** |
| Fix-9 BDD 模糊词 grep | ❌ 搁置 | 动机对（BDD 质量是脊梁），手段错。grep "体验良好""正常工作""更快"无法真正阻止低质量 BDD——分析师可以写"status=200"但没有理解自己为什么要写。BDD 质量是 P1 分析师的能力问题，不是门槛检查能解决的。**建议：在 P1 analyst.md 角色定义里强化 BDD 写法指导，不加机械门槛。** |
| Fix-10 P7 覆盖度门槛 | ❌ 搁置 | 假设 subagent 会恶意写空壳 P7 来绕过 gate——这在协作式角色体系里是极低概率事件。数 P2 设计要点数来做覆盖度检查，增加了 gate 复杂度（"设计要点"怎么定义？一条算一个还是分解？），但保护的是一个几乎不可能发生的失败模式。**在 P7 空壳实际出现一次之后再考虑。** |
| Fix-11 清理 D7 改进项 | ⚠️ 随手做 | 删一行、加一行，零成本。**建议：合入第 1 批或第 2 批 commit，不作为独立 Fix。** |
| Fix-12 链式上溯拦截 | ❌ 搁置 | 纯理论风险。P5→P4→P2 链式回退需要：(a) P5 耗尽 2 次 retry → (b) P4 也耗尽 3 次 retry → (c) 整个过程无人注意。全局步数上限（20 步）已经是这个场景的兜底。再加一层拦截规则，解决的是一个概率接近零的问题，代价是状态机规则又多了一条。**等真有人踩中再修。** |

**结论：第 3 批全部搁置。Fix-11 顺手做，不独立追踪。**

## 缺失什么

修复方案遗漏了三个实操层面的考虑：

### 1. 无实际痛点锚定

12 个 Fix 全部来自文档审查（专家评审），没有一个来自"T00x 任务在 Pn 阶段实际卡住了"。修复方案没有引用任何真实任务的问题回溯。这意味着大量 Fix 修的是**理论上可能发生的问题**，而非**已经发生过的问题**。

### 2. 无回归风险说明

Fix-3 改 P7 gate、Fix-4 改 P4 gate（P3 skip 分支）、Fix-7 补 P8 gate——这些改的是状态机转移规则。状态机的 gate 命令是主 Agent 实际执行的逻辑。修改后现有进行中任务的状态判定是否会受影响？修复方案未提及。

### 3. 无"什么时候算修完"的标准

修复方案列出 12 项修复，但没有说"修到什么样就算这个方案完成了"。建议：**以"跑通一个完整 P1-P8 任务且 P3 gate 通过"作为验收标准。** 这比"12 项全部改完"更有意义。

## 执行顺序

三批独立 commit 的方案是合理的，但调整后应该是：

```
第 1 批（阻塞修复）：
  Fix-1 + Fix-2 + Fix-3 + Fix-11（顺手）
  → commit "fix(wf-v4): create check-tdd-red.sh, fix P7 grep, unify summary trigger point"

第 2 批（逻辑缺口）：
  Fix-4 + Fix-6 + Fix-7 + Fix-5（顺手）
  → commit "fix(wf-v4): P3-skip P4 gate, PAUSED recovery, P8 gate cross-ref"

搁置：
  Fix-8, Fix-9, Fix-10, Fix-12 → 不 commit，在 improvement-backlog.md 记录观察项
```

原方案 12 Fix / 3 commit → 调整后 7 Fix / 2 commit（其中 2 项降级为顺手改）。
