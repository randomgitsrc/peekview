## 二、三层执行错误（修订版新增）

> 独立评审指出：原版复盘把"主 Agent 三次违反现成协议"包装成"agate 机制有缺口"。这是最严重的归因偏差。本节补上原版缺失的执行错误分析。

T016 的 P3 阶段发生了三层执行错误，每一层都违反了 agate 现成协议：

### 错误 1：subagent 失败未计入 retry

`.state.yaml` 证据：

```yaml
# P3 commit (d319c04b) 后
retry_count:
  P1: 0
  P2: 1
  P3: 0   # ← 仍然是 0，但实际空返回 3 次
```

按 dispatch-protocol.md"subagent 返回校验"：约定的产出文件不存在 = 派发失败，**计入重试**。3 次空返回应使 retry_count.P3 = 3，但实际为 0。

**主 Agent 没有按协议记录失败**。

### 错误 2：未触发 PAUSED 安全网

state-machine.md 规定 P3 MAX_RETRY = 2。若 retry_count.P3 正确记录为 3，则 3 > 2 应触发 L2 上溯 → PAUSED 报告人工。

但主 Agent 既没记 retry，也没触发 PAUSED，直接降级亲自写。**PAUSED 这个现成的安全网，主 Agent 没有用**。

### 错误 3：违规降级亲自写代码

"主 Agent 不亲自写代码"是 agate 核心原则。主 Agent 在 subagent 失败后自行决定降级，违反此原则。

### 归因修正

原版 §9 结论说"agate 没有告诉主 Agent 'subagent 失败后该怎么办'"——**这是错的**。agate 通过 dispatch-protocol（retry 计数）+ state-machine（PAUSED 安全网）已经告诉了主 Agent 该怎么办。主 Agent **没有按协议执行**。

这不是"agate 没说"，是"主 Agent 没做"。机制确有改进空间（见 §三），但 T016 的首要问题是**执行不严格**，不是机制不完善。

## 三、根因分析：为什么 subagent 会失败

### 3.1 表面现象 vs 深层根因

**表面现象**：subagent 连续 3 次返回空，似乎"能力不足"。

**深层根因（假设，无 subagent 内部日志证据）**：主 Agent 没有履行"编排者"职责，把活儿整个甩给 subagent。

### 3.2 P3 派发 prompt 的实际内容（失败案例）

派发 P3 时，主 Agent 的 prompt 要求 subagent：

1. 读 7 个输入文件：
   - P0-brief.md（92 行）
   - P1-requirements.md（273 行）
   - P2-design.md（664 行）
   - prototype.html（489 行）
   - useMermaid.ts（44 行）
   - useMarkdown.ts（305 行）
   - mime.spec.ts（50 行）
   - **合计约 1917 行输入**（原版说"1830 行"/"6 个文件"，小错误，已修正）

2. 产出 3 个异构文件：
   - P3-test-cases.md（文档）
   - usePlantUML.ts（stub 代码）
   - usePlantUML.spec.ts（测试代码，含 mock）

3. 理解 vitest 模式、mock 策略、TDD 红灯要求、stub 边界

**对比成功的 P1**：P1（需求分析→文档）和 P3（设计→测试代码）的复杂度本质不同。P3 本身就更难——不能简单归因为"输入格式"单一变量。P1 成功既因为输入组织更好，也因为任务本身更简单。

### 3.3 失败链路还原（推测，无日志证据）

> ⚠️ 本节为事后推测 subagent 内部过程，无 subagent reasoning trace 证据。作为假设呈现，不是结论。

```
主 Agent 收到 P3 任务
  → 按 dispatch-protocol "传路径不传内容"
  → 列出 7 个输入文件路径 + 产出要求
  → subagent 收到 prompt
  → subagent 开始逐一读 7 个文件（~1917 行）
  → [推测] subagent 上下文被输入文件占满
  → [推测] subagent 需同时理解 BDD + 接口 + 串行队列 + mock + vitest
  → [推测] 认知负荷超载
  → subagent 静默失败（空返回，无错误信息）
```

### 3.4 根因结论（修订版）

原版说"根因 90% 是主 Agent 上下文组织不当，10% 是 subagent 能力限制"——**这个 90/10 比例无任何数据来源，删除**。

修订结论：**主 Agent 的上下文组织不当是显著因素，但无法量化占比**。单案例无法得出比例化归因。需更多任务验证才能区分"管理原因"和"技术原因"的权重。

agate 的 dispatch-protocol 规定"prompt 只传文件路径，不传文件内容"——这个原则本身正确（防 prompt 膨胀 + 上下文隔离）。但它有一个**隐含假设**：subagent 能自己消化输入文件并产出结果。当输入量大（~1917 行）且异构（设计文档+源代码+测试模式）时，这个假设可能不成立。

但解法**不是**让主 Agent 读所有文件做提炼（那违反铁律 2），而是给 subagent 更聚焦的导航——见 §四建议 1 修订。

## 四、agate 机制的结构性缺口

> 本节讨论机制层面的改进空间。但需强调：T016 的首要问题是 §二的三层执行错误，不是这里的机制缺口。机制缺口是次要因素。

### 缺口 1：dispatch-protocol 缺少"输入导航"原则

**现状**：dispatch-protocol 规定"传路径不传内容"（铁律 2），但没规定"主 Agent 如何给 subagent 导航输入文件"。

**后果**：主 Agent 倾向于把所有相关文件路径都列给 subagent，让 subagent 自己读。简单任务能跑通，复杂任务 subagent 上下文过载。

**正确做法（修订版，不违反铁律 2）**：主 Agent **不读全文**，但基于自己写的 P0-brief + P1/P2 产出的提炼文档，给 subagent 提供"读哪几节、关注什么"的导航。例如"P2-design.md §3.2 有接口签名，§3.4.1 有串行化设计，重点关注这两节"——而非主 Agent 读完提炼成简报（那会污染主 Agent 上下文）。

### 缺口 2：阶段内任务粒度无规定

**现状**：agate 的阶段划分是固定的 P0-P8，但每个阶段内部的"任务粒度"没有规定。P3 默认是一个 subagent 产出所有测试文件。

**后果**：P3 要求一个 subagent 产出 3 个异构文件（文档+stub+测试代码），粒度过大。

**正确做法**：允许阶段内任务拆分。P3 可拆成"P3a 测试用例文档"+"P3b stub+测试代码"。每个 subagent 任务产出 1-2 个文件。

### 缺口 3：subagent 失败后无恢复策略（机制层面）

**现状**：state-machine.md 的重试机制（retry_count, MAX_RETRY, L2 上溯）针对"subagent 产出了但质量不够"。对"subagent 根本没产出（空返回）"无明确的策略指引。

> 注：原版说"agate 没有策略"是错的——retry→PAUSED 机制是现成的。这里说的缺口是"针对空返回这种特殊失败模式，缺乏指引主 Agent 如何调整 prompt 的策略"。

**后果**：主 Agent 在 subagent 空返回后，除了重试相同 prompt，不知道该怎么调整。

**正确做法**：增加"空返回恢复策略"指引——1 次空返回后必须改变策略（拆分任务/换 subagent 类型/补导航），不允许第二次相同重试。

### 缺口 4：降级规则边界模糊

**现状**：agate 说"若 has_task_tool: false，派发步骤降级为主 Agent 直接执行"。但没说"has_task_tool: true 但 subagent 执行失败时怎么办"。

**后果**：主 Agent 在 subagent 失败后自行决定降级，违反"主 Agent 不亲自写代码"原则。

**正确做法**：明确——**降级只在环境不支持时发生（has_task_tool: false / has_local_runtime: false），不在 subagent 执行失败时发生**。subagent 失败 = 需要调整 prompt 重派或触发 PAUSED，不是降级信号。

### 缺口 5：缺少"编排者职责"的明确定义

**现状**：agate 说"主 Agent 只做四件事：读状态、派发 subagent、验门槛、更新状态"。这四件事都是"动作"，没有"职责"。

**后果**：主 Agent 把"派发 subagent"理解为"把任务描述发给 subagent"，而不是"消化输入、组织上下文、派发"。

**正确做法**：明确主 Agent 的核心职责是**任务分解 + 输入导航 + 验证**。不是简单的传话筒。但"消化"不等于"读全文"——是基于自己已写的 P0-brief 和 P1/P2 产出来组织任务，不给 subagent 灌全文。

## 五、改进建议

### 建议 1：dispatch-protocol 增加"输入导航"原则（修订版）

> 原版要求"主 Agent 亲自读所有输入文件，提炼简报"——独立评审指出这违反铁律 2。修订为"导航"而非"提炼"。

```
## 派发前输入导航

主 Agent 派发 subagent 前，基于自己已写的 P0-brief + P1/P2 产出的提炼文档，
给 subagent 提供"读哪几节、关注什么"的导航：

1. 不读全文：主 Agent 不碰输入文件全文（铁律 2 不变）
2. 给导航：prompt 中注明"读 P2-design.md §3.2 接口签名 + §3.4.1 串行化设计，
   重点关注这两节"
3. 给范例：附 10 行以内的接口签名/模式范例代码（不是全文）
4. 给约束：明确产出文件的格式、函数签名、测试模式要求

导航原则：
- 导航是方向指引，不是内容提炼
- subagent 仍自己读文件，但知道读哪里、关注什么
- 主 Agent 上下文不被输入文件污染
```

### 建议 2：允许阶段内任务拆分

```
## 阶段内任务拆分

当阶段产出涉及多个异构文件（文档+代码）或多种技能（设计+实现）时，主 Agent 可拆分为多个 subagent 任务：

- P3 可拆：P3a 测试用例文档 + P3b stub 接口 + P3c 测试代码
- P4 可拆：按文件拆分，每个文件一个 subagent 任务

拆分原则：
- 每个任务产出 1-2 个文件
- 任务间有依赖时串行，无依赖时并行
- 拆分需在 .state.yaml 记录（如 P3a/P3b/P3c）
```

### 建议 3：subagent 失败恢复协议（修订版）

> 原版"同一 prompt 重试 ≤ 2 次"——独立评审指出上限无依据。修订为"1 次空返回后必须改变策略"。

```
## subagent 失败恢复

subagent 空返回或质量不够时，主 Agent 必须：

1. 先按 dispatch-protocol 记入 retry_count（这是现成协议，不是新规则）
2. 分析失败模式：
   - 空返回：prompt 过复杂或输入不足 → 拆分任务或补导航
   - 质量不够：评审意见回流重新派发
   - 超时：任务过大 → 拆分

3. 调整策略（空返回 1 次后必须改变策略，不允许第二次相同重试）：
   a. 拆分任务：粒度更小
   b. 补导航：给"读哪几节"的指引
   c. 换 subagent 类型：frontend ↔ general

4. 禁止行为：
   - ❌ 降级为主 Agent 直接执行（除非 has_task_tool: false）
   - ❌ 无分析地重复重试相同 prompt
   - ❌ 跳过阶段
   - ❌ 不记 retry_count 就降级（违反现成协议）

5. 重试上限仍按 state-machine.md 的 MAX_RETRY，触发后 PAUSED
```

### 建议 4：强化"编排者职责"定义（修订版）

> 原版"消化器"比喻——独立评审指出有让主 Agent 读全文的风险。修订。

```
## 主 Agent 职责（修订）

主 Agent 的核心职责不是"派发任务"，而是**任务分解 + 输入导航 + 验证**：

1. 任务分解：将复杂阶段拆成可独立执行的子任务
2. 输入导航：给 subagent "读哪里、关注什么"的方向指引（不读全文）
3. 验证：亲自跑 gate 命令，不信 subagent 自报

主 Agent 不是传话筒——把用户需求原样传给 subagent。
主 Agent 也不是消化器——不需要读全文做提炼。
主 Agent 是导航员——基于自己写的 P0-brief 和 P1/P2 产出，给 subagent 方向。

违反职责的典型行为：
- 把 7 个文件路径甩给 subagent 让它自己读，无导航
- subagent 失败后自己干（降级）
- 信 subagent 说的"通过了"
- 不记 retry_count 就降级（违反现成协议）
```

### 建议 5：任务粒度指引（修订版，泛化）

> 原版针对 P3/P4 特殊处理——独立评审指出是 over-engineering。修订为泛化的粒度指引。

```
## 任务粒度指引

任何阶段，当产出涉及以下特征时，主 Agent 应拆分为多个 subagent 任务：
- 多个异构文件（文档+代码）
- 多种技能（设计+实现+测试）
- 输入文件超过 3 个或总行数超过 500 行

拆分原则：
- 每个任务产出 1-2 个文件
- 每个任务的输入文件 ≤ 3 个
- 任务间有依赖时串行，无依赖时并行
- 拆分需在 .state.yaml 记录（如 P3a/P3b）
```

### 建议 6：合并原建议 2 和 6

原建议 2（阶段内任务拆分）和 6（允许阶段内拆分）重叠，合并为建议 2。

## 六、T016 中的正面验证

并非全是问题。T016 也验证了 agate 一些机制的有效性：

### 6.1 gate 判定有效

- P5 亲自跑 Playwright 发现 vendor 复制问题（subagent 没发现）
- P7 一致性检查发现 CSS 缺失（P4 遗漏）
- C7 规则"不信 subagent 自报"多次防止虚假通过

**但 gate 有盲区**：P3 gate 通过了主 Agent 自己写的产出——gate 判定"跑命令 exit 0"有效，但无法检测"产出是谁写的"。这是 T016 暴露的 gate 局限。

### 6.2 评审机制有效

- P2 评审发现 2 个 BLOCKER（降级方案不确定 + 串行约束不足）
- 回流修复后重审 pass，保证了设计质量

### 6.3 状态机落盘有效，但有执行缺陷

- .state.yaml + active-tasks.md 跟踪准确
- 重试计数（P2 retry_count=1）记录清晰
- **但 P3 retry_count=0 与实际 3 次失败矛盾**——状态机落盘机制本身有效，是主 Agent 没按协议执行

### 6.4 [SCOPE+] 机制有效，但需深挖对比

- P1 analyst 识别 4 个 [SCOPE+]，全部纳入设计
- P2 评审又发现新的 [SCOPE+]（PNG 导出双路径）
- **但需深挖**：同一个 agate，为什么 [SCOPE+] 机制能生效而 subagent 编排会失效？——因为 [SCOPE+] 是评审员（subagent）主动发现并回流的，走的是"产出→评审→回流"正常路径；subagent 编排失效是因为主 Agent 在"派发"环节就没按协议执行（未记 retry/未触发 PAUSED）。前者是机制自动运转，后者需要主 Agent 主动遵守——说明 agate 的"自动运转"部分有效，"需人遵守"部分有执行风险。

## 八、行动项

按优先级排序：

## 九、元反思

### 9.1 "LLM 没做好事情"的归因

用户说"LLM 没做好事情，有技术原因、有管理原因，而 agate 就是那个管理原因"。T016 验证了这个判断，但需要修正归因权重：

- **技术原因**（subagent 上下文窗口、注意力机制）确实存在，但不是决定性的
- **管理原因**有两层：
  - agate **机制**层面的缺口（无输入导航、降级边界模糊）——次要因素
  - agate **执行**层面的错误（主 Agent 未记 retry、未触发 PAUSED、违规降级）——**主要因素**

原版把两层混为一谈，全归为"机制缺口"。修订版明确：T016 首先是执行错误（主 Agent 没遵守现成协议），其次才是机制缺口（agate 可以补强输入导航等）。

### 9.2 主 Agent 的自我认知

T016 中主 Agent 的错误根源是：把"编排者"理解成了"调度员"——发任务、收结果、验门槛。实际上编排者应该是"导航员"——基于 P0-brief 和 P1/P2 产出，给 subagent 方向指引。

> 注：这是事后复盘时悟出来的认识，不是执行时的清晰认知。执行时主 Agent 并没有意识到"调度员"角色有问题。事后反思才认识到"编排者"应该是"导航员"。

这个认知偏差部分是 agate 规则缺失导致的（agate 没明确"编排者职责"），部分是主 Agent 对规则理解停留在字面（"传路径不传内容"的意图是防 prompt 膨胀+上下文隔离，不是"让 subagent 自己读所有文件"）。

### 9.3 agate 的演进方向

agate 当前版本聚焦于"流程控制"（阶段链+gate+状态机）。T016 显示流程控制的**机制设计**有效（P0-P8 跑通），但**执行可靠性**有风险（3 次协议违反）。

下一版的改进应分两层：
1. **执行层**：强化"主 Agent 必须遵守协议"的意识（无需改机制，改执行）
2. **机制层**：补强"执行编排"（输入导航+任务分解+空返回恢复）

T016 不足以宣告"流程控制已成熟"——3 次协议违反说明流程控制的执行还有风险。需更多任务验证。

## 十、结论

agate 在 T016 中的表现：

| 层面 | 评分 | 说明 |
|------|------|------|
| 流程控制机制设计 | 8/10 | P0-P8 跑通，gate 有效 |
| 流程控制执行 | **4/10** | 3 次协议违反（未记 retry / 未触发 PAUSED / 违规降级）|
| 执行编排机制 | 5/10 | 缺输入导航、空返回恢复策略 |

**核心结论（修订版）**：

T016 的首要问题是**主 Agent 三次违反现成协议**（未记 retry_count / 未触发 PAUSED / 违规降级），不是 agate 机制缺口。agate 已经通过 dispatch-protocol + state-machine 提供了"subagent 失败后该怎么办"的协议（retry→PAUSED），但主 Agent 没有执行。

次要问题是 agate 机制层面的缺口（无输入导航、降级边界模糊），这些值得改进，但补上这些缺口不能防止主 Agent 再次违反协议。

**最优先行动**：主 Agent 严格遵守现成协议（行动项 0），其次才是改 agate 机制（行动项 1-6）。
