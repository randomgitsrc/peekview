## 评审定位

前面所有评审（含上一轮 self-critique）都接受了 v3 的基本前提，只在前提内找问题。本轮质疑前提本身：

- v3 假设"派发子 Agent 一定比自己做省上下文"——真的吗？
- v3 假设"所有任务都该走 P1-P7"——真的吗？
- v3 假设"Header 靠 subagent 自觉写"——可靠吗？
- v3 假设"Txxx 就是任务目录名"——和现实一致吗？

找到 4 个元假设问题，其中 2 个有实际证据支撑。

---

## 问题 A（🔴 元假设错误）：派发不总是省上下文，小任务反而更费

**v3 的核心卖点**：派发 subagent 隔离上下文，主 Agent 只增加"路径+摘要"。

**但被忽略的成本**：每次派发，主 Agent 要**写出**一个完整的派发 prompt。实测 dispatch-prompt.md 模板约 25 行（角色定义路径 + 输入路径 + 任务 + 输出规范 + Header 要求 + 门槛 + 返回要求）。

**算一笔账**：
- 一个任务 7 阶段（加评审约 10 次派发）= 10 × 25 行 = **250 行派发 prompt 进主 Agent 上下文**
- 加上 10 次返回摘要、10 次门槛判定推理、每轮读 active-tasks.md

**对比"主 Agent 自己做一个小任务"**：
- 改个 typo、加个配置字段——自己做可能就 20 行上下文
- 走 v3 派发 = 250+ 行编排开销

**结论：派发有固定开销（每次约 25 行 prompt）。只有当"被隔离的内容" > "派发开销"时，派发才划算。** 对大文件（500 行，验证报告 Phase 3 那种）派发大赚；对小任务，派发的编排开销反而超过自己做。

v3 把"派发"当成无条件最优，这是错的。它有一个**盈亏临界点**。

### 修正建议
- README 加"何时用 v3"：v3 适合**多阶段、有大产出、需要评审门槛**的任务。小任务（单文件小改、typo、配置调整）不该走 P1-P7，直接做更高效
- 给一个粗略判据：如果任务产出 < 100 行且无需评审，不值得走完整 v3 流程
- 这不是 v3 的缺陷，是**适用边界**——但文档完全没提边界，会导致用户对小任务过度工程

## 问题 B（🟠 元假设缺失）：没有"轻量路径"，P1-P7 是唯一档位

承接问题 A：v3 只有"完整 P1-P7"一条路径。但真实任务的复杂度是光谱：

- **微任务**：改 typo、改文案 → 0 阶段，直接做
- **小任务**：加一个配置项、修一个明确的 bug → P1（定义）+ P4（实现）+ P5（验证）够了，不需要 P2 设计评审
- **中任务**：新功能 → 完整 P1-P7
- **大任务**：跨模块重构 → P1-P7 + 拆成多个子任务

v3 只服务"中任务"，对微/小任务过度，对大任务（上一轮问题 5）不足。

### 修正建议
定义**可裁剪的阶段**：
- 核心阶段（不可跳过）：P1 定义、P4 实现、P5 验证
- 可选阶段（按复杂度加）：P2 设计 + 评审（方案不明确时）、P3 测试先行（需要 TDD 时）、P6 一致性（多文件改动时）
- 在 P1 阶段，analyst 顺便判定任务复杂度，声明走哪些阶段（写进 task Header 的 `phases: [P1,P4,P5]`）
- 主 Agent 按声明的 phases 推进，不强制全 7 阶段

## 问题 C（🟠 实证不一致）：任务目录命名 Txxx vs 实际的 Txxx-描述

**v3 文档全程使用 `docs/tasks/Txxx/`**（纯编号，如 T002）。

**但实际仓库的任务目录是**（实证）：
```
docs/tasks/T001-mcp-namespace-map/
docs/tasks/T002-fix-db-migration/
```

带描述后缀（`Txxx-描述`），不是纯 `Txxx`。

**影响**：主 Agent 按 v3 文档拼路径 `docs/tasks/T002/P1-problems.md`，会找不到文件——实际在 `docs/tasks/T002-fix-db-migration/P1-problems.md`。**这是会导致派发直接失败的实证 bug，不是理论问题。**

文件内容本身已确认匹配（T002 目录下确实有 P1-problems.md、P2-design.md 等，命名和 v3 一致），唯一的错位是**任务目录名**。

### 修正建议（必修）
- v3 文档统一改为 `docs/tasks/{task_id}/`，并说明 task_id 格式是 `Txxx-描述`（如 T002-fix-db-migration）
- 或在 README 明确："任务目录名是 Txxx-描述，文档中的 Txxx 是占位简写，实际拼路径用完整目录名"
- dispatch-prompt 模板的 `{Txxx}` 占位符改为 `{task_dir}`，提示用完整目录名

## 问题 D（🟡 元假设乐观）：Header 完全靠 subagent 自觉写，无生成辅助

v3 多处依赖 Header（phase/task_id/parent/trace_id）做门槛判定。dispatch-protocol 也有"Header 不完整→门槛不通过→重试"的校验。

**但 Header 完全靠 subagent 手写。** trace_id 格式 `{Txxx}-{Pn}-{YYYYMMDD}` 要 subagent 自己拼，日期自己填。LLM 写这种结构化元数据容易出错（日期写错、格式不一致、漏字段），导致门槛校验失败、无谓重试。

这不是结构性问题，但会增加重试率，降低自动化流畅度。

### 修正建议
- 派发 prompt 里**直接给出完整的 Header**让 subagent 照抄，而不是给格式让它自己拼：
  ```
  你的产出文件必须以这段 Header 开头（直接复制）：
  ---
  phase: P2
  task_id: T002-fix-db-migration
  parent: P1-problems.md
  trace_id: T002-P2-20260612
  ---
  ```
- 主 Agent 派发时已经知道所有这些值（它在编排），直接填好给 subagent，消除 subagent 拼错的可能
- dispatch-prompt.md 模板更新：Header 从"要求格式"改为"给出成品"
