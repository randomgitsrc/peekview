### 8.2 重复出现的问题模式

**模式 1：P6 验收的"门禁陷阱"**
- 每次 P6 验收都花大量时间在 gate 格式调整上
- T048：3 轮评审，部分原因是 gate 格式要求
- T052：4 次格式修正才通过 provenance 审计
- **根因**：gate 检查格式正确性 > 功能正确性

**模式 2：发布后视觉回归**
- T049：3 个问题（发布后才发现）
- T052：2 个问题（发布后才发现）
- **根因**：P6 验收环境（debug backend）与生产环境（登录态 + 真实数据）有 gap

**模式 3：主 Agent 越俎代庖**
- T052 P5/P6：主 Agent 亲自写验证脚本和验收报告
- T048 P7：主 Agent 遗漏 DESIGN_GAP 配对
- **根因**：子 Agent 产出质量波动大，主 Agent 遇到困难时倾向亲自上手（违反 agate 规范）

**模式 4：任务拆解颗粒度**
- T048 → T051（续命） → T052（独立）：好的拆解
- T049（含移动端 + 图表清洗 + config 三件事）：拆得太粗
- T050（专做归零）：正确的拆解

## 9. Agate 工作流评估

### 9.1 做得好的

- **P0-brief 机制**：强制在任务开始前列风险、约束、裁剪倾向，减少范围 creep
- **P2 方案设计**：多候选方案 + 评审减少了实现方向的错误（T051 评审几轮都不多）
- **P5 gate 重跑**：每次版本 bump 后重跑测试有效防止了回归（T051→T052 的 811 测试一直绿）
- **子 Agent 专业化**：backend/frontend/general 分工在 T051 P4 并行派发时效果明显
- **状态绑定检查**：`.state.yaml` + active-tasks.md 一致性检查防止了阶段混乱

### 9.2 有问题的

| 问题 | 频率 | 严重度 | 建议 |
|------|------|--------|------|
| Provenance 审计拦截 commit（格式问题） | 每次 P6 | 高 | 降低为 WARNING，不做 commit 阻塞 |
| P6 verifier 产出格式与 gate 期望不一致 | 每次 P6 | 中 | 在 verifier role 中固定产出格式模板 |
| 发布后才发现视觉问题 | T049, T052 | 高 | P6 验收环境必须包含登录态 + 真实数据 |
| gate 调格式占 > 30% 的 P6 时间 | 所有任务 | 中 | 简化格式检查或提供自动格式化工具 |
| 子 Agent 分阶段落盘（progress.md）总是空的 | 所有任务 | 低 | 检查 subagent 模板是否正确注入落盘指令 |

### 9.3 超时/中断的影响

T052 执行过程中出现了多次 tool abort（Edit/Bash 卡住），推测原因：
- **Bash heredoc 超长**：`cat > file << 'SCRIPT'` 含 100+ 行 TS 代码 → shell 阻塞
- **Edit oldString 不精确**：文件已因之前编辑而改变行号，oldString 找不到
- **`make debug` 类命令未设 timeout**：默认 120s 限制触发

## 10. 改进建议

### 10.1 给 agate 协议

1. **Provenance 审计降级为 WARNING（exit 2）**：格式问题不阻塞 commit，只在 pre-commit 输出 warning。功能问题由 P6 验收决定，不是由格式检查决定。
2. **固定 P6-acceptance.md 产出模板**：在 verifier role 中嵌入标准 YAML frontmatter + PASS 行格式，减少格式偏差。
3. **P6 验收环境规范中加入"登录态测试"**：ui_affected = true 的任务必须在至少一种登录态下验收（可用 API key 模拟）。
4. **增加"发布前最终检查"清单项目**：包括"stop debug service"、"verify no hardcoded colors"、"test in light mode"等。

### 10.2 给主 Agent 执行策略

1. **subagent 返回空时先查 progress.md**——不要立即原样重试，也不要降级亲自做
2. **E2E/Playwright 脚本用 Write 工具创建**（非 bash heredoc）——减少 shell 阻塞风险
3. **Edit 前先 `grep -n` 确认 oldString 唯一性**——避免匹配失败
