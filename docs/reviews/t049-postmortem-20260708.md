## 2. 技术归零

### 2.1 定位准确

#### 问题 A：`config get` 默认值不显示

- **现象**: `peekview config list` 显示 `sanitize_enabled: True`，同命令 `config get diagram.sanitize_enabled` 返回 `(not set)`
- **位置**: `backend/peekview/cli.py`，`config_get` 命令中的 `get_default` helper 函数（第 651-666 行）
- **根因**: 该函数的 `if/elif` 链缺少 `section == "diagram"` 分支。同名函数 `_get_default`（用于 `config list`）则正确包含该分支
- **机理**: P4 实现时，开发者在 `config_list` 的 `_get_default` 中添加了 `diagram` 分支，但在 `config_get` 的 `get_default` 中遗漏——两个函数结构实质相同，修改时只改了一处
- **复现条件**: 任何没有在 config file 中显式设置 `diagram.sanitize_enabled` 的环境

#### 问题 B：清洗规则不全面

- **现象**: 开启 `diagram.sanitize_enabled = True` 后，前端仍显示 Mermaid 渲染错误（UnknownDiagramError、ParseError、LexicalError）
- **位置**: `frontend-v3/src/utils/diagramSanitize.ts`，registerRule 注册的规则集
- **根因**: 
  - 规则集由 P2 设计者凭经验列出 7 条规则，未经真实数据验证
  - P3 测试用例只覆盖已实现的规则，未发现缺失
  - P6 验收使用人工构造数据（已知可修复的错误），而非用户实际条目内容
  - 未分析的错误模式包括：`gitgraph` 大小写、`TBsubgraph` 缺换行、全角括号/引号、智能引号、HTML 标签混合

#### 问题 C：移动端 header 布局

- **现象**: 登录状态下，`header-right` 有 4 个按钮（Expires in 15d + Edit + TOC + Theme toggle），宽度远超预期，挤压 `header-tags` 区域
- **位置**: `frontend-v3/src/views/EntryDetailView.vue` + 相关 CSS
- **根因**:
  - P2 设计假设 `header-right` 宽度 120px，未计算登录态所有按钮的实际宽度
  - P6 测试仅在非登录态执行，未暴露宽度挤压问题
  - CSS 使用 `flex-shrink: 1`（默认）使 `title-group` 可被压缩

### 2.2 机理清楚

三层技术原因叠加：

```
P2 设计阶段
  └─ 凭经验猜测规则，未收集真实数据
     └─ P3 测试只覆盖"有的规则"，未挑战"缺的规则"
        └─ P4 实现遗漏 get_default 分支（copy-paste 没改全）
           └─ P5 测试全绿（测的都是"做了的"）
              └─ P6 用人工数据验收，未用真实条目
                 └─ P7 一致性检查机械核对格式，未问"这真的解决问题了吗"
                    └─ P8 发布 → 用户验证立即发现问题
```

每个阶段都通过了 gate，但 gate 只检查格式/存在性，**不检查方案有效性**。

### 2.3 问题复现

#### 复现 A
```bash
peekview config list | grep sanitize          # 显示 True
peekview config get diagram.sanitize_enabled   # 显示 (not set)
```

#### 复现 B
```bash
# 创建含错误 gitgraph 的条目
curl -X POST http://127.0.0.1:8888/api/v1/entries \
  -H "Content-Type: application/json" \
  -d '{"slug":"test-gitgraph","summary":"test","content":"```mermaid\ngitgraph\n  commit id: \"A\"\n```","is_public":true}'
# 访问 http://127.0.0.1:8888/test-gitgraph
# → 显示 UnknownDiagramError
```

#### 复现 C
登录 http://127.0.0.1:8888（JWT 有效），访问任何含多个 tag 的条目，移动端模式（<768px）。

## 3. 管理归零

### 3.1 过程清楚

| 阶段 | 实际执行 | 应有行为 | 偏差 |
|------|---------|---------|------|
| P1 (analyst) | 写 BDD，未要求分析真实错误数据 | 应收集用户真实条目中的错误 diagram 源码 → 归纳模式 | 未跳出"需求规格说明书"思维 |
| P2 (architect) | 设计规则架构 + 列出 7 条规则 | 应基于 P1 的错误模式分析设计规则，对每类引擎回答"覆盖了哪些模式/遗漏了哪些" | 规则数量和质量未设底线 |
| P4 (implementer) | 实现 spec 所有条目 | 实现过程中应交叉验证——`config_get` 与 `config_list` 是否一致 | copy-paste 遗漏 |
| P5 (verification) | pytest + vitest 全绿 | 应增加基于真实条目的集成测试 | 测试设计被 spec 局限 |
| P6 (acceptance) | 手写 test mermaid code → CDP 测试 | 应用真实条目内容作为验收数据；mobile 应测试登录/非登录两态 | 验收数据失真 |

### 3.2 责任明确

**第一责任人：主 Agent（我自己）**

不是 subagent 的错。subagent 按我给的 prompt 工作，产出符合 prompt 要求。问题出在：

1. **P1 派发 prompt 没要求 analyst 分析真实数据** — 我说"写需求基线"就够了，没指定"去用户生产实例收集错误 pattern"
2. **P2 派发 prompt 没问 architect"这些规则够全面吗？"** — 接受了"先做基础规则"的默认路径
3. **P3/P4 派发没提供足够导航** — dispatch-context 只说了"实现这个模块"，没说要交叉验证 config_get 一致性
4. **P6 验收数据我自己选的** — 不是 subagent 选的，责任在我
5. **P7 一致性检查我做的** — 机械地配对 BDD 数量、文件路径，没质疑方案有效性

**Agate 流程没有阻止我** — 但 Agate 的 gate 设计为格式检查，不负责判断方案质量。这是对 gate 用途的误解。

作为主 Agent（Orchestrator），我是 T049 失败的**第一责任人**。以下是我承认的错误：

每个阶段我都在问"gate 过了吗？文件写了吗？"而没有问"用户的真实问题解决了吗？"

- P4 implementer：应该验证 `config_get` 与 `config_list` 一致性，我没有提示

我亲手构造了 `p6-acceptance.spec.ts` 中的测试数据。如果我花了 10 分钟从生产实例拉几个真实条目当测试数据，问题 B 和 C 都能发现。

### 5.4 心态错误：赶进度

T049 做了一整天（11 小时），到 P6 时我倾向于"尽快完成"，验收标准降低了。具体表现：
- 构造简单的人工测试数据（10 分钟搞定），而不是拉真实数据（需要 curl/production access，但也要做）

- P7 一致性检查 10 分钟搞定，没有深入质疑

### 5.5 为什么赶进度？

- 已经在单个任务上花了大量时间
- 想尽快出结果展示给用户
- 对"P6 过了等于产品 OK"的过度信任

Agate 提供的 gate 机制是正确的——它防止格式错误，确保流程完整。**问题不在 gate 不够严格，而在我（Orchestrator）对 gate 的误用：把 gate 通过当作"任务完成"的标准，而不是"任务正确性"的标志。**

## 6. Agate 流程问题

虽然主责任在我，但 Agate 流程也有一些值得改进的地方：

### 6.1 gate 类型单一

当前 gate 全是否定型（exit 1/0），没有"这合理吗"的开放式检查。`check-gate.sh` 检查文件存在、格式、BDD 数量——这些都是机械检查。

**建议**：增加"有效性 gate"——在 P2 后、P6 后问"列出 3 个你认为方案可能失败的方式"，作为无害检查（WARNING，不 exit 1）。

### 6.2 没有"真实数据检查"的要求

Agate gate 没有检查"验收数据是否包含真实/生产数据"。这是流程级漏洞——明知人工数据会失真，但没有机制阻止。

**建议**：P6 gate 增加可选检查：如果任务涉及"规则/匹配/分析"，要求至少 50% 测试用例来自生产数据。

### 6.3 缺少"跨命令一致性"检查

两个相似功能的 CLI 命令（`config_get` 和 `config_list`）行为不一致，agate gate 无法检测。这是已知问题（T016/T022 也出现过）。

**建议**：P4 gate 增加"对相似逻辑/重复代码的一致性检查"。

## 8. 举一反三——T050 设计要求

T049 暴露的每个问题在 T050 中必须有对应拦截机制：

### 8.1 适用于其他任务的检查点

| 检查点 | 问题 | 适用场景 |
|--------|------|---------|
| P1 真实数据收集 | 凭经验猜测规则 | 涉及规则/分类/分析的任何任务 |
| P4 交叉一致性 | copy-paste 遗漏 | 两个类似函数/命令有重复逻辑 |
| P6 真实数据验收 | 人工数据失真 | 涉及前端渲染效果的任务 |
| P6 多态测试 | 漏掉登录/非登录态 | 涉及 UI 条件渲染的任务 |
| P7 有效性审查 | 机械 gate 不发现本质问题 | 所有任务 |

### 8.2 T050 具体要求

1. **P1**: 必须收集用户真实条目中的错误 diagram 数据，列成表格，作为规则设计的输入
2. **P2**: 输出完整的 mermaid/plantuml/svg 错误模式对照表（常见错误 × 规则覆盖 × 规则缺失）
3. **P4**: 实现后必须验证 `config_get` 与 `config_list` 行为一致（自动化测试）
4. **P5**: 增加基于真实条目的集成测试
5. **P6**: 验收测试必须包含：
   - 至少 3 个有真实错误的条目（从生产实例收集）
   - 登录态 + 非登录态的移动端 header 测试
   - 开启/关闭清洗的对比测试
6. **P7**: 必须回答"T050 的解决方案是否覆盖了以下真实错误模式？"并列出对照表
