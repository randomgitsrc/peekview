## 1. 为什么没用 agate？

### 决策依据

用户报告的是明确的 UI bug（"点击无反应"、"label 未对齐"、"图标不显示"），不是需求或设计问题。判断为"微/小任务（单文件改动、简单 bug fix）直接做"——这是 AGENTS.md 明确允许的：

> 微/小任务（单文件改动、简单 bug fix）直接做

### 判断失误

这个判断**部分错误**。Bug #3 的修复涉及 ShareDialog 组件的定位机制重构（从 `position: absolute` + `display:contents` → Teleport + fixed 定位 + JS 位置计算），实际上是**架构级改动**，不是"简单 bug fix"。如果按 agate 流程走，P2 阶段会强制评估候选方案（至少 2 个），可能发现 Teleport 方案引入的新问题（如 bug #3b）。

**具体失误**：
- Bug #3 的根因分析（sticky header + absolute 定位 + sibling click-outside）涉及三个独立机制交叉，复杂度超出了"直接做"的阈值
- Bug #3 的修复同时引入了新 bug（#3b），形成"修一个冒一个"的挤牙膏模式——这正是 agate 要避免的

### 时间黑洞：Bug #3b

Bug #3b（popover 内按钮点击导致 popover 关闭）消耗了全过程的 **24%** 时间。根因是 Vue 响应式更新与 DOM 事件冒泡的时序冲突——这是一个**不可能通过直觉预判**的问题，必须实际运行才能发现。

调试路径：
1. Playwright `click()` → popover 消失 ❌（0.5h）
2. JS `btn.click()` → popover 不消失但视图不切换 ❌（误导：以为 Vue handler 未触发）
3. 捕获阶段日志 → 所有事件 `inPopover=true` → 但 popover 仍消失 ❌（矛盾点）
4. 直接在 `handleClickOutside` 内加详细日志 → `popoverRef.value.contains(e.target)` 返回 false ✅（根因定位）
5. 进一步调查 → `popover.contains(btn)` 在 click 时刻返回 false，因为 Vue 已移除按钮 ✅（完整根因链）

**教训**：当 DOM API 返回值与直觉矛盾时（"按钮在 popover 里，但 contains 返回 false"），应立即考虑 DOM 结构变化（Vue 响应式更新）的时序问题，而不是继续排除 CSS/事件问题。

## 5. 直接修复的适用边界

AGENTS.md 说"微/小任务直接做"。但本迭代的经验表明，**判断"简单"本身就是风险最高的决策点**。

### 适用"直接做"的场景

- 单行 CSS 修复（如 `text-align:left`）
- 配置值修改（如 z-index）
- 明确的单点逻辑错误（如 `==` 改 `===`）

### 不适用"直接做"的场景

- 涉及**多个机制交叉**的 bug（如 #3：sticky header + absolute 定位 + click-outside）
- 修复可能**改变组件定位/渲染机制**（如 #3：absolute → Teleport+fixed）
- 修复**涉及 DOM 事件时序**（如 #3b：Vue 响应式 vs 事件冒泡）
- 被修组件有**多种交互路径**（如 ShareDialog：打开/关闭/内部导航/创建/撤销）

### 判断标准

**三问法**：
1. 修复是否只改 CSS/配置？→ 可直接做
2. 修复是否涉及 JS 逻辑变更？→ 谨慎，先列测试场景
3. 修复是否涉及组件定位/渲染/事件机制？→ **必须走 agate**

Bug #3 在第 3 问就应触发 agate 流程。

## 6. 具体技术教训

### 6.1 `display:contents` + `position:absolute` 陷阱

`display:contents` 使元素在布局中消失（无 box），但其子元素正常渲染。`position:absolute` 的定位上下文是最近的 positioned ancestor——如果 `display:contents` 的父元素没有 `position:relative`，absolute 元素会跳到更上层的 positioned ancestor（如 sticky header）。

**解决方案**：Teleport + `position:fixed` + JS 计算位置。这是通用模式，未来类似 popover/dropdown 都应采用。

### 6.2 Vue 响应式 vs DOM 事件冒泡时序

Vue 3 的响应式更新在 microtask 中执行（`nextTick` 前），可能在 click 事件冒泡到 document 之前完成。如果 `@click` handler 改变了 `v-if` 条件导致 DOM 节点被移除，`document.addEventListener('click', ...)` 的 handler 收到的 `e.target` 已经是 disconnected node。

**检测方法**：`e.target.isConnected === false`

**通用规则**：任何 `document.addEventListener('click', handleClickOutside)` 的实现，都必须处理 target 被 Vue 移除的情况。

### 6.3 SVG 在 flex 容器中的 width 问题

SVG 元素的 `width` HTML attribute 在某些 flex 布局上下文中不被浏览器正确计算为 computed style。具体表现为 `svg width="18"` 但 `getComputedStyle(svg).width === "0px"`。

**解决方案**：给 SVG 显式设置 CSS `width` 和 `height`（不仅依赖 HTML attribute）。

**根因推测**：LandingView 的 scoped `.btn` 重定义覆盖了全局 `.btn` 的部分属性，虽然不直接设置 SVG width，但可能改变了 flex 容器的布局计算方式，导致浏览器的 SVG width 计算出错。ThemeToggle 使用 scoped CSS 后不再依赖全局 `.btn`，问题消失。

### 6.4 Playwright vs JS `.click()` 的差异

Playwright 的 `locator.click()` 产生完整的鼠标事件序列（pointerdown → mousedown → pointerup → mouseup → click），而 `element.click()` 只触发 click 事件。某些 bug 只在完整事件序列下复现（如 #3b：Vue 在 mousedown/mouseup 之间可能触发响应式更新）。

**规则**：验证交互功能时必须用 Playwright 的真实 click，不能只用 JS `element.click()`。

## 10. 结论

本迭代是"直接修复"决策失误的典型案例。Bug #3 看似简单（"点击无反应"），实际涉及三个机制交叉，修复引入了新 bug，形成挤牙膏模式。如果走 agate 流程：

- P1 会列出完整 BDD 条件（包括 popover 内交互），不会漏掉 #3b
- P2 会评估至少 2 个候选方案，可能提前发现 Teleport 方案的时序风险
- P6 逐条实跑 BDD，一次发现所有问题
- 总耗时可能更短（2.5h vs 4h），因为不需要返工

**核心教训**：判断"简单"本身就是风险最高的决策。当修复涉及组件定位/渲染/事件机制时，无论 bug 表象多简单，都必须走 agate。
