## 6. 测试基础设施改进

### 6.1 当前问题

| 问题 | 影响 | 建议 |
|------|------|------|
| 无代码覆盖率阈值 | 覆盖率可能逐步下降 | 设定最低覆盖率 ≥ 80% (后端)、≥ 70% (前端) |
| conftest.py fixture 不完整 | `test_client` 用了 `AsyncClient` 但 fixture 非 async | 改用 `@pytest_asyncio.fixture` + `httpx.AsyncClient` |
| 无测试数据工厂 | 每个测试手动构造数据，冗余且易错 | 引入 factory 模式（如 `EntryFactory.create()`） |
| 无 mock 策略定义 | 文件系统/数据库/外部依赖 mock 不一致 | 明确 mock 边界：文件系统用 temp dir，DB 用内存 SQLite，外部不依赖 |
| CI 无 lint/type-check | 代码质量无门禁 | 添加 ruff/mypy 到 CI |
| 前端 jsdom 可能不支持 Shiki | Shiki 依赖浏览器 API | 关键高亮测试改用 `happy-dom` 或 Playwright |
| 无测试并行化 | 测试套件变慢 | pytest-xdist (后端)、Vitest 内置 (前端) |
| SQLite 测试隔离不明确 | 测试间可能残留数据 | 推荐使用 `:memory:` 内存数据库 + 每测试重建 |
| 无 pre-commit hook | 质量问题可能绕过 CI | 添加 pre-commit: lint + type-check + 快速测试 |

@pytest.fixture

    """预置样本文件（各类型）"""

## 8. 安全测试需求

### 8.1 关键判断

> 需求规格写"无认证（本地使用）" ≠ 无安全测试

Peek 接受外部输入（文件路径、文件内容、slug、tags），存在以下攻击面。

### 8.2 必须覆盖的安全测试

| # | 攻击类型 | 测试场景 | 优先级 |
|---|----------|----------|--------|
| 1 | **路径遍历** | `local_path: "../../etc/passwd"` → 拒绝 | P0 |
| 2 | **路径遍历** | `path: "../../../etc/shadow"` → 拒绝 | P0 |
| 3 | **路径遍历** | `path: "..%2F..%2Fetc%2Fpasswd"` (URL编码) → 拒绝 | P0 |
| 4 | **SQL 注入** | `tags: ["'; DROP TABLE entries; --"]` → 无害化 | P0 |
| 5 | **SQL 注入** | `search?q=' OR 1=1 --` → 无害化 | P0 |
| 6 | **XSS（存储型）** | `summary: "<script>alert(1)</script>"` → 前端转义 | P0 |
| 7 | **XSS（Markdown）** | Markdown 中嵌入 `<img onerror=alert(1)>` → 转义 | P0 |
| 8 | **XSS（代码高亮）** | 文件内容含 `<script>` → 转义 | P1 |
| 9 | **SSRF** | `local_path: "http://internal-service/admin"` → 拒绝 | P1 |
| 10 | **符号链接攻击** | `local_path` 指向 symlink → 跟随 or 拒绝 | P1 |
| 11 | **DoS — 大文件** | 上传 1GB 文件 → 限制大小 | P1 |
| 12 | **DoS — 大量请求** | 无限创建条目 → 速率限制？ | P2 |
| 13 | **信息泄露** | 错误响应不含堆栈/SQL/内部路径 | P1 |
| 14 | **数据库文件暴露** | `GET /peek.db` → 404 (不在静态文件范围) | P1 |
| 15 | **CORS 配置** | 跨域请求策略验证 | P2 |
| 16 | **Content-Type 验证** | 非法 Content-Type → 拒绝 | P2 |

### 8.3 Markdown 安全特化

## 11. 总结

测试计划覆盖了核心正向流程，但存在以下系统性缺陷：

1. **安全测试完全缺失** — 这是最高优先级缺口，本地服务≠免疫攻击，路径遍历/XSS/注入风险必须覆盖
2. **边界值和异常输入严重不足** — 现有测试以"Happy Path"为主，缺少输入验证类测试
3. **服务层零覆盖** — entry_service/file_service 是 API 和 CLI 的共享核心，绕过它们等于绕过了最重要的业务逻辑
4. **性能指标无验证** — 需求规格承诺了性能指标，测试计划没有对应验证
5. **前端交互测试浮于表面** — 复制/下载等核心交互未验证实际效果，错误态/加载态完全未覆盖
6. **集成测试场景不足** — 仅 3 个场景，跨系统/跨入口一致性未验证

**建议：在开发启动前补充 P0 级测试用例定义，避免"先写代码后补测试"导致安全漏洞和缺陷遗漏。**
