## 总体评价

文档整体质量**良好**，功能覆盖面广，架构选型合理，前后端分离清晰。核心定位"Agent 产出 → Peek 格式化 → 人类友好查看"表达准确。但在以下维度存在明显不足：需求完整性（缺少多个关键场景）、用户故事缺失、边界条件处理不足、竞品对比差距明显。以下按评审维度逐一展开。

**总体评分：6.5 / 10**

## 一、需求完整性 — 有无遗漏场景

### 🔴 严重遗漏

| # | 遗漏项 | 说明 | 影响 |
|---|--------|------|------|
| 1 | **单文件内容大小限制** | 需求未定义单个文件的最大体积（如 1MB? 10MB? 100MB?），技术设计也无限制 | 恶意/意外上传超大文件可能耗尽磁盘或拖慢渲染 |
| 2 | **条目文件数量上限** | 一个条目可包含多少文件？100? 1000? 无限制？ | 大量文件的条目会导致目录树渲染缓慢、API 响应膨胀 |
| 3 | **条目总大小限制** | 一个条目的总存储空间上限 | 同上，且 zip 打包可能 OOM |
| 4 | **全局存储配额** | 整个实例的数据量上限 | 本地部署场景下磁盘可能被撑满 |
| 5 | **重复内容检测** | 上传完全相同的文件/条目无任何去重 | Agent 可能重复创建相同内容，浪费存储 |
| 6 | **条目更新历史** | 已列入"不在范围内"，但至少应保留 last_updated 语义 | 用户无法判断内容是否被修改过（updated_at 虽有但无 diff） |
| 7 | **API 限流/速率限制** | 无认证 + 无限流 = 任意客户端可无限创建条目 | 本地部署也需防护（脚本误操作、Agent 循环调用） |
| 8 | **错误信息规范** | API 错误响应格式未定义 | 前端和 Agent 无法统一处理错误 |

### 🟡 中等遗漏

| # | 遗漏项 | 说明 |
|---|--------|------|
| 9 | **文件编码检测** | 仅默认 utf-8，未考虑 GBK/GB2312/Shift-JIS 等编码；files 表有 encoding 字段但无检测逻辑 |
| 10 | **空文件处理** | 0 字节文件的展示和交互逻辑未定义 |
| 11 | **符号链接处理** | `local_path` 引用是否解析 symlink？安全风险 |
| 12 | **条目分享权限** | `published` 状态的语义不清——是否表示公开可访问？与 active 的区别？ |
| 13 | **批量操作** | 无批量删除、批量打标签功能；索引页管理效率低 |
| 14 | **条目排序** | 索引页仅提到"分页"，未定义默认排序规则（设计文档有 created_at 倒序，需求文档缺失） |
| 15 | **搜索范围** | "关键词搜索概述和文件内容"——是否搜索 Markdown 渲染后的纯文本？是否搜索文件名？范围模糊 |
| 16 | **标签管理** | 仅 Agent 可打标签，无标签重命名/删除/合并，无标签使用统计 |
| 17 | **导出/导入** | 无数据导出导入机制（迁移场景） |
| 18 | **前端 URL Hash/Query** | 是否支持 `?file=main.py` 直接定位到某文件？分享场景高频需求 |
| 19 | **Mermaid 渲染失败降级** | Mermaid 语法错误时如何展示？需求未提及 |
| 20 | **图片文件展示** | 需求提到"支持图表、图片"但仅在 Markdown 语境，独立图片文件（.png/.jpg/.svg）如何展示未说明——只给下载链接还是内联显示？ |

### 🟢 建议补充

| # | 遗漏项 | 说明 |
|---|--------|------|
| 21 | **OpenAPI 自动文档** | FastAPI 自带，但需求未提及是否暴露 /docs |
| 22 | **健康检查端点** | `/health` 或 `/ready` 端点未定义 |
| 23 | **WebSocket/实时更新** | 已排除，但应说明 Agent 更新条目后前端是否需手动刷新（当前暗示手动刷新，需明确） |
| 24 | **文件行数统计** | 代码文件展示行数信息（如"共 128 行"），增强可读性 |
| 25 | **打印友好样式** | 代码分享场景可能需要打印/PDF 导出 |

## 二、用户故事清晰度

### 🔴 严重问题：**用户故事完全缺失**

三份文档均未使用用户故事（User Story）格式描述需求。功能需求以功能列表形式列出，缺乏：

- **谁**（作为 Agent / 人类用户）
- **要做什么**（创建/查看/管理）
- **为什么**（价值/目标）
- **验收标准**（完成定义）

**影响**：开发团队无法从需求文档直接理解用户意图和验收标准，容易产生实现偏差。

### 建议补充的核心用户故事

```
US-01: 作为 Agent，我想创建一个包含多文件的代码展示条目，
       以便人类用户可以通过链接查看格式化后的代码。

US-02: 作为 Agent，我想指定自定义 URL slug，
       以便分享的链接具有可读性和语义。

US-03: 作为人类用户，我想在浏览器中查看语法高亮的代码，
       以便快速理解代码内容，无需下载和打开 IDE。

US-04: 作为人类用户，我想在索引页搜索和过滤条目，
       以便快速找到我需要的内容。

US-05: 作为 Agent，我想设置条目过期时间，
       以便临时展示的内容能自动清理。

US-06: 作为人类用户，我想复制代码片段到剪贴板，
       以便在 IDE 中直接使用。

US-07: 作为人类用户，我想在手机上查看代码内容，
       以便在移动场景下也能访问。

US-08: 作为 Agent，我想通过 MCP 协议调用 Peek，
       以便无需了解 REST API 细节即可集成。

US-09: 作为人类用户，我想下载条目中的文件或打包下载，
       以便离线查看或编辑代码。

US-10: 作为人类用户，我想在深色/浅色模式下查看内容，
       以便适配我的阅读习惯和环境光线。
```

每个用户故事应附带**验收标准（Acceptance Criteria）**，如：

> **US-03 验收标准**：
> - 代码块有行号
> - 支持自动换行切换
> - 语言根据文件扩展名自动检测
> - 高亮效果与 VSCode 一致
> - 加载 1000 行代码文件首屏 < 500ms

## 三、优先级合理性

### 当前优先级评估

需求文档**未定义功能优先级**，仅在测试计划中有 P0/P1/P2 分级。这是需求规格的重大缺陷——优先级不应只在测试计划中体现。

### 优先级调整建议

| 功能 | 建议优先级 | 理由 |
|------|-----------|------|
| 创建条目（API + CLI） | **P0** | 核心功能，无此无法使用 |
| 条目详情查看（前端渲染） | **P0** | 核心价值，无此 Agent 产出无法展示 |
| 代码语法高亮 | **P0** | 核心体验，定义产品差异化 |
| 索引页（条目列表） | **P0** | 基础导航，否则无法管理条目 |
| 文件内容大小/数量限制 | **P0** ⚠️ | **当前缺失，必须补入 P0**，防止资源滥用 |
| 错误响应规范 | **P0** ⚠️ | **当前缺失**，Agent 和前端无法正确处理异常 |
| 搜索与标签过滤 | **P1** | 提升查找效率，条目少时非必须 |
| 更新条目 | **P1** | Agent 修正输出场景有用，但可重建替代 |
| 过期清理 | **P1** | 自动化运维，手动删除可临时替代 |
| 打包下载（zip） | **P1** | 便利功能，非核心 |
| Mermaid 渲染 | **P1 → P2** | 锦上添花，实现复杂度高，建议降级 |
| MCP Server | **P1** | Agent 集成是核心定位，但 CLI 可先替代 |
| 存档/发布状态管理 | **P2** | 简单场景用不到 |
| 响应式/移动端适配 | **P1 → P2** | 核心场景是桌面浏览器查看，移动端可延后 |
| 主题切换 | **P2** | 默认暗色即可，切换非必须 |

### 🔴 关键问题：MVP 边界不清

文档未明确 MVP（最小可行产品）范围。建议定义：

**MVP（v0.1）= P0 功能**：
- API: 创建/查看/删除条目、文件下载
- CLI: create / list / get / delete
- 前端: 条目详情页（代码高亮 + Markdown 渲染）、索引页（基础列表）
- 限制: 文件大小上限、条目文件数上限、错误码规范

## 四、边界条件与异常场景

### 🔴 未覆盖的关键边界条件

| # | 场景 | 当前处理 | 建议处理 |
|---|------|---------|---------|
| 1 | **local_path 指向不存在的文件** | 未定义 | 返回 400 错误，提示文件不存在 |
| 2 | **local_path 路径遍历攻击** (`../../etc/passwd`) | 测试计划提到路径遍历防护 | 需求/设计应明确安全约束：必须在允许目录内 |
| 3 | **文件内容为空** | 未定义 | 允许创建空文件，前端显示"空文件"提示 |
| 4 | **超大文件上传** | 未定义 | 定义限制（如单文件 10MB），超限返回 413 |
| 5 | **条目无文件** | 空文件列表返回 400 | ✅ 已覆盖，但需确认"仅概述无文件"是否合法 |
| 6 | **slug 包含特殊字符** | 未定义 | 规范 slug 格式（如 `[a-z0-9_-]`），拒绝非法 slug |
| 7 | **随机 ID 冲突** | "不冲突"一笔带过 | 短随机码有碰撞概率，需定义冲突检测和重试逻辑 |
| 8 | **同时指定 slug 和过期时间** | 无异常 | ✅ 正常组合，但需验证 |
| 9 | **更新已过期条目** | 未定义 | 过期条目是否可更新？应禁止 |
| 10 | **删除正在被查看的条目** | 未定义 | 前端应处理 404 优雅降级 |
| 11 | **SQLite 并发写入** | 未讨论 | SQLite 单写锁，Agent 并发创建时可能失败，需 WAL 模式 + 重试 |
| 12 | **磁盘空间不足** | 未定义 | 写入文件时磁盘满的处理策略 |
| 13 | **文件名包含 Unicode/中文** | 未定义 | 磁盘存储路径如何处理中文文件名？ |
| 14 | **base64 内容格式错误** | 未定义 | 声明 binary 但 base64 解码失败时的错误处理 |
| 15 | **zip 打包超大条目** | 未定义 | 打包内存限制，可能 OOM |
| 16 | **Shiki 不支持的语言** | 未定义 | 应回退到纯文本展示 |
| 17 | **Markdown 中恶意 HTML/JS** | 未定义 | markdown-it 需配置安全策略，防 XSS |
| 18 | **Mermaid 渲染超时** | 未定义 | 复杂图表渲染可能卡死，需超时降级 |
| 19 | **条目 ID 与 slug 混用** | entries.id 可以是随机码或 slug，逻辑混乱 | 建议分离：id 始终为内部 ID，slug 为可选别名 |

### 🔴 entries.id 设计问题

当前设计中 `id` 字段既做主键又承载 slug 语义（短随机码或 slug），这导致：
- 更新 slug 时主键变更，所有外键需级联更新
- slug 冲突时 `id` 被追加后缀，语义混乱
- API 路径 `/entries/{id}` 中 id 和 slug 含义混用

**建议**：`id` 始终为不可变内部标识（如 UUID 或随机码），`slug` 为独立的可选唯一字段，API 支持 `/entries/{id}` 和 `/entries/slug/{slug}` 双路径。

### 🔴 关键功能缺失（建议纳入 roadmap）

1. **代码行高亮/标记**
   - Carbon/ray.so 的核心功能：高亮某几行代码来引导注意力
   - URL hash 支持 `#L5-L10` 行范围选择
   - **优先级：P1**，与"格式化展示"定位高度相关

2. **嵌入式分享（Embed/iframe）**
   - Gist/Carbon 的关键分享方式
   - 支持 `<script src=".../embed.js">` 或 `<iframe>` 嵌入到博客/文档
   - **优先级：P1**，团队分享场景高频需求

3. **美化截图/导出为图片**
   - Carbon 和 ray.so 的核心价值
   - 代码块可导出为 PNG/SVG，适合社交媒体分享
   - **优先级：P2**，但这是与 Carbon/ray.so 差异化最大的缺失项

4. **隐私/Secret 链接**
   - Gist 的 secret gist 模式
   - 未持有链接的人无法访问
   - **优先级：P2**，但当前无认证场景下所有条目都是"公开"的，风险较高

5. **代码 Diff 视图**
   - Gist 版本对比功能
   - 虽然版本历史已排除，但单个条目的文件更新 diff 有实用价值
   - **优先级：P2**

## 七、技术设计专项问题

### 🔴 严重问题

| # | 问题 | 详情 |
|---|------|------|
| 1 | **entry_id 即 slug 导致主键可变** | 如前述，id 设计混乱，slug 冲突时 id 追加后缀，所有外键引用需同步更新 |
| 2 | **SQLite 全文搜索性能** | `json_each()` 做标签过滤 + LIKE 做搜索，数据量增长后性能堪忧；建议引入 FTS5 |
| 3 | **文件存储扁平化丢失目录结构** | `src/main.py → src_main.py` 的命名策略在还原时无法反向解析；应保留原始相对路径信息 |
| 4 | **二进制文件 base64 传输效率** | JSON 体内 base64 编码增加 33% 体积；大文件应使用 multipart/form-data 上传 |
| 5 | **前端无构建产物缓存策略** | Vite 构建后如何设置长期缓存？API 和静态文件的缓存策略未定义 |
| 6 | **CORS 策略** | 开发时前后端分离需要 CORS，生产模式由 FastAPI 托管则不需要；但未定义开发/生产切换逻辑 |

### 🟡 设计建议

| # | 建议 | 说明 |
|---|------|------|
| 1 | API 响应增加 `request_id` | 便于调试和日志追踪 |
| 2 | 前端路由增加 `/view/:id/:file` | 直接定位到某文件，支持分享链接指向特定文件 |
| 3 | 数据库增加 `entries.updated_at` 索引 | 索引页按更新时间排序需要 |
| 4 | 配置文件支持 XDG 规范 | `~/.peek/` → `$XDG_DATA_HOME/peek/`，Linux 惯例 |
| 5 | 增加 `/api/v1/stats` 端点 | 返回条目总数、存储用量等统计信息 |
| 6 | Docker 部署增加 volume 挂载说明 | 数据持久化关键配置 |

## 十、结论

Peek 项目的核心定位清晰，"Agent 产出格式化展示"是一个有价值且差异化的方向。MCP 集成和自托管是相对于 Carbon/ray.so 的独特优势。但当前需求文档在**完整性**和**严谨性**上存在明显不足：

1. **缺少用户故事和验收标准**——团队对"做完"的定义不统一
2. **缺少资源限制和错误规范**——上线后可能出现资源滥用和不可控行为
3. **entries.id 设计缺陷**——应尽早修正，否则技术债累积
4. **代码行高亮和嵌入式分享**——这是代码展示类产品的核心功能缺失

建议在开发启动前，先完成 P0 级别的文档补充，尤其是资源限制、错误规范和 ID 方案重构。这不会增加太多时间成本，但能显著降低开发过程中的返工风险。
