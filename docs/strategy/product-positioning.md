# PeekView 产品定位与差异化战略

> 创建：2026-06-11
> 最后更新：2026-06-29
> 性质：产品方向文档，非技术规格
> 背景：与 TokDoc 等同类工具对比后，对 PeekView 差异化路线的系统性思考
> 2026-06-29 更新：基于两轮评审校准——①护城河四点交集叙事（review-peekview-expert）②single-shot 现实 + 数据驱动纪律（review-position-02）

---

## 一、我们在解决什么问题

AI Agent 的工作方式产生了一个新问题，这个问题在 LLM 出现之前不存在：

**Agent 大量产出有价值的内容，但这些内容困在终端和聊天上下文里——没有渲染，没有持久化，没有可分享的形态，也没有其他 Agent 能读取的接口。**

这不是"又一个 pastebin 的问题"。pastebin 解决的是"人把内容贴进去"的问题，而这里的问题是"Agent 如何把内容送出去，让人和其他 Agent 都能消费"。

两件事看起来相似，但驱动方式完全不同：
- pastebin：人 → 粘贴 → 人看
- PeekView 的目标：Agent 自主 → 发布 → 人看 + Agent 读

这个区别决定了产品的每一个设计决策。

---

## 二、用户是谁

**核心用户**：在本机或服务器上运行多个 AI Agent，让它们执行真实工作任务的用户。

这类用户的特征：
- 同时跑多个 Agent（Claude Code、Codex、OpenCode、Hermes 等），各自执行不同任务
- Agent 产出的是有价值的内容：代码分析、报告、文档、调研结果
- 需要把这些产出**变成可访问的链接**分享给他人，或作为工作流的中间产物
- 不想每次都手动复制粘贴，希望 Agent 自己完成发布
- 在意谁发布了什么（多 Agent 的内容来源可追溯）

**不是我们的核心用户**：
- 偶尔分享代码片段的开发者（GitHub Gist 已经够用）
- 需要在线协作编辑文档的团队（Notion/语雀更合适）
- 只需要本地文档管理的个人用户（TokDoc 更合适）

---

## 三、差异化的核心命题

**PeekView 不是"更好的 Pastebin"，而是"Agent 可写的高质量可分享渲染器"。**

> 2026-06-29 校准：原定位为"AI Agent 的输出基础设施"。经评审后认识到——"基础设施"暗示了尚未发生的多方协作行为。PeekView 今天的真身更小：一个 agent 生成内容 → 发布 → 人看一眼的单向管道。这个更小的真身恰恰是 Artifact 杀不死的（§四）。"基础设施"是 upside，不是地板。

这两个定位导向完全不同的产品决策：

| 问题 | Pastebin 视角 | Agent 渲染器视角 |
|------|--------------|-----------------|
| 主要入口是什么？| Web 表单，人手动操作 | MCP 工具、CLI、API，Agent 自动调用 |
| 谁是内容的生产者？| 人 | Agent |
| 谁是内容的消费者？| 人 | 人（Agent 读是 upside，不是地板）|
| 内容从哪来？| 人从编辑器里粘贴 | Agent 运行时直接产出 |
| 核心价值是什么？| 方便分享 | 让 Agent 产出被高质量渲染并可分享 |

确立了这把尺子，后续每个功能的取舍就有统一判据：**它服务的行为现在是否存在？** 凡是强化已存在行为的，做；凡是为尚未发生的行为造基础设施的，冻结等数据。

## 四、已有的差异化基础

PeekView 的护城河不是某一项能力，而是以下四者的**交集**——单独看每一项都有替代方案，但同时满足全部四点的只有 PeekView：

**1. 跨厂商中立**

PeekView 适用于任何 Agent——Claude Code、Codex、OpenCode、Cursor、Hermes，不绑定任何平台。MCP 是核心入口，Streamable HTTP transport、双模式（local/remote）、publish_files 绕过上下文污染——这些都是针对 Agent 工作方式量身设计的。

特别是 local 模式的设计哲学：我们诊断出"2 分钟问题"的根因是文件内容经过 LLM 上下文，然后把 `publish_files` 设计成传路径而不是传内容，从根本上解决了这个问题。这不是一个工程技巧，而是对"LLM-in-the-loop"约束的深度理解。

**2. 对组织外公开分享**

分享链接（T027）支持对组织外的公开分享。平台内建的内容分享（如 Claude Code Artifacts）限于组织内部，而 PeekView 的链接可以发给任何人。

**3. 持久快照（vs Artifact 的活进程窗口）**

Artifact 是"一个活进程的窗口"——价值绑在"会话还在跑"上，看的是 agent 正在做的事。PeekView 是"一次发布动作的记录"——我做完了，这是结果。持久、可公开、可归属。

PeekView 在"活"这条轴上永远赢不了 Artifact（因为 Artifact 在会话里面，PeekView 在会话外面），但也不需要赢——持久快照是一个不同的价值。Artifact 的内容随会话消失，PeekView 的链接永久可访问。

> **注意**：PeekView 的内容理论上可变（`update_entry` 覆盖式更新），但 URL 语义不明确——覆盖不留版本，旧内容丢失。这个中间态是未来版本化/时间契约的触发点，但当前实践中几乎没有更新，暂不需要处理。

**4. 数据主权/自托管**

内容托管在自己的服务器上，MIT 协议，不依赖任何大厂的云服务。重度自托管用户本就不愿把 Agent 产出交给第三方平台。

---

> **交集的含义**：这四项单独看都不足以形成护城河（跨厂商有 Gist、公开分享有 pastebin、自托管有 PrivateBin），但同时满足全部四点的产品几乎不存在。PeekView 的生存空间就是这个交集——**它真实存在，但比最初设想的窄。更重要的是，这个交集不依赖"多 Agent 总线"——就算总线永远不发生，这四点交集依然是一个 Artifact 杀不死的小生态位。**

---

## 五、战略方向：两层纪律

> 关键纪律：**让真实数据（不是战略文档、也不是评审建议）决定每一个重功能要不要做。**

### 地板层（现在就做——这是真实在用的）

PeekView 今天的真实使用形态是 **single-shot**：生成 → 发布 → 查看 →（也许）丢。把这条路做到无可挑剔：

#### 方向 1：冷打开链接体验

这是真实高频路径，值得打磨：
- 加载速度、移动端适配
- "这是什么 / 谁发的 / 什么时候"一眼清楚
- entry 详情页的元信息呈现

#### 方向 2：渲染质量持续提升

已经强（Shiki 100+ 语言 / Markdown / Mermaid / PlantUML / HTML sandbox），继续：
- 结构化报告渲染：长 Markdown 自动生成目录、章节跳转、锚点链接
- 数据表格：客户端排序和筛选
- 多文件项目视图：文件树导航

#### 方向 3：内容来源可追溯

entry 列表显示来源账号（"哪个 Agent 发的"），支持按账号过滤。这是最小可行版本的 Agent 行为追溯，工作量小，让多 Agent 用户立刻感受到独特价值。

---

### 探针层（不造重的，只留可观测的最小原语）

多 Agent 总线会不会发生，取决于一件事——有没有 agent 真的去读另一个 agent 的产出。

- 这条路径技术上已存在：MCP `getEntry` / `listEntries` 工具都在
- 但**没有任何读取埋点**：T027 的 `view_count` 只服务分享链接，不区分人/agent，也不在 MCP 读取路径上
- 所以**不知道**这条路径有没有被用

#### 方向 4：MCP 读取路径埋点

给 entry 的读取路径埋一个最小探针：**有没有非创建者的身份（尤其是不同 API key）在读 entry？读的频率？**

判据：
- 半年数据都是零 → 总线愿景从战略文档**降级**
- 开始出现信号 → 那时再**按真实形状**去加强它（检索、引用关系、语义查询）

---

### 重活层（一个都别提前）

等数据显示出需要它们的行为，再做：

| 项目 | 触发条件 |
|------|----------|
| 版本化 / 时间契约 | 出现"内容被改写"或"多 agent 引用"行为 |
| /embed iframe | 出现"链接被贴进宿主页面"行为 |
| OG 预览卡片 | 数据显示链接在"被传出去" |
| 多 Agent 总线加强（检索、引用图谱、语义查询） | 探针出现非创建者读取信号 |
| live 更新 | 不做（会话观察是 Artifact 主场）|

---

## 六、不做什么（边界同样重要）

**❌ 在线编辑**

这是 TokDoc 的路线，也是 Notion/语雀的路线。PeekView 的内容由 Agent 生成，编辑是 Agent 的事，不是 PeekView 的事。加在线编辑会把产品拖向"轻量文档工具"，失去 Agent 基础设施的定位。

**❌ 版本控制（当前）**

Git 和 GitHub Gist 在这个方向已经做到极致。PeekView 有过期时间和内容替换已经够用。真正需要版本管理的内容，应该在 Git 里管，发布到 PeekView 的是最终产出，不是草稿。如果未来出现"内容被改写"的真实行为，再重新评估。

**❌ 协作评论和审批流程**

这是企业知识管理工具的领域（Confluence、语雀）。PeekView 不是文档协作平台。

**❌ 为了"功能完整"而追赶 Gist/SnipShare**

fork、star、collections、密码保护——这些功能 Gist 已经做得很好了。与其在 Gist 已有的功能上追赶，不如在 Gist 做不到的地方（Agent 集成、自动发布、多 Agent 身份）建立壁垒。

**❌ 主动送达/通知/收件人管理**

PeekView 是内容呈现层，不是消息路由层。链接本身就是解耦点——任何送达层（邮件、A2A 协议、Slack webhook、人手动转发）都能拿 PeekView 的链接去送达。PeekView 是信纸，不需要长出投递功能。

**❌ Live 更新/实时推送**

Artifact 是"活进程的窗口"，PeekView 是"发布记录"。追"活"是去 Artifact 主场正面打，没胜算。

## 七、产品原则

从"Agent 可写的高质量可分享渲染器"的定位推导出的产品原则：

**原则 1：Agent 是一等公民，人是消费者**

所有核心功能的主要触发方都应该是 Agent，而不是人。人通过浏览器看内容、管理账号——这是消费端，不是生产端。任何需要人手动操作才能完成的发布流程，都是设计失败。

**原则 2：绕过 LLM 上下文是设计目标，不是实现细节**

publish_files 传路径、Path Namespace 映射——这些设计都服务于同一个目标：让大内容的发布不经过 LLM 推理链。这个原则要贯穿所有涉及内容传输的设计。

**原则 3：身份体系是内容可信度的基础**

多 Agent 场景下，没有身份的内容是噪音。每个 entry 的来源账号、发布时间、关联上下文，是 PeekView 内容价值的一部分。身份不是认证功能，是内容质量的保障。

**原则 4：渲染是产品的门面**

人最终看到的是渲染后的内容。代码高亮、Mermaid 图表、结构化报告——这些决定了"Agent 用 PeekView 发布内容"这件事在人类眼中的价值。渲染质量直接影响用户是否愿意把 PeekView 链接发给别人。

**原则 5：简单可用优先于功能完整**

MCP 工具的数量不代表能力。"选择越少，出错越少"——local 模式只暴露 publish_files 就是这个原则的体现。每增加一个工具，Agent 就多一次选择错误的机会。

**原则 6：数据驱动，不预设功能**

用"它服务的行为现在是否存在"作为统一判据。凡是为尚未发生的行为造基础设施的，冻结等数据。view_count、MCP 读取埋点——这些探针才是判断方向的依据，不是战略文档的设想。

## 八、PeekView 赢的条件

PeekView 的命运和 Agent 生态的发展深度绑定。它的差异化能否兑现价值，取决于：

**赢的条件**：
- AI Agent 工作流越来越普及，"让 Agent 的产出变成可分享的东西"变成高频需求
- 多 Agent 协作成为主流工作方式（一台机器跑多个 Agent 各司其职）
- 用户开始关心"我的 Agent 做了什么"（可追溯性需求出现）

**挑战**：
- **平台内建内容分享（已发生）**：Claude Code Artifacts（2026-06-18）正面命中 PeekView 的核心叙事——不用把终端输出粘到 Slack，agent 原地生成可分享的 HTML/Markdown 页面。Artifact 吃掉了"已付费用 Claude Code 企业版、且只在组织内分享"的那部分用户
- **赛道窄化**：PeekView 的真实生存空间是"跨厂商 + 公开分享 + 持久快照 + 自托管"的交集，比最初设想的窄。这不是否定项目，而是把增长预期校准到"小而精的自托管 Agent 重度用户"的现实规模
- **轻量替代**：PreviewShip 等工具让 agent 通过 MCP 直接把 HTML 发布成预览链接，对偶尔需要的用户可能已经够用
- **概念教育成本**："Agent 输出基础设施"比"pastebin"难理解得多
- **战略与现实之间的裂缝**：战略描述的是多 agent 内容生态，实际在跑的是 single-shot 发即走。必须诚实承认这条裂缝，用数据而非愿景来驱动

**核心押注**：未来真正重度使用 AI Agent 的用户，不会满足于在聊天框里看 Agent 的输出，他们需要的是一个可以持久化、可以分享、可以被其他 Agent 读取的内容层。PeekView 押注这类用户会越来越多，并且他们愿意为此自托管一套工具。但必须诚实：这部分用户是小众，增长不会爆发式，小而精是更现实的预期。**总线愿景是 upside，不是地板——PeekView 不需要总线来证明自己有存在理由。**

## 九、与竞品的关系

| 工具 | 关系 | 说明 |
|------|------|------|
| TokDoc | 互补，可集成 | TokDoc 做本地编辑管理，PeekView 做对外发布；TokDoc 的"线上绑定"功能可直接对接 |
| GitHub Gist | 面向不同用户 | Gist 是开发者手动分享，PeekView 是 Agent 自动发布；使用场景不重叠 |
| Pastebin/SnipShare | 弱竞争 | 功能上有重叠，但用户和使用方式不同；PeekView 不应追赶它们的功能列表 |
| Notion/语雀 | 不竞争 | 完全不同赛道 |
| Claude Code Artifacts | 直接威胁 | 正面命中 PeekView 核心叙事（agent 产出可分享页面），但限于 Team/Enterprise + org-only + 单会话产物。Artifact = 活进程窗口，PeekView = 持久发布记录——两者不同轴 |
| PreviewShip | 轻量替代 | MCP + 部署组合，启动成本几乎为零，对偶尔需要的用户够用，但没有身份体系和渲染深度 |

## 十、近期最应该做的事

基于两层纪律（地板 + 探针），近期最能强化差异化的四件事：

**1. 冷打开链接体验**（地板层——真实高频路径）

加载速度、移动端、"这是什么 / 谁发的 / 什么时候"一眼清楚。这是用户接触 PeekView 的第一印象，做到无可挑剔。

**2. 渲染质量持续提升**（地板层——门面）

继续打磨 Shiki / Markdown / Mermaid / PlantUML / HTML sandbox。这是让用户愿意把 PeekView 链接发给别人看的前提。

**3. MCP 读取路径埋点**（探针层——判断总线的唯一信号源）

给 entry 的读取路径加最小探针：谁在读、读频率、是否非创建者。这是唯一能回答"多 Agent 总线会不会发生"的数据源。半年零信号 → 总线降级；出现信号 → 按真实形状加强。

**4. 内容来源可追溯**（地板层——最小追溯）

entry 列表显示来源账号，支持按账号过滤。工作量小，让多 Agent 用户立刻感受到独特价值。
