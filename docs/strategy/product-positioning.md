# PeekView 产品定位与差异化战略

> 创建：2026-06-11
> 性质：产品方向文档，非技术规格
> 背景：与 TokDoc 等同类工具对比后，对 PeekView 差异化路线的系统性思考

---

## 一、我们在解决什么问题

AI Agent 的工作方式产生了一个新问题，这个问题在 LLM 出现之前不存在：

**Agent 大量产出有价值的内容，但这些内容困在终端和聊天上下文里——没有渲染，没有持久化，没有可分享的形态，也没有其他 Agent 能读取的接口。**

这不是"又一个 pastebin 的问题"。pastebin 解决的是"人把内容贴进去"的问题，而这里的问题是"Agent 如何把内容送出去，让人和其他 Agent 都能消费"。

两件事看起来相似，但驱动方式完全不同：
- pastebin：人 → 粘贴 → 人看
- PeekView 的目标：Agent 自主 → 发布 → 人看 + Agent 读

这个区别决定了产品的每一个设计决策。

---

## 二、用户是谁

**核心用户**：在本机或服务器上运行多个 AI Agent，让它们执行真实工作任务的用户。

这类用户的特征：
- 同时跑多个 Agent（Claude Code、Codex、OpenCode、Hermes 等），各自执行不同任务
- Agent 产出的是有价值的内容：代码分析、报告、文档、调研结果
- 需要把这些产出**变成可访问的链接**分享给他人，或作为工作流的中间产物
- 不想每次都手动复制粘贴，希望 Agent 自己完成发布
- 在意谁发布了什么（多 Agent 的内容来源可追溯）

**不是我们的核心用户**：
- 偶尔分享代码片段的开发者（GitHub Gist 已经够用）
- 需要在线协作编辑文档的团队（Notion/语雀更合适）
- 只需要本地文档管理的个人用户（TokDoc 更合适）

---

## 三、差异化的核心命题

**PeekView 不是"更好的 Pastebin"，而是"AI Agent 的输出基础设施"。**

这两个定位听起来接近，但导向完全不同的产品决策：

| 问题 | Pastebin 视角 | Agent 基础设施视角 |
|------|--------------|-------------------|
| 主要入口是什么？| Web 表单，人手动操作 | MCP 工具、CLI、API，Agent 自动调用 |
| 谁是内容的生产者？| 人 | Agent |
| 谁是内容的消费者？| 人 | 人 + 其他 Agent |
| 内容从哪来？| 人从编辑器里粘贴 | Agent 运行时直接产出 |
| 核心价值是什么？| 方便分享 | 让 Agent 产出可消费 |

确立了这个命题，后续每个功能的取舍就有了标准：**凡是强化"Agent 产出 → 人/Agent 消费"这条链路的，做；凡是向"文档管理"方向漂移的，不做。**

---

## 四、已有的差异化基础

这些是 PeekView 已经建立的、其他工具没有的：

**1. MCP 原生，不是事后加的**

PeekView 的 MCP 是核心入口，不是功能附加。Streamable HTTP transport、双模式（local/remote）、publish_files 绕过上下文污染——这些都是针对 Agent 工作方式量身设计的。

特别是 local 模式的设计哲学：我们诊断出"2 分钟问题"的根因是文件内容经过 LLM 上下文，然后把 `publish_files` 设计成传路径而不是传内容，从根本上解决了这个问题。这不是一个工程技巧，而是对"LLM-in-the-loop"约束的深度理解。

**2. 多 Agent 共享，有身份体系**

同一台机器上的 5 个 Agent，可以共享一个 MCP Server，各自用不同的 API Key，内容按账号归属。这是多 Agent 协作的前提条件。没有身份体系，Agent 的产出就是一堆无归属的内容。

**3. 渲染质量**

Markdown + Shiki（100+ 语言）+ HTML 渲染 + Mermaid 图表。Agent 生成的内容大多是这几种格式，PeekView 都能渲染成可读的形态。

## 五、三个值得深挖的方向

### 方向 1：Agent 之间的内容流转（最重要）

**现状**：PeekView 是单向管道——Agent 发布 → 人看。`get_entry`、`list_entries` 工具存在，但在设计上没有把"Agent 消费内容"当成一等公民。

**应该变成什么**：内容的双向节点。Agent A 发布分析结果，Agent B 通过 MCP 读取这份内容，结合自己的工作继续发布下一层产出。PeekView 变成多 Agent 工作流的**共享内容总线**。

**具体来说**：
- `list_entries` 加强检索能力（按标签、时间、来源 Agent 过滤），让 Agent 能找到它需要的上游产出
- 内容的引用关系：一个 entry 可以声明它基于哪个 entry（parent_slug），形成可追溯的内容图谱
- Agent 查询接口的语义化：不只是"列出条目"，而是"找到关于 X 主题的最新内容"

**为什么重要**：这把 PeekView 从"存档工具"变成"协作基础设施"。Agent 之间不需要通过对话上下文传递中间产出，而是通过 PeekView 共享——这才是多 Agent 协作的正确姿势。

---

### 方向 2：Agent 行为的可追溯性

**现状**：多个 Agent 在跑，各自发布内容，时间一长没人知道谁在什么时候发了什么，为什么发。

**应该变成什么**：Agent 活动的审计日志。每个 entry 有明确的来源（哪个账号/Agent）、时间、关联的 Agent 任务上下文。

**具体来说**：
- 发布时支持 `agent_id` 或 `context` 字段（"这个报告是分析 PR #123 时产生的"）
- 按 Agent 过滤的 timeline 视图
- 统计面板：哪个 Agent 发布最多？什么类型的内容？什么时间段？
- 内容衰减策略：旧的 Agent 产出可以自动过期或归档，保持内容新鲜

**为什么重要**：当你跑的 Agent 越来越多，"哪个 Agent 产出了什么"会变成真实的管理问题。PeekView 有账号体系，天然适合做这件事，只是现在还没用起来。

---

### 方向 3：渲染质量的持续提升

**现状**：Markdown + 代码高亮 + HTML + Mermaid，基础能力完整。

**应该变成什么**：Agent 产出的内容不只是"可以看"，而是"真的好用"。

**具体来说**：
- 结构化报告渲染：长 Markdown 自动生成目录、章节跳转、锚点链接
- 数据表格：Agent 产出的 Markdown 表格支持客户端排序和筛选
- 多文件项目视图：Agent 发布的多文件 entry 有文件树导航，不只是平铺列表
- 嵌入分享（`/embed/{slug}`）：让 PeekView 的内容可以被嵌入到其他页面——Agent 产出的报告能直接嵌进 Notion 或公司 Wiki

**为什么重要**：Agent 的价值最终体现在人类能消费它的产出。渲染质量是 PeekView 的门面，也是让用户愿意把 PeekView 的链接发给别人看的前提。

---

## 六、不做什么（边界同样重要）

**❌ 在线编辑**

这是 TokDoc 的路线，也是 Notion/语雀的路线。PeekView 的内容由 Agent 生成，编辑是 Agent 的事，不是 PeekView 的事。加在线编辑会把产品拖向"轻量文档工具"，失去 Agent 基础设施的定位。

**❌ 版本控制**

Git 和 GitHub Gist 在这个方向已经做到极致。PeekView 有过期时间和内容替换已经够用。真正需要版本管理的内容，应该在 Git 里管，发布到 PeekView 的是最终产出，不是草稿。

**❌ 协作评论和审批流程**

这是企业知识管理工具的领域（Confluence、语雀）。PeekView 不是文档协作平台。

**❌ 为了"功能完整"而追赶 Gist/SnipShare**

fork、star、collections、密码保护——这些功能 Gist 已经做得很好了。与其在 Gist 已有的功能上追赶，不如在 Gist 做不到的地方（Agent 集成、自动发布、多 Agent 身份）建立壁垒。

## 七、产品原则

从"Agent 输出基础设施"的定位推导出的产品原则：

**原则 1：Agent 是一等公民，人是消费者**

所有核心功能的主要触发方都应该是 Agent，而不是人。人通过浏览器看内容、管理账号——这是消费端，不是生产端。任何需要人手动操作才能完成的发布流程，都是设计失败。

**原则 2：绕过 LLM 上下文是设计目标，不是实现细节**

publish_files 传路径、Path Namespace 映射——这些设计都服务于同一个目标：让大内容的发布不经过 LLM 推理链。这个原则要贯穿所有涉及内容传输的设计。

**原则 3：身份体系是内容可信度的基础**

多 Agent 场景下，没有身份的内容是噪音。每个 entry 的来源账号、发布时间、关联上下文，是 PeekView 内容价值的一部分。身份不是认证功能，是内容质量的保障。

**原则 4：渲染是产品的门面**

人最终看到的是渲染后的内容。代码高亮、Mermaid 图表、结构化报告——这些决定了"Agent 用 PeekView 发布内容"这件事在人类眼中的价值。渲染质量直接影响用户是否愿意把 PeekView 链接发给别人。

**原则 5：简单可用优先于功能完整**

MCP 工具的数量不代表能力。"选择越少，出错越少"——local 模式只暴露 publish_files 就是这个原则的体现。每增加一个工具，Agent 就多一次选择错误的机会。

## 八、PeekView 赢的条件

PeekView 的命运和 Agent 生态的发展深度绑定。它的差异化能否兑现价值，取决于：

**赢的条件**：
- AI Agent 工作流越来越普及，"让 Agent 的产出变成可分享的东西"变成高频需求
- 多 Agent 协作成为主流工作方式（一台机器跑多个 Agent 各司其职）
- 用户开始关心"我的 Agent 做了什么"（可追溯性需求出现）

**挑战**：
- Agent 平台（Claude.ai、Cursor 等）可能自带内容分享功能，从上层把这个需求吃掉
- 赛道足够小众，早期增长慢
- "Agent 输出基础设施"这个概念对普通用户来说比"pastebin"难理解

**核心押注**：未来真正重度使用 AI Agent 的用户，不会满足于在聊天框里看 Agent 的输出，他们需要的是一个可以持久化、可以分享、可以被其他 Agent 读取的内容层。PeekView 押注这类用户会越来越多，并且他们愿意为此自托管一套工具。

## 九、与竞品的关系

| 工具 | 关系 | 说明 |
|------|------|------|
| TokDoc | 互补，可集成 | TokDoc 做本地编辑管理，PeekView 做对外发布；TokDoc 的"线上绑定"功能可直接对接 |
| GitHub Gist | 面向不同用户 | Gist 是开发者手动分享，PeekView 是 Agent 自动发布；使用场景不重叠 |
| Pastebin/SnipShare | 弱竞争 | 功能上有重叠，但用户和使用方式不同；PeekView 不应追赶它们的功能列表 |
| Notion/语雀 | 不竞争 | 完全不同赛道，PeekView 的内容可以嵌入进去（/embed） |
| Claude.ai / Cursor | 潜在威胁 | 平台层若自带分享功能会侵蚀需求，但自托管和 API 可编程性是差异 |

---

## 十、近期最应该做的事

基于以上定位，近期最能强化差异化的两件事：

**1. 让 Agent 消费内容变成一等公民**（方向 1 的第一步）

在 MCP 工具描述里，把 `list_entries` 和 `get_entry` 定位成"Agent 读取上游产出的入口"，而不只是"管理工具"。配合检索能力增强（按标签/时间过滤），让 Agent 真正能找到它需要的内容。

**2. 内容来源可追溯**（方向 2 的第一步）

entry 列表显示来源账号（"哪个 Agent 发的"），支持按账号过滤。这是最小可行版本的 Agent 行为追溯，工作量小，但能让多 Agent 用户立刻感受到 PeekView 的独特价值。
