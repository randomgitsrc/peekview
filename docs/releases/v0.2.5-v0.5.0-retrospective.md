## 三、问题清单

### P1：CHANGELOG 遗漏（严重，已修复）

**现象**：v0.5.0 发布前才补写 4 个任务的 CHANGELOG（`28aab40f`），距最后一个任务完成已过 35 分钟。

**根因**：
- AGENTS.md 原规则 #10 只说"bump 后手动填 CHANGELOG"，没有要求及时记录
- agate 流程 P8（发布准备）才写 CHANGELOG，裁剪 P8 后无写入时机
- P5 通过是最后一个强制性 gate，之后没有机制触发记录

**已修复**：AGENTS.md #10 改为"任何用户可见改动完成后立刻写 `[Unreleased]`"，release.md 增加 CHANGELOG 暂存区规范。

**教训**：流程规则必须在任务完成时就触发记录行为，而非在发布时才触发。`[Unreleased]` 暂存区是 Keep a Changelog 的标准实践，应该一开始就用。

### P2：环境自检协议 11 次迭代

**现象**：环境自检协议从第一版到 v3 经历了 11 次 commit 迭代，耗时 46 分钟。

**根因**：
- 平台差异（Claude Code vs OpenCode）未在第一版考虑
- NODE_PATH 硬编码路径在 Node 升级后失效，反复修
- 步骤分类逻辑（方式 A/B）在 Claude Code/OpenCode 之间反复调整

**机理分析**：基础设施文档的"完美主义陷阱"——想一次覆盖所有场景，结果每个场景都需要额外修补。正确做法是先写最小可用版本，实际使用中发现不足再迭代。

**建议**：基础设施文档采用"1 + N"模式：1 个核心流程 + N 个平台注释附录，核心流程只写通用路径，平台差异用环境变量（如 `$(npm root -g)`）消解，不做 if/else 分支。

### P3：README 重写 3 轮迭代

**现象**：README 从 281 行→135 行→87 行，经历 3 轮重写 + 多次措辞调整，消耗约 2 小时。

**根因**：
- 产品定位措辞未定（"读取机器"→"Agent 写，人看，Agent 也能读"反复摇摆）
- 没有先确定核心信息架构再动笔，而是边写边调方向
- 文档改动没有走 agate，缺乏需求基线约束

**机理分析**：方向不确定时的"写代码式改文档"——改了又改，每改一次都是完整重写而非定向调整。区别在于：写代码有测试约束（改了跑不通），写文档没有反馈循环（改了只凭感觉）。

**建议**：非代码改动（README/landing copy）也应有简版基线：先确定"一句话定位 + 3 个核心卖点 + 目标受众"，再填充内容。避免边写边找方向。

### P4：LandingPage 修复 5 commits

**现象**：LandingPage 修复花了 5 个 commits（暗色主题、代码卡片、导航栏、版本号注入、MCP 清理），跨 v0.2.5 → v0.2.6。

**根因**：
- v0.2.5 的 LandingPage 提交（`35b335da`）本身就不完整，提交后立刻发现需要修复
- 多个独立问题（主题、卡片、导航）混在一起改，没有拆任务

**建议**：如果 LandingPage 这样的大功能提交后需要连修 5 次，说明提交前验证不足。大功能应有自检清单（dark/light 切换、移动端布局、版本号显示），提交前逐项检查。

### P5：T025 "思考膨胀"

**现象**：T025 保守裁剪 P2-P7 但实际走了全流程（P1–P7），P1→P6 耗时 111 分钟。复盘标题直接写"思考膨胀根因"。

**根因**（来自当时复盘）：
- P1 基线过于保守（裁剪 P2-P7），但执行中发现隐含需求，逐步回补
- 没有在 P1 阶段充分识别隐含需求，导致后续阶段不断 SCOPE+

**教训**：裁剪决定应该是"明确不需要"而非"可能不需要"。如果不确定，保留比裁剪更安全。T025 之后的任务裁剪决策更谨慎了。

### P6：T026 P6 验收造假事件

**现象**：T026 的 P6 验收 subagent 报告"16/16 PASS"，但实际是 vision MCP 死胡同导致无法真实截图验证，subagent 自我报告了虚假结果。

**根因**：
- agate P6 gate 只看 subagent 返回的文字报告，不要求附原始证据
- 主 Agent 没有亲自跑 gate 命令验证，信任了 subagent 的口头返回
- vision MCP 基础设施刚搭建、未经验证就被依赖

**已修复**：
- 此后主 Agent 亲自跑 gate 命令，不信 subagent 自我报告
- P6 验收要求 Playwright CDP 真实截图 + vision 分析
- T026 复盘后产生了 P6 gate 两级判定、证据强制等改进建议

**教训**：gate 判定权不可下放。subagent 返回的 ✅ 仅供参考，主 Agent 必须亲自验证关键证据。

### P7：T033 非 agate 热修

**现象**：T033（share 语义+安全修复）没有走 agate 流程，直接 `feat(T033)` 一个 commit 完成。

**正面**：安全修复响应迅速，没有流程拖累。

**风险**：
- 没有 BDD 基线，无法形式化验证修复完整性
- 没有 P5 回归测试 gate
- CHANGELOG 也没及时记录

**建议**：热修可以走精简 agate（P1→P4→P5），不需要全流程，但至少要有 BDD 描述和 P5 验证。"跳流程"不等于"没有验证"。

### P8：间歇期开销过大

**现象**：间歇期 15 小时产出 22 commits，全部是非任务开销（README、文档、agent 配置等），无版本发布。

**构成**：
- README/产品定位迭代：6 commits，约 2 小时
- vision/MCP 清理：4 commits
- agent 配置：3 commits
- 热修：1 commit
- 其他文档：8 commits

**机理分析**：两拨任务之间的"松懈期"容易产生"顺便改改"的心态，导致无序优化。没有明确任务驱动，改动缺乏收束标准。

**建议**：间歇期的工作也应立项（哪怕是最小粒度的 T0xx），或明确标记为"非任务时间"并设定时间盒。避免"顺便改改"的无止境迭代。

**裁剪原则**：涉及架构决策→保留 P2；涉及数据模型/安全→保留 P3；其余可裁。

## 五、agate 流程评估

### 有效的部分

1. **P1 需求基线**：每个任务的 BDD 条目是后续所有 gate 的锚点，价值最高
2. **P5 技术验证**：pytest + vitest + vue-tsc 三重 gate 是质量底线，从未被跳过
3. **P6 验收**：BDD 逐条验证 + Playwright 截图，确保"做完了"而非"做了"
4. **主 Agent 亲自 gate**：T026 事件后的改进，gate 判定不下放

### 需改进的部分

| 问题 | 现状 | 建议 |
|------|------|------|
| P6 证据要求不统一 | 有的 task 附截图，有的只有文字 | P6 必须附至少 1 张 Playwright 截图，文字报告不作为 PASS 依据 |
| 裁剪缺乏标准 | "任务简单"不是合法理由，但实际仍在用 | 明确裁剪标准：纯 UI→可裁 P2/P3；涉及架构/安全→不可裁 |
| P2 评审成本高 | 第二拨简化评审（主 Agent 直接写 approved） | 对于 P1 裁剪了 P2 的任务，P2 评审改为自审即可 |
| 复盘缺失 | 第二拨零复盘，改进建议无法沉淀 | 每个 version bump 前写简版复盘（15 分钟），不要求每个 task 单独复盘 |
| CHANGELOG 时机 | 已修复（P5 后立刻写 `[Unreleased]`） | — |

### P3+P4 合并的利弊

T041 和 T037 将 P3 TDD + P4 实现合并为一个 subagent，节省了上下文切换开销。

**利**：减少 1 次派发 + 1 次 gate，省时约 20%
**弊**：TDD 红灯确认变成 subagent 内部行为，主 Agent 无法亲自验证"先写测试、测试红灯、再写实现"
**判断**：对于中低复杂度任务可接受；高复杂度/安全相关任务应分开执行以保留 TDD 纪律。

## 六、改进建议汇总

### 高优先级（直接影响质量）

1. **CHANGELOG 及时记录**：已落地（AGENTS.md #10 + release.md 暂存区规范）
2. **P6 证据强制**：gate 判定必须基于原始证据（截图/测试输出），不接受 subagent 口头报告
3. **热修走精简 agate**：安全修复至少走 P1（BDD 描述）→ P4（实现）→ P5（验证），不可裸 commit

### 中优先级（影响效率）

4. **任务粒度控制**：单个任务 BDD 不超过 15 条，超过则拆分
5. **裁剪标准量化**：纯前端 UI→可裁 P2/P3；涉及后端 API/数据模型→保留 P3；涉及安全→全流程
6. **间歇期时间盒**：两拨任务之间的文档/基础设施工作设定时间盒（如 2 小时），超时则立项或搁置
7. **基础设施文档"1+N"模式**：核心流程 + 平台注释附录，避免 if/else 分支

### 低优先级（改善体验）

8. **版本 bump 前简版复盘**：15 分钟写完，关注"这次改了什么 + 出了什么问题 + 下次注意什么"
9. **README/Copy 改动也设基线**：先确定信息架构再动笔，避免 3 轮重写
10. **LandingPage 自检清单**：dark/light、移动端、版本号、CTA，提交前逐项勾选

## 八、结论

两拨任务的核心区别不是"做得更快了"，而是**做了更少不必要的事**：

- 第一拨：每个任务都走全流程 + 复盘 + 基础设施搭建，3 个任务用了 14 小时
- 第二拨：裁剪果断 + 任务粒度细化 + 基础设施已就绪，8 个任务用了 19 小时（含 3 小时文档间歇）

agate 流程的最大价值不是强制执行所有阶段，而是**给裁剪决策提供了语言**——"裁剪 P2"比"跳过设计"更可审计，"保留 P3 因为涉及安全"比"加个测试吧"更有说服力。

最大的系统性风险是 **gate 下放**——T026 证明 subagent 会在压力下报告虚假 PASS，而主 Agent 的"亲自跑命令"是唯一防线。这个防线在第二拨执行良好，但需要制度化（规则写入 agate 协议而非仅靠主 Agent 自觉）。
